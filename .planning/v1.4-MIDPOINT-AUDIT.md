## Cross-phase integration — 6/6 wired

Phase 44 is infrastructure, Phase 45 is a feature; they meet at the deploy/PWA seam. That seam
was the point of this audit, and the highest-risk hypothesis going in was this: Phase 45 added a
brand-new shipped asset (`assets/md-render.js`), and Phase 44 introduced a *whitelist* staging
transform. If that whitelist enumerated files individually, the new asset would be silently
dropped from every deploy — read-mode rendering would break in production while passing 100% of
local tests.

**It does not.** `scripts/build-staging.sh` copies with `cp -r assets/`, a recursive directory
copy; the whitelist operates at the directory/file-class level, never per-asset. `md-render.js`
ships automatically. The trap was real, and the code doesn't fall into it.

| Seam | Status |
|------|--------|
| build-staging.sh whitelist → new `md-render.js` asset | ✓ wired (`cp -r assets`, no per-file enumeration) |

## Two process gaps in Phase 44's paperwork

Neither affects shipped behavior; both are gates that never ran.

**1. No `44-SECURITY.md`.** All five Phase 44 plans carry a `threat_model` block, but
`/gsd-secure-phase` was never run, so those mitigations are unverified. This is the phase that
touches the deploy pipeline, `CF_ZONE_ID`/`CF_PURGE_TOKEN` secrets, and a second public origin —
the one phase where a threat-model verification actually earns its keep. Phase 45 by contrast
has a `45-SECURITY.md` (status: verified, 0 open threats).

**2. `44-VALIDATION.md` is stale bookkeeping.** It reads `status: draft`,
`wave_0_complete: false`, every Wave-0 checkbox unticked, `Approval: pending` — while all three
Wave-0 test files exist and pass (confirmed in 44-VERIFICATION.md). The coverage is real; the
paperwork just never got closed. This is the second time (see the pending
`2026-07-10-phase43-nyquist-bookkeeping.md` todo), which makes it a pattern rather than a slip.

rich-text foundation exists on your machine and nowhere else. The good news: I ran

legacy notes render unchanged — RTXT-10), but it is a real deploy that rolls the PWA cache for
every installed user. Ship it now, or hold until the toolbar lands with it.

## Tech debt — 10 open Info-level items

All ten were explicitly triaged as "no fix required" by their reviewers. None contradicts a
must-have. Two are worth a second look before Phase 46 builds on this code:

- **45 IN-03** — the "app's ONE sanctioned innerHTML path" comment in `add-session.js:280-281`
  is factually wrong; `export-modal.js:564` is a second (equally safe, MdRender-routed) path.
  Both are secure, but the comment will misdirect the next security audit that greps for the
  sanctioned exception and stops reading. Cheap to fix, and Phase 46 will add editor surfaces
  that make the miscount worse.
- **44 IN-02** — `build-staging.sh` never cleans its target directory, so the "no-leak"
  invariant it advertises only holds on a fresh checkout. Moot in CI, real if you ever stage
  locally.
