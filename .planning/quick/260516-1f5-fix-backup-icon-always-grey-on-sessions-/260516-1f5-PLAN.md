<root_cause>
CONFIRMED root cause (diffed working vs broken page script loading):

`assets/app.js` is route-agnostic. `initCommon()` (called on every page) runs
`mountBackupCloudButton()` and `updateBackupCloudState()`. Both derive the icon
color exclusively from `BackupManager.computeBackupRecencyState()`:

- app.js:438-442 — `if (typeof BackupManager !== 'undefined' && ...)` else
  `initialState` stays `'never'`.
- app.js:524-528 — same guard; else `state` stays `'never'`.
- `'never'` → CSS class `backup-cloud-btn--never` → grey (app.css:3659).

`BackupManager` is the IIFE export of `assets/backup.js`. The 4 WORKING pages
all load the script trio AFTER `app.js`:
  - index.html:337-340       backup.js + jszip + backup-modal.js
  - add-client.html:196-197  jszip + backup.js + backup-modal.js
  - add-session.html:596-598 jszip + backup.js + backup-modal.js
  - settings.html:318-320    jszip + backup.js + backup-modal.js

The 2 BROKEN pages load `app.js` but DO NOT load `backup.js`,
`jszip.min.js`, or `backup-modal.js`:
  - sessions.html  — ends: app.js → sessions.js  (no backup trio)
  - reporting.html — ends: app.js → reporting.js (no backup trio)

So on Sessions/Reporting, `BackupManager` is `undefined` → the guard fails →
the icon is permanently `--never` (grey). Secondary effect: the cloud-icon
click handler (app.js:475-481) has no `window.openBackupModal`, so it bounces
to `./index.html?openBackup=1` instead of opening in-place — also fixed by
loading the same trio (this is exactly the Phase 25 round-5 UAT-D2 fix already
applied to add-client/add-session/settings).

The CSS state-color classes (`backup-cloud-btn--fresh|warning|danger`,
app.css:3664-3674) are already present — both broken pages load `app.css`. The
ONLY missing piece is the `backup.js` runtime on these two pages.
</root_cause>

<canonical_script_pattern>
Every working page places the trio immediately AFTER `<script src="./assets/app.js"></script>`
and BEFORE the page-specific script, in this order:

  <script src="./assets/jszip.min.js"></script>
  <script src="./assets/backup.js"></script>
  <script src="./assets/backup-modal.js"></script>

`assets/jszip.min.js` exists (97 KB, present). `backup-modal.js` self-injects
the modal markup into <body> when `#backupModal` is absent and defines
`window.openBackupModal` (see tests/25-09-modal-global-inplace.test.js header).
Load order matters: the trio must come AFTER app.js (app.js's
`updateBackupCloudState` is invoked from `initCommon`, but app.js is parsed
top-to-bottom and `initCommon` runs on DOMContentLoaded — by then the
later-in-document trio scripts have executed and `BackupManager` is defined).
This is the exact ordering proven working on the other 4 pages.
</canonical_script_pattern>
