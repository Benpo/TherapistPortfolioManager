<root_cause>
## Root cause (already investigated — do not re-derive)

**It is NOT a date-format / timezone / parse bug. `initBirthDatePicker.setValue` parses ISO
"YYYY-MM-DD" correctly.** The bug is missing data plus two code gaps:

1. **Legacy data shape.** Before commit `9dafbb6 feat(add-client): overhaul client form
   fields and types`, the client form had a *direct* age number input (`#clientAge`, old
   `add-client.html:58`; old `add-client.js:63/125/132` read/wrote `editingClient.age`
   directly). The overhaul replaced it with the three-dropdown `birthDatePicker`, and age is
   now *derived from* a `birthDate` ISO string at save time
   (`assets/add-client.js:201-202`). **No DB migration back-fills `birthDate` from a legacy
   `age`** — `assets/db.js` `MIGRATIONS` (DB_VERSION 6, ~lines 217-309) never touch the
   `age`/`birthDate` fields. So a legacy client legitimately has `age: 40` and
   `birthDate: undefined`.

2. **Display falls back to the stored age, edit form does not.**
   - Display (overview list `assets/overview.js:619-621`, add-session spotlight
     `assets/add-session.js:1427-1429`): `client.birthDate ? computeFromBirthDate :
     client.age` → shows the legacy **40**.
   - Edit form populators only fill the picker when a `birthDate` exists:
     `assets/add-client.js:123-125` (`if (birthDatePicker && editingClient.birthDate)`) and
     `assets/add-session.js:344-348`. With no `birthDate`, all three dropdowns stay on the
     em-dash "—". The legacy `age` is shown read-only elsewhere but is invisible/uneditable
     here → the user's "how can that be?".

3. **Latent data-loss bug (fix this too).** On save, both edit paths do
   `const age = birthDate ? <computed> : null` — `assets/add-client.js:201-202` (edit branch
   spread at 224-230) and `assets/add-session.js:472-473` (spread at 489-495). Editing a
   legacy client and saving — even just changing a phone number — **overwrites the stored
   `age` with `null`**, silently destroying the displayed 40.

**Chosen fix (honest, no fabricated dates):** (a) preserve the existing stored `age` on save
when no birth date is entered; (b) surface the legacy age in the edit form with a localized
note so the empty picker is explained. We do NOT fabricate a birth year from the age (we
don't know the real month/day; estimating would silently mutate data on save).
</root_cause>
