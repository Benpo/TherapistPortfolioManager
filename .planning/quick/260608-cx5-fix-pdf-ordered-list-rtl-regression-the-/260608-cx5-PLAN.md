Quick task 260608-cx5 — fix the RTL regression introduced by 260608-c8x.

The c8x split-row branch in `assets/pdf-export.js` (around line 1043) fires for EVERY RTL ordered-list item, including Hebrew-content items that the previous unified-row path was rendering correctly. The unified row let UAX #9 handle the whole row as one RTL paragraph, producing the correct visual layout for Hebrew items. The split-row shapes "1. " and the content as two independent paragraphs, and the two calls' outputs do not compose like a single bidi paragraph would — the user sees the digit + period mis-arranged relative to the Hebrew content.

Gate the split-row branch on the CONTENT's first-strong direction:
- RTL doc + ordered + wi === 0 + content first-strong is LTR → split-row (keep c8x Bug B fix).
- Otherwise (RTL content or no-strong-chars content) → unified-row (the previous-stable path).

Use the already-defined `firstStrongDir` helper from the same file (defined around line 231, called from the same closure scope by `shapeForJsPdf` and `shapeForJsPdfWithMap`).

Purpose: restore correct Hebrew-content rendering in RTL ordered lists while preserving the c8x English-content fix.
Output: 1 falsifiable behavior test (RED first), gated list-renderer code path (GREEN), updated comment block.
