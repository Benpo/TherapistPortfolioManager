<objective>
Fix the session PDF export so that single line breaks inside a paragraph
(notably a line of dashes typed on its own line, e.g. `----------------`)
render on their OWN line in the exported PDF — matching the editor
live-preview — instead of being collapsed inline with the surrounding text.

Purpose: A therapist typed a dash separator line between two text lines in a
session note. The editor preview correctly shows it on its own line, but the
exported PDF flattens the dashes and surrounding text onto a single inline
line. This breaks editor↔PDF fidelity (a core promise of the export feature).

Output:
- A one-line fix in `assets/pdf-export.js` `parseMarkdown()`.
- A falsifiable Node behavior test that fails on the old code and passes on the
  fixed code.
</objective>

THE BUG (root cause, confirmed against current source):
- Editor preview — assets/md-render.js line 63: within a paragraph block,
  single newlines become `<br>`:
      var inner = applyInline(block).replace(/\r?\n/g, "<br>");
  → a dash line stays on its own line.
- PDF export — assets/pdf-export.js parseMarkdown() line 595: a paragraph's
  contiguous source lines are joined with a SPACE, destroying intra-paragraph
  line breaks:
      blocks.push({ type: 'para', text: paraLines.join(" ") });
  The para renderer (~line 1113-1145) then word-wraps that single space-joined
  string via doc.splitTextToSize(...), so the dashes render inline.

THE FIX (minimal, one line) — pdf-export.js parseMarkdown() line 595:
  change   paraLines.join(" ")   →   paraLines.join("\n")

WHY THE ONE-LINE FIX IS SAFE (verified against current source — keep in risk notes):
1. jsPDF's doc.splitTextToSize(text, width) treats an embedded `\n` as a
   FORCED line break (in addition to width wrapping). So the para renderer at
   line 1124 produces one sub-line per source line.
2. parseInlineBold() (line 421) tolerates embedded `\n`: a `**bold**` span
   aborts at a newline (charCodeAt===10 checks at lines 433 and 460); otherwise
   the `\n` falls through into the regular (non-bold) buffer at line 473. So
   paraStripped (line 1122-1123) contains the `\n` and it lives in a non-bold
   segment.
3. The renderer's offset accounting at line 1142 (`paraOff += subLine.length + 1`)
   consumes exactly ONE separator char per wrap/break boundary — identical
   whether the separator is the old space or the new `\n`. So
   clipSegmentsToRange() still maps wrapped sub-lines back onto bold segments
   correctly. The empty-clip safeguard at line 1138-1140 covers odd-whitespace
   edges.
4. Blank lines already terminate a paragraph in parseMarkdown (the `/^\s*$/`
   check at line 585), so paraLines never contains an empty element —
   join("\n") can never produce a spurious empty sub-line.
5. This is the literal-dashes behavior the user wants (matching preview) — NOT
   converting `----` into an `<hr>`. Do NOT add horizontal-rule handling.

RENDER PATH the test asserts against (assets/pdf-export.js):
- Each para sub-line (one per splitTextToSize output, line 1126-1144) calls
  drawSegmentedLine(clipped, y, BODY_SIZE) exactly once, at a single baseline y.
- drawSegmentedLine (line 776) issues one or more doc.text(text, x, y, ...)
  calls, all sharing that sub-line's baseline y.
  → distinct body baseline-y values == distinct rendered rows.
