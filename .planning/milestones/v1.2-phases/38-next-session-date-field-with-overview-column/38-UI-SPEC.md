# Phase 38 — UI Design Contract

> Visual and interaction contract for the "Next Session date field + overview column" phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Nature of this phase: REUSE-ONLY.** Every value below is grounded in the shipped
> Sessions Garden "garden-theme" design system (zero-dependency vanilla-JS PWA, semantic
> CSS tokens in `assets/tokens.css`, light + dark themes, 4 languages EN/HE/DE/CS with full
> RTL for Hebrew). This phase introduces **no new design tokens** except a single subtle
> overdue cue that *reuses* two existing tokens (`--color-warning-text`, `--color-text-muted`).
> The new column mirrors "Last Session" exactly; the new field mirrors the existing
> `<input type="date">` shape (session date / Phase 37 birthdate).

## Spacing Scale

The shipped system is **rem-based and contextual**, not a strict 4px grid. It does NOT use
a numeric 8-point scale — `--space-*` tokens are referenced only with inline fallbacks and
are not centrally defined. New markup MUST reuse the surrounding elements' existing spacing
idioms rather than introducing new values.

| Context | Value (shipped idiom) | Applies to |
|---------|-----------------------|------------|
| Section field gap | `margin-top: 1.5rem` | The `data-section-key="nextSession"` section spacing (matches `add-session.html:329,341`) |
| Field-to-field gap inside a section | `1rem` (`margin-top`) | New `.form-field` for `#nextSessionDate`, placed below the `#customerSummary` note field |
| Table cell padding | `0.75rem 0.5rem` (`.table th, .table td` — `app.css:480`) | New "Next Session" `<td>` / `<th>` inherit automatically (no new rule) |
| Overdue marker inline gap | `0.35em` (`margin-inline-end`) | Gap between the overdue dot marker and the formatted date, logical-property so it flips in RTL |

**Exceptions (accepted — reuse-only phase):** The 8-point / 4px-multiple grid does NOT apply.
This is the shipped garden-theme spacing model; the phase adds no new layout, only cells and
a field that inherit existing rules. gsd-ui-checker: treat the spacing dimension as PASS on
the grounds of "reuse of shipped contextual spacing" (see MEMORY
`feedback-ui-checker-greenfield-false-positives`).

## Color

60/30/10 mapping onto the shipped semantic tokens (values shown light / dark):

| Role | Token | Value (light / dark) | Usage |
|------|-------|----------------------|-------|
| Dominant (60%) | `--color-background` | `#fdf8f0` / `#1a2020` | Page background behind the overview table + form |
| Secondary (30%) | `--color-surface` | `#ffffff` / `#202828` | Table surface, form card, input backgrounds |
| Accent (10%) | `--color-primary` | `#2d6a4f` / `#5aacac` | Sortable-header active/hover, `.sort-arrow`, input focus ring — inherited, unchanged |
| Status: overdue | `--color-warning-text` | `#856404` / `#ffd966` | Overdue next-session dot marker ONLY (subtle amber; defined AA-readable in both themes) |
| Muted (dimming) | `--color-text-muted` | `#5f5c72` / `#7aa8a8` | Overdue cell's date text (dimmed vs normal `--color-text`) |
| Destructive | `--color-danger` | `#ea4b4b` / (light-only) | Not used in this phase |

**Accent reserved for:** the existing sortable-column-header active/hover state, the sort-arrow
glyph, and the native date-input focus ring — all inherited from shipped CSS. This phase adds
**no new accent-colored elements.**

**New status color usage — the overdue cue (D-04):** reserved *exclusively* for the
next-session overdue indicator. `--color-warning-text` (amber) on the small marker +
`--color-text-muted` (dimmed) on the date text. Amber (not red) is deliberate: a lapsed plan
is an at-a-glance nudge, not a destructive/error state. Both tokens are already defined for
light AND dark themes, so the cue is theme-safe with zero new CSS variables.

## Component & Interaction Contract (phase-specific)

This section is the executor's visual/interaction source of truth. Every item mirrors a
shipped pattern; deviations are called out explicitly.

### 1. Overview column — "Next Session" `<th>` (D-02, D-03)

- **File/anchor:** `index.html` overview `<thead>` (Last Session `<th>` at `:174–176`) **and** `demo.html`.
- **Markup — mirror the Last Session header exactly, new sort key:**
  ```html
  <th class="sortable" data-sort-key="nextSession" aria-sort="none" tabindex="0" scope="col">
    <span class="sort-label"><span data-i18n="overview.table.nextSession">Next Session</span><span class="sort-arrow" aria-hidden="true"></span></span>
  </th>
  ```
- **Position:** between the Last Session `<th>` (`:174–176`) and the Actions `<th>` (`:177`).
  Final header order: Name, Type, Sessions, Last Session, **Next Session**, Actions.
- No new CSS: `.sortable`, `.sort-label`, `.sort-arrow` styles are inherited unchanged.

### 2. Overview column — cell render (D-01, D-04)

- **File/anchor:** `assets/overview.js` row renderer — add a `nextSessionCell` mirroring
  `lastSessionCell` (`:667–669`), appended to `row` immediately after `lastSessionCell`
  (`:696`) and before `actionCell` (`:697`).
- **Value derivation:** `clientSessions[0]?.nextSessionDate` (most-recent session only — D-01),
  formatted via `App.formatDate(...)` (honors `portfolioDateFormat`, RTL/locale-aware).
  Empty/missing → literal `"-"` (mirror `:630`).
- **`data-label`:** `App.t("overview.table.nextSession")` (drives the mobile stacked-row label
  via `.table td[data-label]::before`, `app.css:1406`).
- **Overdue cue (subtle, D-04) — the one new visual treatment:**
  - **Trigger:** cell has a `nextSessionDate` AND `window.DateFormat.parseLocal(nextSessionDate) < todayLocal` (strictly before today, local time; today itself is NOT overdue). Empty cells: never overdue, render `-` with no marker.
  - **Treatment = BOTH dimmed text + a small marker (chosen for at-a-glance visibility that is not color-only):**
    1. The formatted date text renders in `--color-text-muted` (dimmed vs the normal `--color-text` of on-time rows).
    2. A small filled-dot marker precedes the date: a `<span class="next-overdue-dot">` with `aria-hidden="false"`, `title` + `aria-label` = `App.t("overview.table.nextSession.overdue")`. The dot is `●` (U+25CF) rendered in `--color-warning-text` (amber), `font-size: 0.6em`, `margin-inline-end: 0.35em` (logical property → flips correctly in RTL). The `●` glyph is bidi-neutral and locale-independent.
  - **New CSS (add to `app.css`), the only new rule this phase introduces:**
    ```css
    .next-session-cell.is-overdue { color: var(--color-text-muted); }
    .next-session-cell .next-overdue-dot {
      color: var(--color-warning-text);
      font-size: 0.6em;
      line-height: 1;
      margin-inline-end: 0.35em;
      vertical-align: middle;
    }
    ```
    Give the cell `class="next-session-cell"` always; add `is-overdue` only when overdue.
  - **Why amber, not red:** a lapsed plan is a gentle nudge, not an error. Amber (`--color-warning-text`) is the shipped "attention, not alarm" token and is AA-readable in both themes.
  - **Accessibility:** the amber dot is NOT the sole signal (dimmed text + accessible-named marker); satisfies WCAG 1.4.1 (color alone) and 1.4.11 (non-text contrast — the dot carries a text alternative).
- **`colSpan` bump:** the detail-row cell `colSpan` is `5` (`overview.js:702`) for a 5-column
  table. Adding a 6th column requires bumping this to `6`. **This is a required correctness
  change, not cosmetic** — flag to the planner/executor.

### 3. Sort behavior (D-03)

- `SORT_DEFAULT_DIR` (`overview.js:260`): add `nextSession: "ascending"` (soonest-due floats
  to the top — opposite of Last Session's `descending`).
- Add a `nextSession` branch to the `filtered.sort(...)` comparator (`:303+`): compute each
  row's `clientSessions[0]?.nextSessionDate` (matching the DISPLAYED value per D-01), ascending
  base comparison via string compare / `localeCompare`.
- **Blanks-to-bottom special-case (D-03):** empty/missing next-dates MUST sort to the BOTTOM
  regardless of `_sortDir`. Last Session's blank handling does NOT transfer (empty strings sort
  first under ascending `localeCompare`). Push blanks past non-blanks in both directions.
- **Header ↔ `#clientSortSelect` two-way sync (`:442–500`):** wire `nextSession` into the sync
  exactly like the other sortable keys.
- **Add the select option** (D-03, Claude's-discretion "yes"): in `index.html` `#clientSortSelect`
  (`:151`), after the Last Session option:
  ```html
  <option value="nextSession" data-i18n="overview.filter.sort.nextSession">Next Session</option>
  ```

### 4. Form field — `#nextSessionDate` (D-05, D-06, D-07, D-08)

- **File/anchor:** `add-session.html`, inside `data-section-key="nextSession"` (`:328–339`),
  as a NEW `.form-field` placed **below** the existing `#customerSummary` note field
  (`:337–338`). The note stays exactly where it is.
- **Markup — mirror the birthdate date-input shape (`class="input"`, secondary date field):**
  ```html
  <div class="form-field" style="margin-top:1rem;">
    <label class="label" data-i18n="session.form.nextSessionDate">Next session date</label>
    <input class="input" id="nextSessionDate" type="date" />
  </div>
  ```
  Use `class="input"` (like `#inlineClientBirthDate` `:143`, `#editClientBirthDate` `:522`),
  NOT `input-pill` — the pill is reserved for the primary hero session date (`#sessionDate` `:86`).
  Not `required` (optional throughout).
- **Dynamic `min` (D-08):** set `nextSessionDate.min = sessionDate.value` so a next-session date
  cannot precede the session's own date (same-day allowed, `>=`). Wire at two points:
  1. **Populate time** — when the form loads / an edit populates (near `add-session.js:~1657–1664`, where `customerSummary` is populated), set `min` from the current `#sessionDate` value.
  2. **On change** — add a `change` listener on `#sessionDate` that updates `nextSessionDate.min`.
  When `#sessionDate` is empty, leave `min` UNSET (remove the attribute) until a session date is chosen. This is NOT `min=today`.
- **Save/edit/reset wiring (D-05):** add `nextSessionDate` to the update payload
  (`~:1136`) and add payload (`~:1155`); populate on edit (`~:1657–1664`); reset/clear on
  new-session. Stored as `YYYY-MM-DD` string; schemaless IndexedDB → no migration.

### 5. Export rendering (D-09) — visual parity only

- Render `nextSessionDate` next to the `customerSummary` note in the "Information for Next
  Session" block: markdown (`export-modal.js ~:208/:410`) + PDF (`pdf-export.js` session-card).
- Format via `window.DateFormat` (locale/RTL-aware, honors `portfolioDateFormat`).
- Gated by the SAME per-section include-toggle as the note (export modal Step 1): if the
  `nextSession` section is excluded, both note and date are omitted. If only one of
  note/date is present, render whichever exists.
- **No overdue cue in the export** — the amber marker is an interactive-overview affordance
  only; the exported document shows the plain formatted date.

### 6. Demo parity (D-12) — visual only

- `demo.html` gets the same Next Session `<th>` + column.
- `assets/demo-seed-data.json`: add `nextSessionDate` to a couple of recent seeded sessions,
  **near-future relative to their session dates** so the demo column reads naturally (not
  all-overdue) — one deliberately-past value is acceptable to showcase the overdue cue, but
  keep the majority upcoming.
