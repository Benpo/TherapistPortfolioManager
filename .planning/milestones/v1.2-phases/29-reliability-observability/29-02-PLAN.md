<objective>
Replace the dead-end "Refresh page" of `showDBMigrationError()` (which re-runs the same failing migration forever) with an OBS-03 "reset & recover" escape hatch (per D-09, the trigger is the migration-failure path; this banner is also the landing spot for Phase 28's "couldn't finish automatically" integrity escalation, 28-CONTEXT D-12): an "Export backup now" action that exports AROUND the failure by opening the DB read-only at its existing un-upgraded version (resolving the D-09 known wrinkle), plus a hard-gated "Reset & recover" (checkbox affirmation + double-confirm, extra-emphatic when no export happened this session) that wipes and reloads into a fresh, usable DB. This is forward-looking insurance for a future migration (v7+) that throws and bricks only users who already have data.

Purpose: A failed IndexedDB migration can no longer trap a user in an unrecoverable "please refresh" loop, and their data can be exported before any wipe.
Output: Modified `assets/db.js` (banner replacement + read-only export-around-failure open + reset), `assets/backup.js` (a data-source-swappable ZIP path), `assets/app.css` (banner button styling), and 4-language strings.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user click → indexedDB.deleteDatabase | An irreversible wipe of all client/session data crosses here |
| failed-migration DB → read-only export | Reading around a broken upgrade; must not re-trigger the failing migration |
| escape-hatch code → network | Must make zero network calls (zero-network invariant) |

Must-NOT: never wipe without BOTH gates (frontmatter prohibition) |
