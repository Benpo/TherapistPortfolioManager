# Phase 28 — UI Design Contract

> Visual and interaction contract for the **two** user-facing surfaces this phase introduces/changes: the **integrity-mismatch nudge** and the **footer version marker**. Everything else in Phase 28 (SW lifecycle, cache naming, integrity tokens, CSP header migration, cache TTL) is non-visual infrastructure owned by plan-phase and is OUT OF SCOPE for this contract.
>
> This is a brownfield, zero-build, zero-npm vanilla-JS app with an established design system. This contract **reuses existing tokens and the existing `.db-error-banner` pattern** — it deliberately introduces no new design primitives. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Of This Contract

| In scope (spec'd here) | Out of scope (owned elsewhere) |
|------------------------|--------------------------------|
| The integrity-mismatch nudge (CONTEXT D-10/D-11/D-12) — appearance, 3 honest states, RTL, early-lifecycle string source | SW-lifecycle / `registration.update()` mechanics, cache deletion, activation sequencing |
| The footer version marker (CONTEXT D-09) — clean vs. mismatch rendering, RTL | The integrity-token stamping mechanism / where the version constant physically lives (VER-02, Claude's Discretion) |
| Copy for both surfaces in EN/HE/DE/CS | CSP `<meta>`→header migration (VER-04), cache-TTL change (VER-05) |
| Reuse of existing color / spacing / type / radius tokens | Removing `unsafe-inline` (deferred, out of scope for v1.2) |

**Central design principle (Ben, this session):** *the words must always match reality.* Every string the nudge shows must be true at the moment it is shown — truly fix it (online), tell the user to reconnect (offline), or admit it can't and point to real recovery (wedged). Never "refresh to complete" when a refresh won't complete anything.

---

## Locked Decisions (from 28-CONTEXT.md — do NOT re-litigate)

These are binding inputs to the contract, not open questions:

- **D-09** Honest footer: footer never claims a clean version it isn't coherently running; on mismatch it shows a subtle marker (`v1.2.0 ⚠`).
- **D-10** Mismatch nudge: small dismissible prompt whose button runs a GENUINE recovery (not a cosmetic `location.reload()`).
- **D-11** Promise gated on feasibility: online → "Refresh to complete"; offline → NO completion promise ("Update pending — reconnect to finish updating").
- **D-12** Honest escalation: if genuine recovery still can't clear the mismatch, the nudge degrades to honest guidance ("Couldn't finish automatically…") that hands off to Phase 29's reset & recover hatch + report-a-problem flow.
- This milestone ships as **v1.2.0** — the first real, trustworthy footer version (replaces the `v1.1.0` static placeholder at `assets/shared-chrome.js:8`).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla JS, zero-build, zero-npm, IIFE-global modules, files served verbatim) |
| Preset | not applicable |
| Component library | none — hand-rolled DOM via `innerHTML` / `createElement`; **reuse the existing `.db-error-banner` family** as the nudge's structural precedent |
| Icon library | none — inline Unicode glyph for the footer marker (`⚠`, U+26A0); no SVG icon needed for the nudge |
| Font | Rubik (self-hosted woff2: Regular 400, SemiBold 600, Bold 700) via `@font-face` in `assets/tokens.css`; CSS uses `var(--font-family, Rubik, system-ui, sans-serif)` |

**Token reuse rule (hard constraint):** the codebase defines *semantic* tokens (`--color-*`) in `assets/tokens.css :root`, but the **scale tokens** (`--space-*`, `--text-*`, `--radius-*`, `--font-family`) are NOT defined in `:root` — every existing usage carries an inline fallback, e.g. `var(--space-md, 16px)`. The nudge MUST follow the exact same `var(--token, fallback)` idiom that `.db-error-banner` / `.db-error-btn` already use (see `assets/app.css:2132–2178`). Do **not** invent bare tokens with no fallback, and do **not** hardcode raw values.

## Surface 1 — Integrity-Mismatch Nudge

### What it is
A small, dismissible, top-anchored prompt shown ONLY when the runtime integrity self-check detects a token disagreement (stale/partial "Frankenstein" cache, the v209 failure mode). It is **not** shown on a clean load. Its primary action runs a genuine recovery (`registration.update()` → activate new SW → delete stale caches → reload).

### Structural precedent — REUSE, don't reinvent
Build it as a new severity variant of the **existing `.db-error-banner` pattern** (`assets/db.js:399–449`, `assets/app.css:2132–2178`):
- Same top-fixed, full-width, centered, `role="alert"` banner shape.
- Same CTA button idiom as `.db-error-btn` (primary-filled, SemiBold, `var(--radius-sm, 6px)`).
- It is **dismissible** (unlike the DB banners) — add a close affordance; dismissal hides it for the session (it may legitimately re-surface on the next load if the mismatch persists, per D-12's "never a looped lie" — dismissal must NOT itself become a silent lie).
- Suggested class: `.integrity-nudge` with state modifiers `--online` / `--offline` / `--wedged` mirroring the existing `--blocked` / `--version` / `--migration` convention.

### Three honest states (the heart of this phase)

| State | Trigger | Color band | Has action button? | Promise it may make |
|-------|---------|------------|--------------------|---------------------|
| **Online / fixable** | mismatch detected AND `navigator.onLine` | info band (`--color-info-bg` / `--color-info-text`) | Yes — runs genuine recovery | "Refresh to complete" — TRUE (recovery can fetch fresh assets) |
| **Offline / pending** | mismatch detected AND offline | warning band (`--color-warning-bg` / `--color-warning-text`) | No completion button (a reconnect hint only) | NO completion promise — "Update pending — reconnect to finish updating" |
| **Wedged / can't fix** | genuine recovery already ran and the mismatch persists | danger band (`--color-danger-bg` / `--color-danger-text`) | Yes — but it points to recovery/report, not a false "refresh" | "Couldn't finish automatically…" → hands off to Phase 29 reset & recover + report-a-problem |

The state is chosen at render time from `navigator.onLine` and a "have we already attempted recovery this load?" flag. The button label and the body copy are bound to the state — they can never disagree with what the button actually does.

### Color band mapping (reuses existing semantic tokens — all dark-mode-safe)

| State | Background | Text | Dark-mode status |
|-------|-----------|------|------------------|
| online | `var(--color-info-bg, #cce5ff)` | `var(--color-info-text, #004085)` | light-only band today (matches `.db-error-banner--version`); acceptable — same precedent |
| offline | `var(--color-warning-bg, #fff3cd)` | `var(--color-warning-text, #856404)` | **dark overrides exist** (`#3a2f00` / `#ffd966`, WCAG-AA on `#202828`) — inherits automatically |
| wedged | `var(--color-danger-bg, #f8d7da)` | `var(--color-danger-text, #721c24)` | `--color-danger-*` is light-only **by design** (Phase 27 decision) — reserved for the moment-of-action; acceptable for the wedged escalation |
| CTA button | `var(--color-primary, #2d6a4f)` bg, `var(--color-text-inverse, #fff)` text | — | primary token has a dark value (`#5aacac`) — inherits |

**Reserved-color discipline:** the danger band is reserved for the genuinely-stuck (wedged) escalation only — it must NOT be used for the routine online/offline states. This honors the project's "red = moment of irreversible action / real failure" convention (Phase 27).

### Early-lifecycle string requirement (critical)
The integrity check may run **before `i18n.js` loads**. Therefore the nudge's strings MUST follow the established inline-string-object pattern — the same one `FOOTER_STRINGS` (`shared-chrome.js:59`) and `DB_STRINGS` (`db.js:153`) use: a literal `{ en, he, de, cs }` object colocated with the nudge code, with a `dbStr`-style getter that reads `localStorage.getItem('portfolioLang')`, falls back to the current lang's table, then to `en`, then to the key. Do **not** route nudge copy through `i18n.js` / `t()`. (If `i18n.js` happens to be loaded the inline strings still win — single source for this surface, no drift.)

### RTL requirement
- HE is RTL. The banner is centered, which is direction-agnostic, but any inline arrangement (text + button, close affordance, the `⚠`/reconnect glyph) MUST use CSS logical properties (`margin-inline`, `inset-inline-end`, `gap`) — matching the project's `html[dir=rtl]` convention (Phase 18). No physical `left`/`right`.
- The close affordance sits at `inset-inline-end` so it lands correctly in both LTR and RTL.
- Lang/dir are read from `localStorage` the same way `dbStr()` does; the banner must render correctly when injected before the page's `dir` is finalized (read `portfolioLang` and set the element's `dir` defensively, as DB_STRINGS-era banners do).

### Spacing (reuse db-error-banner values verbatim)
- Banner padding: `var(--space-md, 12px) var(--space-lg, 16px)`
- Inline gap (text ↔ button): `12px` (matches `.db-error-banner` `gap: 12px`)
- Button padding: `var(--space-xs, 2px) var(--space-md, 12px)`; `margin-top: var(--space-sm, 4px)`
- Z-index: `var(--z-banner)` (= 500) — sits above toast (400) and modals, correct for an alert.

## Surface 2 — Footer Version Marker

### What it is
The footer copy line (`shared-chrome.js:88`, `.app-footer-copy`) renders `© 2026 Sessions Garden · v{APP_VERSION}`. Clean state shows `v1.2.0`. On a detected integrity mismatch it appends a subtle marker: **`v1.2.0 ⚠`**.

### Rendering rules
- **Clean (default):** `v1.2.0` — no marker, no color change. The single source-of-truth version constant feeds this (VER-02); the hand-set semver is `1.2.0` for this milestone.
- **Mismatch:** append a single space + `⚠` (U+26A0). It is a *subtle* marker — same `.app-footer-copy` muted type, no red, no bold, no animation. It signals "this label may not be coherently running" without alarming a therapist mid-session. The nudge (Surface 1) carries the actionable message; the footer marker is the always-honest passive indicator.
- The marker is wrapped so it can carry an accessible label (e.g. `<span class="app-footer-version-warn" title="…" aria-label="…">⚠</span>`) — the title/aria text uses the same inline 4-language string source as the nudge.
- The footer **must never** show a clean `v1.2.0` while a mismatch is detected (D-09). If the check hasn't resolved yet, render clean (optimistic) and upgrade to the marker when/if the check reports a mismatch — never the reverse (never claim clean after detecting dirty).

### RTL
`.app-footer-copy` is centered. The `⚠` follows the version string in source order; with `dir=rtl` it will mirror naturally. Keep it inside the same text node/span so bidi reordering treats `v1.2.0 ⚠` as one unit; verify it does not detach awkwardly in HE (the version is Latin/Arabic-numeral, so it forms an LTR run inside the RTL line — acceptable, matches how `v1.1.0` already renders today).

## Spacing Scale

Declared values (project scale — multiples of 4; values are the inline fallbacks the codebase already uses):

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Button vertical padding, tight gaps |
| `--space-sm` | 8px | Compact spacing, button margin-top |
| `--space-md` | 16px | Banner vertical padding, default element spacing |
| `--space-lg` | 24px | Banner horizontal padding, section padding |
| `--space-xl` | 32px | Layout gaps |

Banner-specific inline gap: **12px** (text↔button), reused verbatim from `.db-error-banner`.

> **Banner-family fallback exception (executor MUST honor — this is the one place the canonical scale above does NOT apply).** The table above is the codebase-wide dominant scale (`var(--space-md, 16px)` and `var(--space-sm, 8px)` are each used ~13× across `assets/`). The `.db-error-banner` / `.db-error-btn` family — which this nudge inherits from verbatim — is a deliberate **tighter-fallback outlier**: it uses `var(--space-md, **12px**)`, `var(--space-xs, **2px**)`, `var(--space-sm, **4px**)`, and a hardcoded `gap: 12px` (see `assets/app.css:2132–2178`). Because none of the `--space-*` scale tokens are defined in `:root`, the *fallback literal you write is the value that actually renders*. The nudge MUST copy the banner family's fallbacks exactly so it sits flush with its sibling banners — do **not** "normalize" the banner's `12px` md-padding up to the canonical `16px`, and do **not** change `2px`/`4px` to `4px`/`8px`. The Surface 1 "Spacing (reuse db-error-banner values verbatim)" block is the binding source for these two surfaces; the canonical table is reference context only.

## Copywriting Contract

All copy lives in an inline `{ en, he, de, cs }` string object colocated with the nudge (NOT in `i18n.js`), per the early-lifecycle requirement. EN is canonical below; HE/DE/CS are written natively (not machine-translated word-for-word) following the project's translation convention. HE must be gender-neutral (Phase 18 convention). **The verb tense/promise of each string is load-bearing — it must match what the code actually does in that state.**

### Nudge — Online / fixable state (info band)

| Element | EN copy |
|---------|---------|
| Body | This app didn't update cleanly. Refresh to finish updating. |
| Primary CTA | **Refresh to complete** |
| Dismiss | (close ✕ — dismiss for now) |

*Why this wording:* online, the genuine recovery CAN fetch fresh assets and clear the stale cache, so "complete" is a true promise.

### Nudge — Offline / pending state (warning band)

| Element | EN copy |
|---------|---------|
| Body | Update pending — reconnect to finish updating. |
| Primary CTA | (none — no completion button; reconnect is the only path) |
| Dismiss | (close ✕ — dismiss for now) |

*Why this wording:* offline there is nothing to complete with, so NO completion promise is made (D-11). No "Refresh" button, because a refresh offline would re-serve the same stale cache and the words would lie.

### Nudge — Wedged / can't-fix state (danger band)

| Element | EN copy |
|---------|---------|
| Body | Couldn't finish updating automatically. Your data is safe. |
| Primary CTA | **Recover the app** → hands off to Phase 29 reset & recover (OBS-03) |
| Secondary | Report a problem → Phase 29 report-a-problem flow (OBS-02) |
| Dismiss | (close ✕ — dismiss for now) |

*Why this wording:* the recovery already ran and failed; promising another "refresh to complete" would be the looped lie D-12 forbids. It admits the failure, reassures about data safety (matching the existing `migrationFailed` reassurance idiom in `DB_STRINGS`), and points to the real recovery hatch. **Phase 28 degrades gracefully to honest guidance; the actual reset & recover hatch and report flow are built in Phase 29** — this phase's CTAs may link/stub to them.

### Footer marker

| Element | EN copy |
|---------|---------|
| Clean | `v1.2.0` |
| Mismatch | `v1.2.0 ⚠` (marker `aria-label` / `title`: "Version may not be running correctly — see the update notice above.") |

### 4-language strings (to be authored at plan/execute time, following DB_STRINGS)

The executor authors HE/DE/CS for each EN string above in a colocated inline object. Reference idioms already in the codebase for tone/encoding:
- "Your data is safe" → mirror `DB_STRINGS.migrationFailed` ("…Deine Daten sind sicher" / "…הנתונים שלך בטוחים" / "…Tvoje data jsou v bezpeci").
- "Refresh" verb → reuse `DB_STRINGS.refresh` (Aktualisieren / רענון / Obnovit) for consistency.

### Destructive actions in this phase
- **The genuine recovery (online state)** deletes stale caches and reloads — but it is **non-destructive to user data** (IndexedDB is never touched; only unsaved form text was ever at risk, and apply-on-next-navigation per D-05 already avoids yanking a typing session). **No confirmation dialog required** for the online "Refresh to complete" action — it is safe and expected.
- **The wedged "Recover the app"** CTA hands off to Phase 29's reset & recover hatch, which owns its OWN destructive-confirmation contract (OBS-03). Phase 28 only links to it; it does not perform the reset itself.

## Accessibility Notes

- Nudge: `role="alert"` (matches the DB banners) so screen readers announce the mismatch. The state-specific body text is the live region.
- Nudge dismiss affordance: real `<button>` with an accessible label, at `inset-inline-end`.
- Footer `⚠`: not announced as a bare glyph — wrap with `aria-label` text from the inline string source; do not rely on color or the glyph alone to convey state (the nudge carries the words).
- Color is never the sole signal: each nudge state pairs a distinct band with distinct, honest *words*.
