<objective>
Build the single source-of-truth versioning foundation for VER-02: a new `assets/version.js` module
exposing the hand-set semver (`1.2.0`) and a deploy-stamped integrity token, readable from BOTH the
page (`window`) and the service-worker (`self`) global scopes. Wire the SW `CACHE_NAME` to derive
from that token via `importScripts`, and add a deliberate deploy-time stamp step that replaces the
token placeholder with the git short-hash. This kills the v209 "forgot to bump → stale cache" failure
class at the root (D-02) and gives the footer and the integrity check (Plan 04) one value to read.

Purpose: VER-02's derivation engine — one constant must drive the footer label, the SW cache name,
and the integrity-check reference. This plan builds the constant and the cache-name + deploy wiring;
the footer and integrity surfaces consume it downstream (Plans 03/04).
Output: `assets/version.js` (NEW), modified `sw.js`, modified `.github/workflows/deploy.yml`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → deployed assets | The deploy GitHub Action transforms `version.js` before publishing; a wrong/empty stamp would mis-derive CACHE_NAME |
| service worker → page | The SW reads the shared `version.js` token via importScripts to name its cache |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | Tampering | deploy.yml sed stamp | mitigate | Stamp acts on the staging copy only; uses the existing `${GITHUB_SHA::7}` (no new secret/input); if the sed no-ops, the 'dev' fallback yields a benign distinct cache name (degrades safe, never silently same-as-prod) |
| T-28-02 | Denial of Service | SW importScripts('/version.js') | mitigate | version.js is added to PRECACHE_URLS so installed users always have it cached; it is a tiny synchronous local file, no network fetch (VER-06) |
| T-28-03 | Information Disclosure | version.js contents | accept | Exposes only a semver + a git short-hash (already public in the deploy commit message); no PII, no secret |
| T-28-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan — zero-build, zero-npm project; nothing to verify. No `[ASSUMED]`/`[SUS]` packages introduced |

No high-severity threat is left unmitigated.
