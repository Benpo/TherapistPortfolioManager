<objective>
Extend the existing `App.installNavGuard` infrastructure (built Phase 22 Plan 12)
to cover the two unprotected nav targets on `add-session.html`:

- The top-of-page brand-link (logo) — currently navigates to `./index.html` on
  click without warning.
- The bottom "Back to Overview" button — currently navigates to `./index.html`
  on click without warning.

Both must fire the same `confirmDialog` ("Leave this session?" / unsaved changes)
when `window.PortfolioFormDirty()` returns true. On clean form, navigation
proceeds silently — same UX contract as the existing settings-gear guard.

Bug discovered during Plan 06 UAT (2026-05-14): therapist filling in a new
session loses all data if they accidentally click the logo or back-to-overview
mid-edit.
</objective>

<tasks>

<task type="auto">
  <name>Task 1: install nav guard on brand link (global, via initCommon) + back-to-overview link (page-local)</name>
  <action>
    Add `initBrandLinkGuard()` to `assets/app.js` mirroring the pattern of
    `initSettingsLink()`. Call it from `initCommon()` after `initSettingsLink()`.
    The guard's `isDirty` callback reads `window.PortfolioFormDirty` lazily
    (a function reference, not a snapshot) so the check evaluates at click time.

    immediately after `window.PortfolioFormDirty` is exposed), install the

    button at the bottom of the form.

    Both installs are idempotent (use `_navGuardInstalled` flag) so repeated
    DOMContentLoaded / initCommon calls don't double-bind.

    No new i18n keys needed — `session.leavePage.*` (4 keys × 4 locales) already
    exist from Phase 22 Plan 12.
