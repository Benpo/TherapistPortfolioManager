<objective>
Fix the silent photo-update loss bug Ben hit on 2026-05-14 when re-uploading
a client photo from inside the edit-client modal on add-session.html.

The chain (a→b→c) was:
- (a) Crop modal opened BEHIND the edit-client modal — invisible to the user.
- (b) User had to close (Cancel) the edit-client modal to reveal the crop modal.
- (c) Once edit-client was closed, confirming the crop fired the onSave
  callback that updated a closure variable + preview img on the now-detached
  edit-client form — but the edit-client Save handler was no longer mounted,
  so the photo was never persisted to IndexedDB.

Single root cause: both `#cropModal` and `#editClientModal` use the `.modal`
class with `z-index: var(--z-modal)`. Stack order falls through to DOM source
order. `#cropModal` is at HTML line 387, `#editClientModal` at line 509 —
edit-client comes later in source so it renders on top of crop.

Fix the z-index. Once crop sits above edit-client, the user never needs to
close edit-client during the crop — they see both at once, confirm or cancel
crop, and edit-client's Save handler is still mounted to persist the change.
</objective>
