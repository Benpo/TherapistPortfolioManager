## Anti-pattern protocol (BLOCKING — per .continue-here.md)

Two rules apply to every task in this plan, sourced from the 24-01 incident:

1. **No grep-gate-only verification for runtime-behavior code.** Every runtime-behavior surface in this plan has its pure-function-extractable logic identified and unit-tested. The four pure-function helpers (`isTriggerUnique`, `validateImportPayload` + `detectImportCollisions`, `filterSnippetList`, `isModifiedSeed`) MUST be extracted as exported helpers and have failing tests written FIRST (RED) before implementation. Grep gates verify shape only (e.g., `addSnippetBtn` exists in HTML) — they do NOT certify runtime behavior.

2. **Identifier-resolution-chain trace for scope-crossing edits.** The Settings UI module accesses several namespaces across scope boundaries: `window.App.*` (different IIFE), `window.PortfolioDB.*` (different IIFE), `window.Snippets.*` (different IIFE), `window.SNIPPETS_SEED` (top-level frozen array). Every such cross-scope access in Plan 05 code MUST use the explicit `window.` prefix. The per-task `<test_strategy>` block documents the resolution chain for each identifier touched.

Marking a task `completed` requires:
- Pure-function helper extraction tasks: grep gate passes AND `node tests/24-05-<topic>.test.js` exits 0 AND output is shown verbatim in the commit/SUMMARY.
