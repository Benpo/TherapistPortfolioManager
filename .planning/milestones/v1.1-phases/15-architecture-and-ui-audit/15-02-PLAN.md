Read all files listed in read_first. Produce a structured report with FIVE sections:

**Section 1: PWA Manifest and Service Worker Audit (per D-08, D-09)**

Check manifest.json:
- Does `start_url` point to `index.html` (the app), NOT `landing.html`? Document the current value.
- Are all required PWA fields present? (`name`, `short_name`, `start_url`, `display`, `background_color`, `theme_color`, `icons`)
- Do all icon paths in the manifest point to files that actually exist on disk? (Check each path)
- Is `display` set to `standalone` for app-like experience?
- Does `scope` correctly encompass the app pages but not interfere with landing page?

Check sw.js:
- List ALL files in the service worker cache array
- Compare against ALL files in the project (HTML, JS, CSS, images, fonts)
- List every file that exists on disk but is NOT in the sw cache
- Determine: does the app (all pages after license gate: index.html, add-client.html, add-session.html, sessions.html, reporting.html, and their JS/CSS dependencies) work fully offline?
- Does the landing page need to work offline? No (per D-09), but document whether it's cached anyway
- Check the cache versioning strategy: how does the SW handle updates? Does it use a version string? Is it bumped consistently?
- Check the SW registration: which HTML pages register the SW? Is it registered on ALL app pages or only some?

**Section 2: Purchase-to-Usage Customer Journey Map (per D-10)**

Trace the COMPLETE flow a new customer takes:
1. Discovers landing page (landing.html)
2. Reads features, reviews, pricing
3. Clicks "Buy" button -- what URL does it link to? Is it a placeholder or real Lemon Squeezy URL?
4. Lemon Squeezy checkout process -- what happens? How does the customer receive their license key?
5. Customer has license key -- where do they go? Is there a redirect back? An email with instructions?
6. Customer enters license key on license.html -- how does validation work? What API does license.js call?
7. After valid license -- what page loads? Is there a disclaimer gate?
8. Disclaimer acceptance -- what happens next? Does the app load?
9. App is now usable -- is it offline-ready at this point?

For EACH step, document:
- Current implementation status (implemented / placeholder / missing)
- File and line number where it's handled
- If missing: what would need to be built

Rate each gap: BLOCKING (cannot launch without this) or IMPROVEMENT (nice to have).

**Section 3: CloudFlare Pages GDPR Compliance (per D-11, D-12)**

Research and document:
- Does CloudFlare Pages inject any cookies, tracking pixels, or analytics by default?
- Does CloudFlare Pages process visitor data (IP addresses, request logs) and where is that data stored?
- Is CloudFlare Pages subject to EU-US Data Privacy Framework?
- What does CloudFlare's DPA (Data Processing Agreement) cover?
- Can CloudFlare Pages be configured for EU-only data processing?
- Alternatives if CF Pages is not GDPR-safe: Netlify (EU region), Hetzner Pages, Bunny CDN, self-hosted

Additionally, audit the ENTIRE surface for GDPR concerns (per D-12):
- Website: does landing.html, impressum.html, datenschutz.html load any external resources (fonts, scripts, images from CDNs)?
- Payment flow: what data does Lemon Squeezy collect? Is LemonSqueezy a GDPR-compliant processor?
- App: does the app make ANY external network calls? (should be zero except license validation)
- Check all HTML files for external script/link/img tags loading from third-party domains
- Check all JS files for fetch/XHR calls to external URLs

Produce a clear COMPLIANT / NEEDS ATTENTION / NON-COMPLIANT verdict for each surface area.

**Section 4: Legal Document Status (per D-13, D-14, D-15)**

Read impressum.html and datenschutz.html. For each:
- Is the content real or placeholder? Quote the actual content.
- If placeholder: list EXACTLY what real content is needed (business name, address, email, phone, tax ID, etc.)
- Are these pages available in all 4 languages?
- Does the globe language switcher work on these pages?
- Are there any data flows discovered in the architecture audit that the Datenschutz should mention? (e.g., Lemon Squeezy license validation, service worker behavior, IndexedDB data handling)

Note: Legal document WRITING happens in a follow-up phase (per D-15). This section only documents what exists and what's missing.

**Section 5: Lemon Squeezy Integration Readiness (per D-16, D-17)**

Read license.js and landing.html/landing.js. Document:
- Current checkout URL: what value is in the "Buy" button href? Is it a placeholder `[YOUR_*]` or a real URL?
- License validation: does license.js call the Lemon Squeezy API? What endpoint? What format does it expect?
- What happens if validation fails? (offline, invalid key, expired key)
- Does the app store the license key? Where? (localStorage, IndexedDB?)
- Is there any webhook handling? (probably not for a static site, but check)
- What needs to happen to wire up a real Lemon Squeezy product: list specific steps (create product, get checkout URL, configure license key format, test validation endpoint)

Format findings with severity tags: BLOCKING / HIGH / MEDIUM / LOW / INFO.

Write the complete report to `.planning/phases/15-architecture-and-ui-audit/15-02-REPORT-pwa-journey-gdpr.md` with a summary table at the top.
