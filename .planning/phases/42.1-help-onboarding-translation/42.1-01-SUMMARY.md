# Phase 42.1 Plan 01: Per-Locale Integrity Gates (RED-first) Summary

Authored the two NEW per-locale integrity gates — one for help content, one for changelog content — RED-first in Wave 0, before any translation lands, so the he/de/cs translation plans have a machine-checkable, non-vacuous target mirroring `tests/39-help-integrity.test.js` and `tests/42-changelog-integrity.test.js`.

## What was built

**`tests/42_1-help-integrity.test.js`** — parameterized over `LOCALES = ['he','de','cs']`. For each locale it loads `i18n-en.js` + `i18n-<loc>.js` + `help-content-en.js` + `help-content-<loc>.js` into a fresh `vm` sandbox and cross-checks the locale corpus against EN:
- (a) Structure parity: identical section `id`s in identical order; identical `group` / `featured` / topic `id`s / topic `priority` / topic `covers[]`; exactly one featured section; both groups (`session-loop`, `technical`).
- (b) `{ui:key}` tokens resolve to non-empty `window.I18N.<loc>` strings, and the `{ui:}` token set is byte-identical to EN per topic (tokens never translated, D-23).
- (c) Locale-appropriate forbidden clinical terms (HE `מטופל`/`טיפול`, DE `Patient*`/`Behandlung`, CS `pacient`/`léčb`/`léčen`) — distinct regexes per locale, NOT the EN `/\bpatient\b/`.
- (d) No emoji (reused EMOJI_RE shape).
- (e) No control chars other than `\n`; rejects C0/C1 and U+202A–U+202E / U+2066–U+2069 bidi-control.

**`tests/42_1-changelog-integrity-locale.test.js`** — same locale loop. For each it loads `changelog-content-en.js` + `changelog-content-<loc>.js` and asserts:
- (a) Schema/order parity: same entry count; `version` + `anchor` byte-identical to EN in identical reverse-chron order; v1.0.0 entry is `origin:true` with no highlights/categories; content entries have non-empty `lede` + `highlights[2–4]`.
- (b) Category-key set identical to EN per content entry; each present category a non-empty `string[]`, keys ⊆ {new,improved,fixed}.
- (c) Every entry has a non-empty `date` — **no** cross-locale date equality assertion (localized month, D-07).
- (d) No emoji + no control/bidi chars in lede/highlight/category strings.
- (e) Locale-appropriate forbidden clinical terms (same design as the help gate).

Both files use the `tests/run-all.js` contract (custom `test()` harness, `process.exit(failed===0?0:1)`) and auto-join the suite with zero registration.

- Acceptance greps confirmed: non-comment `fs.existsSync` count = 3 per file; `Behandlung` present; HE (`מטופל`) + CS (`pacient`) clinical terms present in regexes; `LOCALES === ['he','de','cs']`; no cross-locale `date` equality; origin check keys on `1.0.0`.

## Deviations from Plan

**None on logic.** One implementation-hardening detail worth recording: the control/bidi-char regex was authored escape-only (`\uXXXX`). The initial Write embedded literal control/bidi code points in the character class (invisible in source, flagged by the injection scanner and directly contrary to the plan's Phase-29 NUL/bidi-injection intent); both were rewritten to escape-only form before commit. No behavior change — the regex matches the same code points — but the source no longer contains any literal control character. This aligns with the plan directive to keep forbidden literals confined to regex definitions and to guard against control-char injection.

## Known Stubs

None. Both files are complete, live gates. They are RED by design (the locale content files they validate do not exist yet); Plans 02–10 turn them GREEN as translations land.

## Threat Flags
