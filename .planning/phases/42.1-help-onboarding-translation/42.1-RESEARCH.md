## Summary

This is a **content-and-wiring** phase, not a feature phase: no new libraries, no new UI, no new behavior. Four translation surfaces authored in EN across Phases 39–42 must read natively in Hebrew, German, and Czech. The corpus splits into two mechanically different halves:

1. **Content-data files** (help body + changelog) — currently EN-only globals (`window.HELP_CONTENT_EN`, `window.CHANGELOG_CONTENT_EN`). The changelog already has a **working locale-merge + EN-fallback loader** (`changelog.js:localeEntries()`); the help page does **not** yet (`help.js` reads `HELP_CONTENT_EN` directly). Deliverables: new `help-content-{he,de,cs}.js` + `changelog-content-{he,de,cs}.js` sibling files, a help-side locale merge mirroring the changelog's, static `<script>` tags in the consuming HTML, and `sw.js` precache entries.

2. **UI-chrome i18n keys** (welcome overlay + tour) — these already exist as HE/DE/CS values inside `i18n-{he,de,cs}.js`, but they are in three different states: (a) **stale drafts** written against OLD EN that Ben has since rephrased (11 tour keys + the welcome subtitle), (b) **empty stubs** to author (`help.welcome.subtitle2`), and (c) **drafted-but-ungated** values awaiting only the native-speaker pass. No new keys need inventing — the key contracts are frozen by existing parity gates.

**Primary recommendation:** Structure the phase as *content-file authoring* (help + changelog locale files + loader wiring + precache + per-locale integrity tests) running in parallel with *i18n-chrome re-derivation* (tour stale keys + welcome keys, then a whole-corpus native-speaker gate per locale), converging on a single Sapir Hebrew read + full-suite green before hand-off to Phase 43. Translators must **preserve `{ui:key}` tokens, `{n}`/`{total}`/`{screen}`/`{X.Y}`/`{V}` placeholders, and the `\n\n` paragraph break verbatim** — these are the highest-risk mechanical failure points.

### Pattern 1: Content-data locale sibling + EN-fallback merge (THE canonical pattern)
**What:** Each content-data file registers a locale-suffixed global (`window.HELP_CONTENT_HE`, `window.CHANGELOG_CONTENT_HE`). The loader looks up `window["<NAME>_" + lang().toUpperCase()]` and falls back per-entry to EN when a locale array (or an individual entry) is missing.
**When to use:** Both the help body and the changelog. The changelog already implements it — **copy its shape for the help page.**
**Example (the existing, working changelog merge — replicate for help):**
```javascript
// Source: assets/changelog.js:59-67 [VERIFIED: codebase]
function localeEntries() {
  var en = window.CHANGELOG_CONTENT_EN || [];
  var loc = window["CHANGELOG_CONTENT_" + lang().toUpperCase()];
  if (!loc || !Array.isArray(loc)) return en;          // whole locale missing → all EN
  return en.map(function (base) {
    var match = loc.filter(function (e) { return e && e.version === base.version; })[0];
    return match || base;                              // per-entry EN fallback (D-16)
  });
}
```
For help, the analog keys on **section id** (help sections have stable `id`s), producing a locale-merged `SECTIONS` array inside `help.js:render()`. EN remains the canonical, always-complete ordering; each section is swapped for its same-`id` localized counterpart when present.

### Pattern 2: `{ui:key}` tokens are NOT translated (D-23 live-label interpolation)
**What:** In the help body, every quoted on-screen label is a `{ui:key}` token (e.g. `{ui:settings.action.save}`). `help.js:interpolateUiNodes()` resolves it at render time to the **current live i18n label** — so a Hebrew UI already shows Hebrew button names even under EN help text. `[VERIFIED: help.js:95-110, help-content-en.js header comment]`
**Consequence for translators:** In `help-content-he/de/cs.js`, the translator translates only the surrounding prose and **leaves every `{ui:...}` token byte-for-byte identical to EN.** The token auto-resolves to the correct locale label. This dramatically reduces the surface that needs translating AND guarantees label consistency.
**Gate:** The per-locale integrity test must re-run the "every `{ui:key}` resolves against `window.I18N.<loc>`" check (mirror `39-help-integrity.test.js:109-125`).

### Pattern 3: Chrome keys resolved live via `App.t()` — engine unchanged
**What:** Welcome overlay and tour read keys from `window.I18N.<lang>` via `App.t()`. Switching locale fires `app:language`; help.js/changelog.js/tour.js already re-render. `[VERIFIED: help.js:456, changelog.js:148, 41-lang-rerender.test.js]`
**Consequence:** Translating tour/welcome = editing string values in `i18n-{he,de,cs}.js`. No renderer changes.

### Anti-Patterns to Avoid
- **Forking the changelog data into a second What's-New source.** CHLG-03 forbids it. The popup and page read the SAME per-locale array — fix `whats-new.js` to do the locale lookup, don't create a popup-only file. `[VERIFIED: whats-new.js reads CHANGELOG_CONTENT_EN only]`
- **`innerHTML` with translated content.** Both renderers build every dynamic node via `createElement + textContent` (XSS trust boundary T-39-06/T-42-01). Locale files are data, but the render path is already safe — do not introduce `innerHTML` for RTL or any reason.
- **Translating placeholder/interpolation tokens.** `{n}`, `{total}`, `{screen}`, `{X.Y}`, `{V}`, `{ui:...}`, and the `\n\n` break must survive verbatim (existing parity gates assert `{screen}`/`{n}`/`{total}`; add `\n\n` + `{V}`/`{X.Y}` assertions — see Pitfalls).
- **Manual CACHE_NAME bump.** `CACHE_NAME` derives from the deploy-stamped `INTEGRITY_TOKEN`; editing `PRECACHE_URLS` needs NO manual cache-version bump. `[VERIFIED: sw.js:26 + memory reference-pre-commit-sw-bump]`

## Common Pitfalls

### Pitfall 1: What's-New popup silently stays English (the confirmed asymmetry)
**What goes wrong:** `changelog.js` locale-merges (`localeEntries()`), but `whats-new.js` reads `window.CHANGELOG_CONTENT_EN` **directly** (`whats-new.js:30,75`) and renders `entry.highlights` from it. Result: after translating the changelog, the full changelog PAGE reads natively but the What's-New POPUP still shows English highlights. `[VERIFIED: whats-new.js:73-85]`
**Why it happens:** The popup predates any locale data and was written EN-only; the asymmetry is real (Ben's decision #2 flagged it, and source confirms it).
**How to avoid:** Add a locale lookup to `whats-new.js` `entries()`/`entryFor()` — read `window["CHANGELOG_CONTENT_" + lang.toUpperCase()]` with EN fallback (same shape as `changelog.js:61`), resolving `lang` from `portfolioLang` the way `whats-new.js:53` already does. The popup's title token `{X.Y}` (`whatsNew.title`) is already localized via the i18n dict — only the `highlights` array needs the data-locale fix.
**Warning signs:** A test that renders the popup in HE and asserts a Hebrew highlight string will stay RED until this is fixed.

### Pitfall 2: The `\n\n` paragraph break in `help.tour.step.help.body` gets dropped
**What goes wrong:** EN `help.tour.step.help.body` now carries a literal `\n\n` break, rendered via `white-space: pre-line` on `.sg-tour-body` (tour.css). Bodies are `textContent`-injected, so `\n` is the ONLY break mechanism. A translator who drops it loses the paragraph structure. `[VERIFIED: commit adf33df pins this; STATE.md 260709-tc4]`
**How to avoid:** The HE/DE/CS `help.tour.step.help.body` values must include a `\n\n` at the equivalent point. Add an explicit assertion to the tour parity test: `requireToken('help.tour.step.help.body', '\n\n')` across all locales (mirrors the existing `{screen}`/`{n}`/`{total}` token guard at `41-tour-i18n-parity.test.js:168-182`).

### Pitfall 3: Locale forbidden-words differ from EN "patient/treatment"
**What goes wrong:** `39-help-integrity.test.js:212` forbids `/\bpatient\b/i` and `/\btreatment\b/i` — English words. A German translation might use "Patient"/"Behandlung"; Hebrew/Czech have their own clinical words. A naive per-locale mirror that keeps the EN regex would pass vacuously while the forbidden concept slips in.
**How to avoid:** Each per-locale integrity test's terminology check must use **locale-appropriate forbidden terms** (DE: Patient/Behandlung; CS: pacient/léčba/léčení; HE: the clinical-register equivalents). Ground the correct "approved vs forbidden" vocab in the existing shipped locale files (the app already renders client/session correctly in each) and confirm with the native-speaker gate. This is a genuine design decision for the test author, not a copy/paste.

### Pitfall 4: Structural drift between locale content file and EN
**What goes wrong:** A translator reorders sections, renames an `id`, or drops a topic — breaking the merge (which keys on `id`/`version`) and the deep-link contract.
**How to avoid:** Per-locale integrity tests must assert **structure parity with EN**: same section ids in same order, same topic ids, same `covers`, `{ui:key}` tokens present and resolving, exactly one featured section, both groups present. Only `title`/`body`/`lede`/`highlights`/category strings may differ. (Mirror `39-help-integrity` schema checks + add an EN-parity cross-check.)

### Pitfall 5: New content files missing from precache → help/changelog break offline
**What goes wrong:** `help-content-he.js` etc. added as `<script>` tags but not to `sw.js` PRECACHE_URLS → an installed PWA offline can't load them → help renders EN-only (or the changelog page 404s a script). HELP-07/CHLG-01 require full offline.
**How to avoid:** Add all 6 new files to `PRECACHE_URLS` (alongside the existing `help-content-en.js` at sw.js:95 and `changelog-content-en.js` at sw.js:111). Extend `tests/39-help-precache.test.js` / `42-precache.test.js` to assert the new locale files are precached. CACHE_NAME auto-rolls — no manual bump. `[VERIFIED: sw.js:48-51,95,111]`

### Pitfall 6: RTL correctness for Hebrew content

## Code Examples

### Help-page locale merge to ADD in help.js (mirror of the changelog pattern)
```javascript
// New helper in assets/help.js, called at the top of render() to replace the
// direct `var sections = window.HELP_CONTENT_EN;`. Keys on section id, per-section
// EN fallback (D-16 parity with changelog.js:59-67). [pattern VERIFIED against changelog.js]
function localeSections() {
  var en = window.HELP_CONTENT_EN || [];
  var lang = (window.App && App.getLanguage) ? String(App.getLanguage() || "en") : "en";
  var loc = window["HELP_CONTENT_" + lang.toUpperCase()];
  if (!Array.isArray(loc)) return en;                    // whole locale missing → all EN
  return en.map(function (base) {
    var match = loc.filter(function (s) { return s && s.id === base.id; })[0];
    return match || base;                                // per-section EN fallback
  });
}
```
> Note: help.js already re-renders on `app:language` (`help.js:456`), so the live language switch works once `render()` sources from `localeSections()`.

### What's-New locale fix in whats-new.js

// Replace the EN-only entries() at whats-new.js:73-77 with a locale-aware lookup.
function entries() {
  var lang = (lsGet('portfolioLang') || window.I18N_DEFAULT || 'en').toUpperCase();
  var loc = null, en = null;
  try { loc = window['CHANGELOG_CONTENT_' + lang]; } catch (e) {}
  try { en = window.CHANGELOG_CONTENT_EN; } catch (e) {}
  var base = Array.isArray(en) ? en : [];
  if (!Array.isArray(loc)) return base;

  });
}
