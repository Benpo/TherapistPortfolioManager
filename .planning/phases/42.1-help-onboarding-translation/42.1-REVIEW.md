## Warnings

### WR-01: No static gate for the help-content locale `<script>` tags on help.html â€” the exact failure mode T-42-V5 exists to prevent

**File:** `help.html:151-156`, `tests/42-whats-new-gating.test.js:204-236`
**Issue:** Phase 42.1 added a dedicated static tag gate (T-42-V5) for the changelog locale siblings on all nine popup pages, precisely because the fixture-driven render test "CANNOT see a missing page `<script>` tag" â€” a dropped tag silently reverts users to EN with every other gate green. The identical failure mode exists for `help-content-{he,de,cs}.js` on `help.html` and has **no** gate: `tests/42_1-help-integrity.test.js` loads the files from disk (never the page), `tests/39-help-precache.test.js` only scans `sw.js`, and `help.js localeSections()` degrades silently to all-EN when `HELP_CONTENT_<LOC>` is undefined. A future edit that drops or typos one of the three tags at help.html:153-155 ships an English-only Help center for that locale while the whole 164-test suite stays green â€” this repo's documented "shape gates verify shape not behavior" false-GREEN class.
**Fix:** Add a static assertion (in `tests/42_1-help-integrity.test.js` or a small sibling) mirroring T-42-V5's shape:

```js
// help.html must load all three help-content locale siblings (after -en, before help.js)
var src = fs.readFileSync(path.join(REPO_ROOT, 'help.html'), 'utf8');
['help-content-he.js', 'help-content-de.js', 'help-content-cs.js'].forEach(function (sib) {
  assert.ok(src.indexOf(sib) !== -1, 'help.html missing <script> for ' + sib);
  assert.ok(src.indexOf(sib) > src.indexOf('help-content-en.js'), sib + ' must load after help-content-en.js');
  assert.ok(src.indexOf(sib) < src.indexOf('assets/help.js'), sib + ' must load before help.js');
});
```

### WR-02: What's-New popup has no EN-fallback direction handling under RTL â€” the analog of help's BLOCKER-2 fix is missing

**File:** `assets/whats-new.js:80-91` (entries merge), `assets/whats-new.js:155-162` (highlight render)
**Issue:** This phase fixed EN-fallback-under-RTL for the help center (`localeSections()` marks `.is-en-fallback`; `help.css` forces those cards LTR under `[dir=rtl]` â€” BLOCKER 2) but not for the popup that uses the same fallback pattern. When `entries()` falls back to EN â€” the advertised path when `CHANGELOG_CONTENT_HE` fails to load (script 404, partial deploy, stale SW cache miss) or a future EN entry lands before its HE sibling â€” the English `lede`/`highlights` render as `<li>` text inside an `[dir=rtl]` document with no direction override: Latin sentence punctuation jumps to the wrong edge, the exact scrambling BLOCKER 2 was raised for. `changelog.js` (out of this review's file scope) has the same gap in `localeEntries()`. Reachability is low today because the per-locale integrity gate enforces full version parity, so this is a latent robustness defect in the fallback path itself, not a live rendering bug.
**Fix:** In `entries()`, track which merged entries are EN-fallback (mirror `help.js` `_fallbackIds`), and in `show()` set `popup.dir = 'ltr'` (or add an `is-en-fallback` class styled `direction:ltr; text-align:start` in app.css) when the rendered entry is a fallback and the document direction is RTL:

```js
var isFallback = !m;                         // in the entries() map
// in show(), when entry._enFallback && document.documentElement.dir === 'rtl':
popup.classList.add('is-en-fallback');       // css: .whats-new-popup.is-en-fallback { direction: ltr; text-align: start; }
```

### IN-01: `.is-en-fallback` LTR override does not cover rail items or jump-strip links

**File:** `assets/help.css:378-393`, `assets/help.js:224-264` (buildRailItem), `assets/help.js:471-481` (jumpNav)
**Issue:** The RTLâ†’LTR override applies only to `.card-body` and `.card-head h2`. When a section falls back to EN under an RTL page (whole-locale-load failure), its EN title also appears in the rail, the rail sub-list, and the mobile jump strip â€” all still `direction: rtl`. Titles are short so scrambling is minimal (trailing punctuation/quotes can flip edges, e.g. the troubleshooting title's quotes).
**Fix:** Either extend the marker to rail/jump nodes (`item.classList.add('is-en-fallback')` in `buildRailItem` using `_fallbackIds`, plus a matching css rule), or accept as a known cosmetic limit of the fallback path and note it beside the BLOCKER-2 comment.

### IN-02: Language switch while the What's-New popup is open yields a mixed-locale popup

**File:** `assets/whats-new.js:142-188`
**Issue:** `sub`, `seeAll`, and `close` carry `data-i18n`, so the global `applyTranslations()` pass on `app:language` re-translates them in place â€” but the interpolated `h2` title and the highlight `<li>`s (resolved once at `show()`) keep the old locale. A user who switches language with the popup open sees half-translated chrome around old-locale highlights. Edge case: the popup fires once per version and the switch window is seconds.
**Fix:** Either drop the `data-i18n` stamps (freeze the popup in its mount locale, consistent within itself) or subscribe to `app:language` and re-render title + highlights from `entries()` while mounted.

### IN-03: `help.chrome.reportHint` flattening dropped the "Report a problem" emphasis

**File:** `help.html:103`, `assets/i18n-en.js` (`help.chrome.reportHint`)
**Issue:** The static hint previously wrapped the feature name in `<strong>Report a problem</strong>`. Wiring it through `data-i18n` (textContent-only by design â€” correct for the XSS boundary) required flattening to plain text. DE/CS compensate by quoting the label ('Problem melden' / 'NahlĂˇsit problĂ©me'); EN and HE carry no marker at all, so the actionable feature name no longer stands out in the contact band.
