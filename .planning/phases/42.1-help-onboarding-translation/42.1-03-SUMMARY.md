# Phase 42.1 Plan 03: Hebrew Help Corpus Summary

Authored `assets/help-content-he.js` — the Hebrew mirror of `help-content-en.js` (~2,600 words across 12 sections), structurally byte-identical to EN with only titles and body prose translated into the locked Hebrew house register; passes the `he` half of the Phase 42.1 integrity gate 10/10.

## What was built

A single new file, `assets/help-content-he.js`, that registers `window.HELP_CONTENT_HE` = an ordered 12-section array. It is a strict structural clone of `help-content-en.js`:

- Every section `id`, `group`, `featured`; every topic `id`, `priority`, `covers[]`; every body node `type` and glyph `name`; and the exact order of sections/topics/body nodes are preserved byte-identical to EN.
- All 65 unique `{ui:...}` tokens are copied verbatim (never translated) and resolve to non-empty strings in `window.I18N.he`.
- Only section `title`, topic `title`, and body `text` / `steps[]` items are translated into Hebrew.
- `window.HELP_DEEPLINKS` is NOT redefined (it lives only in the EN file, per the loader contract).

## Verification

`node tests/42_1-help-integrity.test.js` — all 10 `[he]` assertions PASS:
- structure parity with EN (section ids/order, group, featured, topic ids/priority/covers)
- exactly one featured section; both groups present
- every `{ui:}` token resolves against `I18N.he`
- `{ui:}` token set byte-identical to EN per topic
- no forbidden clinical terms (מטופל/טיפול)
- no emoji; no control/bidi-control chars (only `\n` allowed)

Acceptance criteria confirmed independently:
- vm-load succeeds; `HELP_CONTENT_HE` is a 12-element array.
- Isolated load (he file only, no EN): `HELP_DEEPLINKS` undefined, `HELP_CONTENT_EN` undefined, `HELP_CONTENT_HE` defined — the locale file registers only its own global.
- Section-id order matches EN exactly.

Full suite: `node tests/run-all.js` → 157 passed / 7 failed, **identical with and without this file** (verified by temporarily removing it). This file introduces zero new failures. The 7 failing files are all pre-existing RED gates from other unlanded 42.1 plans (de/cs help siblings, precache, i18n-parity, tour i18n-parity, changelog-locale, whats-new-gating). The `de`/`cs` failures inside `42_1-help-integrity.test.js` are the expected RED guards for their not-yet-authored siblings — the plan explicitly notes "de/cs may still be RED until their plans land."
