## Typography

The cohesion problem #5 is "several heading weights/sizes." Today the modal mixes **four** heading treatments: title `1.75rem/700`, section-heading `1.4rem/700`, Import-only override `1.125rem/700`, and the `<details>` summary reusing `1.4rem`. This contract collapses them to **exactly 3 type roles, 2 weights (400 body / 700 emphasis)** ÔÇö one section-heading size for ALL sections (Export, Import, Test-password, Contents, Reminders summary), and one modal title.

| Role | Size | Weight | Line Height | Phase 27 Usage |
|------|------|--------|-------------|----------------|
| Body | 16px (1rem) | 400 | 1.5 | Section helper lines ("Save a copy of all your dataÔÇŽ"); Contents-item labels render at Label weight, see below |
| Helper | 14.4px (0.9rem) | 400 | 1.5 | `.helper-text` (existing app.css:1158) ÔÇö section descriptions; keep existing 0.9rem so it matches every other helper in the app |
| Heading | 18px (1.125rem) | 700 | 1.2 | **ONE** size for ALL modal section headings ÔÇö Export / Import / Test-password / Contents / "How reminders work" summary. (Down from the mixed 1.4rem/1.125rem; up the Import one to match. Single size = single rhythm.) |
| Display | 28px (1.75rem) | 700 | 1.2 | Modal title "Backup & Restore" only (unchanged from Phase 25 ÔÇö gives the consolidated surface page-replacement gravitas). |

**Contents-list item labels:** 14.4px (0.9rem) **700** ÔÇö they read as a checklist, not body copy. (Matches the existing `.backup-contents-item { font-weight: 700 }`.) This is the only place 700 weight appears at body size; it is the "Label" intent.

**Removed from the modal type system:**
- `.backup-modal-section--import .backup-modal-section-heading { font-size: 1.125rem }` (the Import-only override) ÔÇö superseded by the single 1.125rem heading for all sections. The "Export is more prominent than Import" hierarchy (Phase 25 D-07) is now carried by **placement + the primary-vs-ghost button**, not by a smaller Import heading.
- The `1.4rem` value on `.backup-modal-section-heading` drops to `1.125rem`.

**Why 18px for headings (not the old 1.4rem):** Four section headings at 1.4rem inside a 560px modal competed with the 1.75rem title for attention and made the modal read as "many big things." 1.125rem/700 is clearly a heading, clearly subordinate to the title, and gives a calm ladder: 28 title Ôćĺ 18 section heading Ôćĺ 16/14.4 body. Heading-to-body contrast (18 vs 16) stays Ôëą the 1.125├Ś ratio that reads as hierarchy without shouting.

**Hebrew/RTL:** Rubik covers Hebrew; all sizes/weights dir-agnostic; `html[dir=rtl]` (app.css) flips layout automatically. Per 25-UI-SPEC D-27 convention, Hebrew strings use noun/infinitive forms.

## Decided Forks (load-bearing ÔÇö autonomous decision, planner-overridable)

> ROADMAP told me to ask the user only for truly load-bearing forks. AskUserQuestion is unavailable inside subagents, so both forks are decided here with rationale. Each is a reversible CSS-only choice; the planner may override either in a single plan task without re-running discuss-phase.

### Fork 1 ÔÇö Section rhythm Ôćĺ **PLAIN sections + dividers** (drop the tinted cards)

**Decision:** Remove both tinted `--color-surface-alt` cards (`.backup-modal-section--contents` and `.backup-test-password-card`). Every section becomes a plain block on the modal surface, separated by the existing thin `--color-border-soft` divider (`.backup-modal-section + .backup-modal-section`, app.css:3719) that ALREADY runs between sections.

**Rationale:**
- The modal's own CSS already ships the plain-section + hairline-divider cadence (`.backup-modal-section + .backup-modal-section { border-block-start: 1px solid --color-border-soft }`). The tinted cards were the round-by-round accretion fighting that cadence. Removing them makes the modal internally consistent with itself.
- It matches the rest of Sessions Garden, which uses plain sections + dividers, not nested tinted panels.
- It is the quietest fix for "too many colours/formats" ÔÇö two surface colours collapse to one.
- 2025 UX guidance: for information-dense surfaces, prefer subtle dividers over nested coloured containers to reduce visual noise (sources below).

**Why not "cards everywhere":** wrapping all 5 sections in tinted cards adds visual weight to a modal that is already tall (560px wide, scrolls on overflow) and doubles down on the device Ben flagged as noisy.
**Why not "cards for read-only only":** it keeps two surface colours alive and still requires the reader to learn a card-means-X convention ÔÇö more rules, not fewer.

### Fork 2 ÔÇö Danger tone Ôćĺ **AMBER inline** (demote the red Import band)

**Decision:** The Import "Replaces all current data" warning moves from the RED danger band to the shared AMBER caution band (`--color-warning-bg` / `--color-warning-text` / 4px `--color-warning-text` border-inline-start). Red is reserved exclusively for the destructive confirm dialog at the moment the user clicks Import.

**Rationale:**
- ROADMAP item 4 verbatim: "demote the full red Import banner to a calm-but-clear warning consistent with the other inline messages." The other inline message in the modal (test-password error) is already amber ÔÇö amber IS the consistent tone.
- It satisfies ROADMAP item 3 ("ONE severity palette") in one move: Import warning + test-password error become the same amber component.
- Severity stays honest: the warning still has a coloured band + warning glyph + bold-ish weight, so destructiveness is clearly signalled; the *actual* irreversible confirmation still gets full red at the decision point.

**Why not "neutral hint":** a plain helper line under-signals that Import wipes data ÔÇö too quiet for a destructive path.
**Why not "keep red, unify others":** keeps the loudest element loud; contradicts ROADMAP item 4's explicit "demote."
