  <action>
Read every JS, CSS, and HTML file in the project. Analyze and document findings in a structured report covering THREE sections:

**Section 1: Security Audit (per D-01, D-02, D-06)**
Scan every JS file for:
- `eval()`, `Function()`, `setTimeout/setInterval` with string args
- `innerHTML` assignments with user-supplied or unsanitized data (note: innerHTML with static HTML templates is acceptable -- only flag cases where dynamic user input is injected without sanitization)
- Any `fetch()`, `XMLHttpRequest`, `navigator.sendBeacon`, or WebSocket calls to external URLs (the app must make ZERO external network calls per architecture constraint; Lemon Squeezy validation in license.js is expected but should be documented)
- Hardcoded secrets, API keys, tokens in source (check all JS files and HTML)
- `localStorage`/`sessionStorage` storing sensitive data without encryption
- Missing Content-Security-Policy meta tags in HTML files
- `postMessage` without origin validation
- File/blob URL handling in backup.js (ZIP processing security)

Severity levels:
- CRITICAL: Exploitable security vulnerability (blocks launch per D-06)
- HIGH: Security risk that should be fixed before launch
- MEDIUM: Security improvement, not exploitable
- LOW: Best practice suggestion

**Section 2: Dead Code and Unused Assets (per D-03)**
Identify:
- JS functions defined but never called (grep for `function functionName` and then check if `functionName` appears elsewhere)
- CSS selectors in app.css/tokens.css/landing.css that match no HTML element or JS-generated class
- HTML files or sections that are unreachable from any navigation path
- i18n keys defined in i18n-en.js but never referenced in any JS or HTML file (search for the key string in all files)
- Unused CSS custom properties defined in tokens.css but never referenced in any CSS file
- JS files loaded via script tags but whose exports/functions are unused on that page

For each item, provide: file path, line number, the dead code snippet, and confidence level (CERTAIN / LIKELY / POSSIBLE).

**Section 3: Architecture and Code Consistency (per D-02)**
Analyze:
- Duplicate logic across per-page JS files (e.g., similar DOM manipulation patterns in add-client.js vs add-session.js that could be shared)
- Inconsistent patterns: do some files use one approach while others use a different approach for the same task? (e.g., event binding, DOM queries, error handling, i18n key access)
- Module structure: is there a clear separation of concerns or is business logic mixed with DOM manipulation?
- CSS organization: are all visual values using tokens or are there hardcoded colors/sizes that bypass the token system?
- db.js migration chain: verify all migrations (v0 through latest) can execute sequentially without error
- Service worker (sw.js) cache list: does it include ALL files the app needs for offline use? List any missing files.

Format each finding as:
```
### [SEVERITY] Finding title
**File:** path/to/file.js:lineNumber
**Description:** What was found
**Impact:** Why it matters
**Recommendation:** What to do (informational per D-04, D-07)
```

Write the complete report to `.planning/phases/15-architecture-and-ui-audit/15-01-REPORT-security-code-architecture.md` with a summary table at the top counting findings by severity.
