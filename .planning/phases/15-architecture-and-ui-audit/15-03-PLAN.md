  <action>
Read ALL i18n files and ALL JS/HTML files. Produce a structured report with FOUR sections:

**Section 1: i18n Key Coverage Matrix**

Extract EVERY key from i18n-en.js (the reference language). For each key:
- Check if it exists in i18n-he.js, i18n-de.js, i18n-cs.js
- Check if it exists in i18n-disclaimer.js (if applicable)
- Mark as: PRESENT in all / MISSING from [list languages]

Also do the REVERSE check:
- Are there keys in HE/DE/CS that do NOT exist in EN? (orphaned translations)

Present as a table:
```
| Key | EN | HE | DE | CS | Used In |
|-----|----|----|----|----|---------|
```

Only list keys that have issues (missing from at least one language, or never used). Do not list keys that are complete and used.

**Section 2: Unused i18n Keys**

For every key defined in any i18n file, search ALL JS and HTML files for usage. A key is "used" if its string appears as:
- An argument to a translation function (e.g., `t('keyName')`, `i18n.keyName`, `translations.keyName`)
- A data-i18n attribute value in HTML
- Referenced dynamically via string concatenation or template literals (note these as POSSIBLY used)

List all keys that appear in NO JS or HTML file as UNUSED. For each: key name, which language files contain it, confidence (CERTAIN if literal match not found anywhere / POSSIBLE if dynamic access might reach it).

**Section 3: Wrong-Language and Placeholder Bugs (per D-18)**

Scan for:
- Hardcoded English (or any single-language) strings in JS files that should be translated but are not going through i18n (e.g., alert messages, error text, placeholder attributes set in JS)
- HTML elements with hardcoded text that should use data-i18n attributes
- Known bug patterns from Phase 14: German text appearing in Hebrew mode, untranslated trust badges, mixed-language legal page headers
- Strings in i18n files that are obviously in the WRONG language (e.g., an English string in i18n-he.js, a German string in i18n-cs.js)
- Placeholder text like "TODO", "PLACEHOLDER", "[YOUR_", "Lorem ipsum" in any language file
- Empty string values "" in any language file (key exists but translation is missing)

For each finding provide: file, line number, the problematic string, which language it should be in, severity (HIGH if visible to user, MEDIUM if edge case, LOW if internal).

**Section 4: RTL Layout Issues**

Check all CSS files for:
- `left`/`right` properties that should be `inset-inline-start`/`inset-inline-end` (per Phase 2 decision DSGN-03)
- `margin-left`/`margin-right` that should be `margin-inline-start`/`margin-inline-end`
- `padding-left`/`padding-right` that should be `padding-inline-start`/`padding-inline-end`
- `text-align: left` that should be `text-align: start`
- `float: left/right` usage
- `direction` or `dir` attribute handling — are all pages setting dir="rtl" for Hebrew?

Check all HTML files:
- Do they have the `dir` attribute or mechanism to set it based on language?
- Are there inline styles with directional properties?

For each finding: file, line number, the property, what it should be, severity.

Write the complete report to `.planning/phases/15-architecture-and-ui-audit/15-03-REPORT-i18n-translations.md` with a summary at the top: total keys checked, keys with coverage gaps, unused keys, wrong-language bugs found, RTL issues found.
