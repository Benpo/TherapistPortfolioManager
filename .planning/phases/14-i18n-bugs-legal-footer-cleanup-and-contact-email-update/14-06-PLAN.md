<task type="auto">
  <name>Task 1: Fix demo horizontal resize handles</name>
  <files>landing.html, assets/landing.css, assets/landing.js</files>
  <action>
    The problem: resize handles are 8px wide, positioned inside the demo-window-body overlapping the iframe edge. The iframe intercepts the initial pointerdown before pointer-events:none is applied. Users see browser scroll instead of resize.

    Fix approach — widen handles, move outside iframe, add permanent overlay on hover:

    1. In assets/landing.css, update `.demo-resize-handle`:
       - Change `width: 8px` to `width: 24px` — wider grab zone
       - Change `opacity: 0` to `opacity: 0` (keep invisible by default)
       - Add `touch-action: none` to prevent browser scroll during drag
       - For `.demo-resize-handle--start`: change `inset-inline-start: 0` to `inset-inline-start: -12px` — extend 12px outside the window body
       - For `.demo-resize-handle--end`: change `inset-inline-end: 0` to `inset-inline-end: -12px` — extend 12px outside
       - Update hover/dragging state: change opacity to 0.85 (not just 1) for softer appearance
       - Update `::after` grip dots: change `width: 3px` to `width: 4px`, change `height: 32px` to `height: 40px` for better visibility

    2. In assets/landing.css, add a transparent overlay on demo-window-body that prevents iframe from stealing events near edges:
       ```css
       .demo-window-body::before,
       .demo-window-body::after {
         content: '';
         position: absolute;
         top: 0;
         width: 24px;
         height: 100%;
         z-index: 5;
       }
       .demo-window-body::before {
         inset-inline-start: 0;
       }
       .demo-window-body::after {
         inset-inline-end: 0;
       }
       ```
       These invisible strips sit on top of the iframe edges, ensuring pointer events reach the resize handles.

    3. In assets/landing.js, update the resize IIFE:
       - Add `e.currentTarget.setPointerCapture(e.pointerId)` in onPointerDown — this ensures all subsequent pointermove events go to the handle element even if pointer moves over the iframe.
       - In onPointerUp, add `if (activeHandle) activeHandle.releasePointerCapture(e.pointerId)` before the cleanup.
       - This is the proper way to handle drag operations over iframes — pointer capture guarantees events route to the capturing element.

    4. Verify `.demo-window-body` has `position: relative` (it does, line 1094) so the absolutely-positioned pseudo-elements and handles work correctly.
  </action>
  <verify>
    <automated>grep -c "width: 24px" assets/landing.css && grep -c "setPointerCapture" assets/landing.js && grep -c "touch-action" assets/landing.css</automated>
  </verify>
  <done>Demo resize handles are 24px wide with pointer capture — horizontal dragging works reliably even near iframe edges.</done>
