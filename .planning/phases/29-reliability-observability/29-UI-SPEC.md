# Phase 29 — UI Design Contract

> Visual and interaction contract for the Phase 29 UI surfaces: OBS-02 "Report a problem" screen, OBS-03 "Reset & recover" escape hatch, and the OBS-01 surfacing (Settings entry point + optional crash-log clear). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**This is a mature ~28-phase zero-build PWA. This contract REUSES the established design system — it does NOT invent a new one.** Tokens, type scale, spacing, color and component patterns are all sourced from the existing codebase (`assets/tokens.css`, `assets/app.css`, `assets/db.js`). Where this phase's two surfaces have different rendering constraints, that split is called out explicitly below.

---

## Two Rendering Contexts (read this first)

This phase has **two visually different surfaces with different constraints**. Do not apply one's rules to the other.

| Surface | When it renders | Styling source | Token availability |
|---------|-----------------|----------------|--------------------|
| **A. Report-a-problem screen (OBS-02)** + Settings entry point (OBS-01 surfacing) | Normal app runtime — `app.css` + `tokens.css` + `i18n.js` all loaded | Full app stylesheet, full `--color-*` semantic tokens, `data-i18n` | All semantic tokens defined |
| **B. Reset & recover escape hatch (OBS-03)** + migration/integrity banner | The **failure surface itself** — must render when IndexedDB is broken and possibly **before `i18n.js` loads** | The early-loading `db.js` pattern: inline `DB_STRINGS`-style strings + the `.db-error-banner` / `.integrity-nudge` CSS family in `app.css` | `--space-*` and `--text-*` scale tokens are **NOT defined in `:root`** — the **fallback literal renders** (see Spacing FLAG below) |

Surface B inherits the exact pattern Phase 28 established for `.integrity-nudge` (`app.css:2180–2251`) and the existing `.db-error-banner` family (`app.css:2132–2178`). The escape-hatch is an extension of `showDBMigrationError()` (`db.js:429–450`), so it lives in the same early, no-i18n, 4-language, RTL-safe regime.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (zero-build, zero-npm, IIFE-global modules served verbatim; no shadcn / React / Tailwind / component lib) |
| Preset | not applicable |
| Component library | none — hand-rolled DOM + `app.css` classes |
| Icon library | none — inline SVG / Unicode glyphs (matches existing `.button-icon`, `\2713` checkmark pattern) |
| Font | Rubik (400 / 600 / 700), `--font-family` with `system-ui, sans-serif` fallback (`tokens.css:1–22`) |

**Constraints that override all visual freedom (from CONTEXT):**
- **Zero network calls.** No `fetch`, no analytics, no remote fonts/CDN in any new surface. The report flow assembles + previews + copies + hands off via `mailto:`; transmission is always the user's manual action.
- **4 languages EN / HE / DE / CS, RTL-safe.** All new strings cover all four. Use logical CSS properties only (`inset-inline-*`, `margin-block`, `text-align: start`) — never physical `left/right`. (Established RTL rule: `html[dir=rtl]`, Phase 18.)
- **Surface B must render before `i18n.js`** — use the inline `DB_STRINGS`/`FOOTER_STRINGS` early-string pattern (`db.js:152–183`), not `data-i18n`.
- **The escape hatch must work when IndexedDB is broken** — no dependency on the DB being openable; backup-export path opens the DB read-only at its existing un-upgraded version (the D-09 wrinkle, planner to resolve).

## Spacing Scale

The project does **NOT** define a numeric `--space-*` scale in `:root`. App CSS uses two coexisting idioms:

1. **rem-based literals** in full-app components (e.g. `.button` padding `0.6rem 1.3rem`, `.modal-card` padding `1.8rem`, `.confirm-card` padding `2rem`).
2. **`var(--space-*, <fallback>)` with the FALLBACK as the live value** in the banner/error family — because the tokens are undefined, the literal fallback is what renders.

| Token (as written) | Rendered value | Usage |
|---------------------|----------------|-------|
| `var(--space-xs, 2px)` | 2px | banner button vertical padding |
| `var(--space-sm, 4px)` / `var(--space-sm, 8px)` | 4px or 8px (context-dependent literal) | banner button margin-top (4px); inline callout padding (8px) |
| `var(--space-md, 12px)` / `var(--space-md, 16px)` | 12px or 16px | banner padding (12px); callout horizontal padding (16px) |
| `var(--space-lg, 16px)` | 16px | banner horizontal padding |
| rem literals: `0.5rem`, `0.75rem`, `1.3rem`, `1.6rem`, `1.8rem`, `2rem` | as written | full-app modal/button/card spacing |

> ⚠️ **Spacing FLAG (carried over verbatim from Phase 28 `app.css:2182–2184`):** none of the `--space-*` scale tokens are defined in `:root`, so the fallback literal is what renders. For **Surface B** (escape-hatch / banner), the executor MUST copy the existing `.db-error-banner` / `.integrity-nudge` fallback literals **verbatim** (12px / 16px / 4px / 2px). **Do NOT "normalize" these up to an 8-point scale** — doing so changes the rendered pixels and breaks visual consistency with the shipped Phase 16/28 banners. The 8-point-multiple expectation in the generic checker is explicitly waived for this banner family; the contract is "match the existing banner siblings," not "snap to 4px grid."

For **Surface A** (full-app report screen + Settings row), reuse the existing app idioms: `.modal-card` / `.confirm-card` padding, `.settings-row` layout, `.button` padding. Do not introduce new spacing values.

Exceptions: the entire Surface-B banner family is the documented exception above.

## Typography

Sizes are rem-based in the existing system. New surfaces MUST reuse existing roles — do not introduce new sizes.

| Role | Size | Weight | Line Height | Source / where used |
|------|------|--------|-------------|---------------------|
| Banner / body (Surface B) | `var(--text-base, 14px)` → renders **14px** | 400 (Rubik Regular) | 1.5 | `.db-error-banner` / `.integrity-nudge` (`app.css:2141, 2193`) |
| Banner button / small (Surface B) | `var(--text-sm, 13px)` → renders **13px** | 600 (Rubik SemiBold) | 1 | `.db-error-btn` / `.integrity-nudge-btn` (`app.css:2170, 2224`) |
| Body (Surface A) | 0.9rem–1rem (≈14–16px) | 400 / 600 | 1.5 | app body, `.confirm-body` line-height 1.5 |
| Dialog title (Surface A) | 1.35rem (≈22px) | 800 | 1.2 | `.confirm-title` (`app.css:1560–1564`) |

Font weights in play: **400 (Regular)**, **600 (SemiBold)**, **700 (Bold)**, **800** (confirm titles only). This is the established project set — the escape-hatch UI uses 400/600 (the banner family); the destructive confirm dialog uses the existing `.confirm-title` 800 / `.confirm-body` 600. Do not add new weights.

Body line height **1.5** (matches `.confirm-body`, `.error-text`); heading/title line height **1.2** (matches `.confirm-title`); single-line button glyphs `line-height: 1`.

## Color

The 60/30/10 split maps onto the existing semantic tokens (`tokens.css`). Both light and dark mode are already defined for every token used here — the new surfaces inherit both automatically.

| Role | Token | Light value | Usage |
|------|-------|-------------|-------|
| Dominant (60%) | `--color-background` / `--color-surface` | `#fdf8f0` / `#ffffff` | page + report-screen background, modal card surface |
| Secondary (30%) | `--color-surface-alt` / `--color-surface-secondary-btn` | `#fdf7ed` / `#e8f5ee` | preview textarea surface, secondary buttons, settings rows |
| Accent (10%) | `--color-primary` | `#2d6a4f` (green-600) | see reserved-for list below |
| Destructive | `--color-danger` (button) + `--color-danger-bg`/`--color-danger-text` (band) | `#ea4b4b` / `#f8d7da` / `#721c24` | destructive confirm only — see red-discipline below |

**Accent (`--color-primary`) reserved for — explicit list:**
- The primary action button on the Report screen: **"Copy report"** (`.button` default green).
- The **"Open email to support"** primary/secondary button on the Report screen.
- The **"Export backup now"** button in the escape hatch (routes to the existing safe export; it is the recommended/safe path, so it gets the primary accent — NOT red).
- The Settings **"Report a problem"** entry-point link/row affordance.
- The migration banner's safe action button (`.db-error-btn` already uses `--color-primary`).

Accent is **NOT** used for: body text, the preview textarea border in its resting state, the affirmation checkbox, or decorative elements.

**Red-discipline (Phase 27 / Phase 28 carry-over — MUST hold):** `--color-danger-*` is reserved for **the moment-of-action destructive confirm dialog ONLY** — i.e. the OBS-03 double-confirm before `indexedDB.deleteDatabase`. The migration-error band itself may use the existing `.db-error-banner--migration` danger band (it already does, `app.css:2158–2161`) because it is the failure surface. Severity ladder for any inline bands follows the established `info` → `warning` → `danger` family (`--color-info-*` / `--color-warning-*` / `--color-danger-*`). The **"Export backup now"** button is explicitly **green/primary, not red** — it is the safe, encouraged action; red is spent only on the irreversible wipe confirm.

## Interaction Contract (per surface)

### OBS-02 — Report a problem screen (Surface A)
1. **Redacted preview** of the assembled report (crash log + diagnostic context: app version, language, userAgent, DB version, store counts, storage usage — builder's discretion on exact fields, all behind the D-04 redaction). Preview is **read AND editable** (the user's own eyes are the final privacy gate before they paste into email).
2. **"Copy report"** (primary, `--color-primary`) — full multi-line log → clipboard. Show a transient success affirmation reusing the existing toast/pill idiom.
3. **"Open email to support"** — launches a prefilled `mailto:` to `contact@sessionsgarden.app` (the Impressum support address) with a preset subject and a short "paste your report below this line" body. Body does NOT carry the full log (URL-length limits — that's what Copy is for). If `mailto:` proves unreliable in the installed PWA, **degrade to copy-only + the displayed support address** (builder's call, D-06).
4. **Zero network** — nothing transmits from the app. Ever.

### OBS-03 — Reset & recover escape hatch (Surface B)
Replaces the dead-end "Refresh page → `location.reload()`" of `showDBMigrationError()` (`db.js:429–450`) which re-runs the same failing migration forever. Also the landing spot for Phase 28's "Couldn't finish automatically" integrity escalation (28-CONTEXT D-12).

1. Renders in the **danger band** (`.db-error-banner--migration`) with the existing migration message — reassuring copy ("Your data is safe").
2. **"Export backup now"** (primary green, NOT red) — routes through the **existing passphrase export flow** (`backup.js`), opened **read-only at the existing un-upgraded DB version** to export around the failure (D-09 wrinkle — planner resolves the mechanics).
3. **Hard affirmation gate before the wipe:**
   - A **checkbox**: "I have saved a backup I can restore from" — the destructive button stays **disabled** until checked.
   - A **double-confirm dialog** (`App.confirmDialog tone: 'danger'`) on the destructive action.
   - The confirmation is **extra-emphatic when no export happened in this session** (catch the user who clicked straight to reset) — stronger wording / second affirmation.
4. Only then does `indexedDB.deleteDatabase` + reload-into-fresh-DB run.
5. Must function with IndexedDB broken and (if it renders pre-i18n) use inline 4-language strings, RTL-safe.

### OBS-01 surfacing (Surface A, lighter)
- A **"Report a problem"** entry-point row/link in Settings (`.settings-row` idiom) routing to the dedicated report screen.
- Optional **crash-log "clear"** affordance (empty the log) — builder's discretion; if built, reuse `.button.ghost` + a `confirmDialog` neutral-tone confirm.

## Copywriting Contract

All copy is authored in **all 4 languages (EN / HE / DE / CS)**. Surface A uses `data-i18n` keys; Surface B uses inline `DB_STRINGS`-style objects. EN shown here as the contract reference; translations are the executor's deliverable, matching the natural-translation quality bar set in prior phases (no machine-literal strings).

| Element | Copy (EN reference) |
|---------|---------------------|
| Primary CTA — Report screen | **Copy report** |
| Secondary CTA — Report screen | **Open email to support** |
| Settings entry point | **Report a problem** |
| Escape-hatch safe action | **Export backup now** |
| Escape-hatch destructive action | **Reset & recover** (or "Reset the app") |
| Report-screen intro / privacy reassurance | "Nothing is sent automatically. Review the report below, copy it, then paste it into an email to support. Your data stays on this device." |
| Empty state (no crash log) | Heading: "No problems logged" — Body: "Nothing has gone wrong on this device recently. If you're seeing an issue anyway, you can still describe it in an email to support." |
| Error state — migration failure band | Reuse existing `migrationFailed`: "Database update failed. Your data is safe." + next-step actions ("Export backup now" / "Reset & recover") instead of the old dead-end "Refresh page". |
| Affirmation checkbox | "I have saved a backup I can restore from" |

Confirm: "Reset and delete everything" / Cancel: "Go back and export first" |

## Open Questions

These were not fully resolved by CONTEXT; best-judgment defaults are recorded above, but the orchestrator should surface them to the user before/at planning:

1. **Report screen: dedicated page vs. modal.** CONTEXT D-05 says "dedicated screen, not just a Settings button." Default assumed here: a dedicated route/screen (new lightweight HTML page or a full-viewport view) reachable from the Settings "Report a problem" row, reusing app chrome. If a `.modal-card` overlay better fits the multi-page architecture, that is an acceptable substitution — confirm the preference.
2. **Does the OBS-03 escape hatch ever render in normal app chrome, or only as the pre-i18n failure banner?** Default assumed: it appears as the failure-surface banner/panel (Surface B, pre-i18n inline strings). If Ben also wants the reset reachable from a normal-runtime Settings location, that second entry point would be Surface A (full i18n) — but CONTEXT explicitly scopes OBS-03 to the failure path and defers a standing "nuclear reset" button, so the default is failure-path-only.
3. **`mailto:` reliability in the installed PWA.** D-06 leaves the copy-only degradation to implementation. Recommend the executor field-verifies `mailto:` on a real installed PWA (per the Phase 28 field-verification lesson) and, if it fails, ships copy-only + visible support address rather than a broken email button.
