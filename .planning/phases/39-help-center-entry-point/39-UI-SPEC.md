# Phase 39 — UI Design Contract

> Visual and interaction contract for the Help Center & "?" Entry Point.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** This phase ships **production code** (the "?" header popover on every app
> page, a standalone offline `help.html`, `assets/help-content-en.js`, empty-state
> coaching deep-links on the first-run trio, and a static integrity test). Every value
> below is *inherited from the existing garden design system* (`assets/tokens.css`) — this
> contract does not introduce a new palette, scale, or typography. It reuses the approved
> Phase 26 contract (`.planning/milestones/v1.1-phases/26-in-app-onboarding-overview-help-system/26-UI-SPEC.md`)
> as its seed and constrains how the new surfaces compose existing tokens.
>
> **Two owner-approved amendments to the Phase 26 seed apply here:**
> - **D-05 — Soft type on help surfaces only:** Rubik **400** headings (deep garden green
>   `--color-primary-deeper`), muted body ink, roomier cards, botanical watering-can accent
>   in the page header. This deliberately softens the Phase 26 bold-heading rule **for help
>   surfaces only**. Zero new font files (reuses existing Rubik faces already in the repo).
> - **D-15 — Computer-only install:** install instructions target **computers only**
>   (Chrome/Edge address-bar icon + macOS Safari "Add to Dock"). The Phase 26 / HELP-06
>   iOS-Safari / Android wording is superseded.
>
> **Start implementation from the approved mockup**
> `.planning/sketches/002-help-page-ia/hybrid.html` (behavior + concrete values), NOT the
> A/B/C variants. The mockup exposed that a sketch theme can silently fall back to system
> fonts — the build MUST verify type against the real `assets/fonts/Rubik-*.woff2` files.

## Spacing Scale

Inherited 4px-base-grid scale (authored in `rem`, 1rem = 16px). The new help / "?"-popover /
empty-state surfaces MUST use these steps. Matches the Phase 26 contract and the approved
mockup's `--space-*` tokens exactly.

| Token | rem | px | Usage in this phase |
|-------|-----|-----|---------------------|
| xs | 0.25rem | 4px | Icon-to-label gap, rail sub-topic padding, in-card body top padding |
| sm | 0.5rem | 8px | Compact stacking (rail item gaps, search margin, card-head row gaps, popover item padding) |
| md | 1rem | 16px | Default element spacing; card body padding; content-column gap; sticky-rail top offset |
| lg | 1.5rem | 24px | Card head padding-inline; page-head inline padding; contact-band padding; content grid outer padding |
| xl | 2rem | 32px | Page-head vertical rhythm; content grid column gap; section breaks; search-empty padding |
| 2xl | 3rem | 48px | Major help-page section separation |
| 3xl | 4rem | 64px | Page-level bottom safe padding on the content grid |

Exceptions: none as a *rhythm* token. The **44×44px minimum interactive size** (`--tap-target-min`)
is an accessibility constraint (Apple-HIG), not a spacing step — it lives in the Interaction
Contract. Search input vertical padding (`10px`) and the tiny rail eyebrow/letter-spacing values
(`.72rem`, `2px`) are component-intrinsic, not layout rhythm.

## Typography

Constrained 4-role scale drawn from the most-used existing sizes (matches the mockup's
`--text-*` tokens). Legible to non-technical therapists; survives RTL/translation length
variance. **Help body content is EN-only** in `assets/help-content-en.js` (D-18); UI-chrome
strings (menu labels, search placeholder, empty-state coaching) keep 4-locale parity.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Body | 1rem (16px) | 400 | 1.6 | Help page body copy, popover item labels, empty-state coaching sentences, contact-band body |
| Label / UI | 0.9rem (≈14px) | 700 | 1.4 | Buttons ("Show me how", "Write to us"), rail active/featured items, search input text, technical-track sub-labels, rail eyebrow labels |
| Heading | 1.25rem (20px) | **400** (soft type — D-05) | 1.3 | Help page card/section headings (workflow-spine step titles), contact-band heading. Rendered in `--color-primary-deeper` deep garden green with slight letter-spacing. Card sub-topic `h3` uses **600** (SemiBold, gentle in-card emphasis). |
| Display | 1.75rem (28px) | **400** (soft type — D-05) | 1.2 | Help page header title only ("Help center") — soft, calm, letter-spaced. |

**Weight set (soft-type help surfaces — D-05):** **three weights — 400, 600, 700 —** all
from Rubik faces already in the repo (zero new font files):
- **400** — body copy, page title (Display), card/section headings (Heading).
- **600** — card sub-topic `h3` only (gentle in-card emphasis; SemiBold face).
- **700** — interactive labels/buttons, active/featured rail items, "Write to us" / "Show me how" buttons, focus states.

> **Accepted exception for the checker (do NOT spin the 2-weight-max loop):** the Phase 26
> seed locked "exactly two weights (400/700)". D-05 (owner-approved this session) deliberately
> introduces the soft-type hierarchy above, which uses **three** weights (400/600/700) on help
> surfaces. This is intentional, present in the Ben-approved mockup (`hybrid.html`), and reuses
> only existing font files. The greenfield 2-weight rule is a known false-positive source on
> reuse/amendment phases (memory: `feedback-ui-checker-greenfield-false-positives`) — verify
> against the real mockup + `assets/fonts/`, then PASS with this accepted-exception note. The
> rest of the app (outside help surfaces) is unchanged.

**Rules:**
- Exactly these 4 roles in new surfaces. No new font sizes beyond this set (the mockup's `1.3rem`
  soft-heading variant resolves to the Heading role's rendered size band).
- Body line-height **1.6** (not 1.5) deliberately — help is long-form reading for a non-technical
  audience; matches the existing long-copy precedent.
- All headings authored as **nouns / gerunds, not imperatives** ("Adding a client", "Making
  Sessions Garden yours" — never "Add a client!"), so the later HE translation phase produces
  natural noun/infinitive Hebrew (Phase 24 D-05).
- Help copy quotes live UI labels via `{ui:key}` interpolation (D-23) — quoted labels render at
  the surrounding role's size/weight; they must resolve to a real i18n key (guarded by the D-25 test).

## Color

Inherited from `assets/tokens.css` semantic tokens. The 60/30/10 discipline mapped onto the
existing garden palette — the implementation MUST use the listed token vars (which auto-resolve
in `[data-theme="dark"]`), never the literal hex.

| Role | Token (use this var) | Light value | Usage in this phase |
|------|----------------------|-------------|---------------------|
| Dominant (60%) | `--color-background` (+ body gradient `--color-background-gradient-start/-end`) | `#fdf8f0` warm cream | Help page background, popover backdrop scrim (if any) |
| Secondary (30%) | `--color-surface` / `--color-surface-alt` | `#ffffff` / warm cream | Help cards, "?" popover panel surface, contact-band surface, search input field |
| Accent (10%) | `--color-primary` (`--color-primary-soft` / `--color-primary-dark` / `--color-primary-deeper`) | `#2d6a4f` garden green | **Reserved list below — not "all interactive elements"** |
| Destructive | `--color-danger` | `#ea4b4b` | **Not used in this phase** — Phase 39 has no destructive actions (see Copywriting Contract) |

**Accent (`--color-primary` family) reserved EXCLUSIVELY for:**
1. The active **"?" affordance state** (`.is-active` → `--color-primary-soft` bg + `--color-primary-dark` text, exactly mirroring `.settings-gear-btn.is-active`).
2. The active **Help** nav entry (`data-nav` active marking, same as other nav items).
3. **Soft help headings** — page title (Display) and card/section headings (Heading) rendered in `--color-primary-deeper` (the D-05 soft-type deep garden green). This is a *text-color* use of the accent family, deliberate to the calm help voice.
4. The **featured personalization card** — raised `--color-primary`/`--color-border` primary-border card with a "Start here" tag (D-04), and the active/featured **rail item** (`--color-primary-soft` bg + `--color-primary-dark`/`-deeper` text).
5. The **contact-band primary "Write to us" button** and the search no-match "write to us" fallback link (`--color-primary-dark`).
6. **Focus-visible outline**: `2px solid var(--color-primary)`, `outline-offset: 2px` (existing app pattern) — on every interactive element in new surfaces.

Everything else uses **neutral/surface tokens**: the **"Show me how" empty-state secondary button
is soft / non-accent** (D-22 — `--color-surface-secondary-btn` / `--color-green-100`, muted text),
so it never competes with the accent; the "Contact us" popover item, rail non-active items, body
links, technical-track chrome, and body ink use `--color-text` / `--color-text-muted`. The help
page is deliberately **calmer than the rest of the app** (Ben: "take them down a notch") — muted
ink, roomier cards, botanical warmth as de-escalation.

**Botanical illustration usage (D-05/D-14):** reuse existing repo art only — the
`assets/illustrations/watering-can.png` accent in the page header band; a light botanical motif
only. Dark mode: follow the established `invert(1) + screen blend` botanical pattern (Phase 11) —
no separate dark illustration set. All browser-chrome install glyphs are inline SVG, dark-aware.

## Copywriting Contract

Tone: **warmth of calm/wellness apps + clinician-domain fit** — explicitly NOT generic SaaS
product-tour energy. EN targets a **global audience including native US speakers** — natural,
native-quality English (D-17), NOT "simple English for non-natives". No emojis (project-wide).
Terminology matches the live UI everywhere — **Session Format, Heart-Wall, client/session** —
never patient/treatment. UI-chrome strings are `data-i18n`-keyed with 4-locale parity; help BODY
lives EN-only in `assets/help-content-en.js`. Final rendered `help.html` is Sapir-reviewed
(D-19 pipeline: Writer → Gate A factual → Gate B native → Gate C DNA/consistency → Sapir → Ben).

(each with an inline SVG of the real chrome element) | `help-content-en.js` |

**Microcopy rules:**
- No emojis in any new copy (project-wide).
- No imperative-only EN that forces unnatural Hebrew imperatives — headings/labels are noun/noun-phrase (Phase 24 D-05).
- Per-browser install content is **computer-only** and branches by detected environment; **never** a single universal "Install" button.
- P1 steps read **stupid-proof**: full numbered steps, no assumed knowledge, every on-screen label quoted via `{ui:key}` live interpolation (D-13/D-23) so instructions always match what the user sees — including a Hebrew UI under EN help text.

**Destructive actions in this phase:** **NONE.** Phase 39 creates, deletes, or mutates no data.
`--color-danger` is intentionally unused. (Help content *explains* destructive flows that exist
elsewhere — e.g. deactivate/2-device transfer — but introduces no new destructive UI.)

## Interaction Contract (phase-specific — consumed by checker, planner, executor)

Hard constraints. The build MUST match the approved mockup `hybrid.html` behavior.

**Accessibility — minimum interactive size:** every interactive element in new surfaces (the "?"
affordance, popover items, "Show me how" buttons, rail links, search input, contact buttons) has a
**44×44px minimum tap target** (`--tap-target-min`), inherited from the existing global rule and
matching cloud/gear. Visual icon boxes may render at 36×36 while the 44×44 hit area is enforced.

| Surface | Contract |
|---------|----------|
| "?" entry point (D-09/D-12) | Lives in `#headerActions` beside cloud + gear on **every** SW-registered app page. 36×36 visual / 44×44 tap, inline SVG, `.is-active` mirrors `.settings-gear-btn`, focus-visible = 2px `--color-primary` + 2px offset. RTL: order auto-flips (logical layout, no JS). MUST NOT regress the Phase 25 consolidated header. Order within `#headerActions` and Help's slot in `renderNav()` (likely last) are Claude's discretion. |
| "?" popover menu (D-09/D-10) | **Popover from day one** (not direct nav) — reuses the globe language-switcher popover pattern (positioning, dismiss-on-outside-click / Esc, RTL safety). Day-one items: **"Help center"** (→ `help.html`) + **"Contact us"** (→ mailto). Build it so Phases 40–42 can **append** entries (Replay welcome / Take tour / What's new) — addable list, not hardcoded pair. `.is-active` behavior while open is Claude's discretion. |
| Help page IA — Hybrid A+C (D-01) | New standalone `help.html` following the per-page pattern; nav entry via `renderNav()` (`data-nav` active); reuses `SharedChrome` footer + nav. Two-column desktop grid: **sticky rail menu** + **collapsed cards** content column. Clicking a rail item scrolls to AND expands the matching card. Scroll-spy highlights the active section. |
| Card behavior (D-02) | **Multiple cards stay open** — opening a card never auto-collapses another. Cards collapsed by default EXCEPT the featured personalization card. |
| Featured personalization card (D-04) | "Making Sessions Garden yours" — raised primary-border card with a "Start here" tag, **open by default at the top** of the content column. |
| Rail (D-03) | **Top-level sections only, UNNUMBERED**, grouped "The session loop" / "The technical bits" with a divider. Each rail item has a chevron that expands its sub-topics **in-rail** (no page jump); clicking a sub-topic expands the owning card and jumps to the topic anchor. Scroll-spy marks the active section. |
| Deep-link / anchors (D-11) | Every topic is **anchor deep-linkable** (`help.html#<topic-id>`). Empty-state "Show me how" buttons and future callers open `help.html` with the **matching card auto-expanded** and scrolled into view. Help **always opens at the top** by default — no per-page context-aware landing (no page→anchor map to maintain); context-targeting is reserved for the unambiguous empty-state deep-links only. |
| In-page search (D-06) | Client-side substring live-filter over card titles / topic headings / body text (~40 lines vanilla JS, offline). Filtering hides non-matching cards AND rail items; empty rail groups hide their group label + divider. **No-match → calm "write to us" contact fallback**, never a dead end. Debounce (if any) is Claude's discretion. Fuzzy/typo tolerance deferred. |
| Empty-state coaching (D-21/D-22) | First-run trio only — Overview no-clients, Sessions no-sessions-at-all, Reporting no-data — each gets: keep the calm sentence + a soft **non-accent** "Show me how" secondary button below it → opens `help.html` with the matching card auto-expanded. | Contact / closing band (D-07) | "Still need help?" band at the page foot: real-person reassurance + `contact@sessionsgarden.app` mailto button + a hint to the existing Report-a-problem flow in Settings ("nothing is ever sent automatically"). |
| Mobile treatment (D-08) | Rail hidden on small screens; a sticky search box + a collapsible **"Jump to a section"** dropdown pinned at top; cards behave identically to desktop. (Mobile is expectation-set, not officially supported — D-15/D-16.) |
| RTL / dark | Every new surface: **logical properties only** (`*-inline-*`, `inset-inline-*`), `[data-theme="dark"]` reads dark token vars, **no** physical `right:`/`left:`/`margin-left`, **no literal hex**. Botanical dark mode via the `invert(1) + screen blend` pattern. |
| Offline (HELP-07) | `sw.js` `PRECACHE_URLS` MUST gain `help.html` + `assets/help-content-en.js` + any new CSS/SVG assets. **Known repo gotcha:** the pre-commit hook bumps `CACHE_NAME` but `PRECACHE_URLS` edits need a **manual chore follow-up**. Verify by static test + **real offline navigation on an installed PWA** (Safari stale-SW is a known failure mode). |
| Anti-rot substrate (D-23/D-24/D-25) | (1) **Live-label interpolation** `{ui:settings.tab.backups}` renders the label's current value. (2) **Per-topic `covers` metadata** lists the files/pages each topic documents (seeds the Phase 43 gate).
