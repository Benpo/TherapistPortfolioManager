## Summary

This phase redesigns the visual identity of the existing landing page. The content, structure, pricing, and i18n are already complete (Phase 5). The scope is limited to replacing the current geometric/minimalist visual style with hand-drawn botanical doodle illustrations, adding a hero gradient, refining typography hierarchy, and improving card/section aesthetics to feel premium and natural.

The primary technical work is SVG authoring: hand-drawn doodle-style SVGs must be crafted inline in HTML. These differ fundamentally from the current stroke-only geometric SVGs — they require irregular, organic paths that mimic pencil or Micron pen drawing. The secondary work is CSS: hero gradient, typography refinements, card polish, and dark mode illustration inversion.

Because the stack is vanilla HTML/CSS/JS with no build tools, all changes are direct file edits. The existing pattern of `currentColor` for SVG strokes enables dark mode automatically — new illustrations must follow this same pattern so they render as black lines in light mode and white/cream lines in dark mode without extra CSS.

**Primary recommendation:** Author all doodle SVGs as path-based inline SVGs using `currentColor` for strokes and no fills. CSS handles theming. Keep all SVG viewBox dimensions consistent with existing layout anchors in `landing.css`.

### Pattern 1: Hand-Drawn SVG with currentColor
**What:** SVGs authored with irregular, slightly-wobbly path coordinates and `stroke="currentColor"`, no fill. The CSS `color` property on the container controls stroke color. In light mode the container has `color: var(--color-primary)` (green), producing black-ish doodle lines at low opacity for the hero, or full opacity for icons.

**When to use:** All botanical illustrations — hero plants, section dividers, feature card icons.

**Example:**
```html
<!-- aria-hidden="true" on all decorative illustrations -->
<div class="hero-botanical hero-botanical-left" aria-hidden="true">
  <svg width="220" height="380" viewBox="0 0 220 380"
       fill="none" stroke="currentColor"
       stroke-linecap="round" stroke-linejoin="round"
       xmlns="http://www.w3.org/2000/svg">
    <!-- Organic stem — slightly wobbly coordinates give hand-drawn feel -->
    <path d="M110,370 C107,310 103,250 109,195 C115,150 109,99 107,53 C106,34 107,17 109,6"
          stroke-width="2.3"/>
    <!-- Leaf — closed path with inner vein -->
    <path d="M108,318 C96,310 70,298 53,288 C67,295 90,307 108,318Z" stroke-width="1.7"/>
    <path d="M108,318 C86,308 69,296 53,288" stroke-width="0.9"/>
  </svg>
</div>
```

**Hand-drawn feel techniques:**
- Use coordinates with slight irregularities (not perfectly symmetrical)
- Vary stroke-width across segments: main stems ~2–2.5, leaves ~1.4–1.8, veins/details ~0.7–1.0
- Closed leaf shapes: path ends with `Z` after tracing one side of leaf, giving filled silhouette feel without fill
- Flower petals: overlapping elliptical paths, each slightly rotated
- Berries/seeds: small `<circle>` elements with stroke only
- Use `stroke-linecap="round"` and `stroke-linejoin="round"` globally on the SVG — this alone makes lines feel hand-drawn

### Pattern 2: Hero Gradient Background
**What:** CSS gradient on `.landing-hero` replacing the flat `var(--color-background)`.

**When to use:** Hero section only — creates visual separation as decided.

**Example:**
```css
/* Source: tokens.css primitive token values */
.landing-hero {
  background: linear-gradient(
    160deg,
    var(--color-green-700) 0%,          /* #1e5c3a — deeper green at top */
    var(--color-green-600) 30%,          /* #2d6a4f */
    var(--color-green-500) 55%,          /* #3a7d5f */
    var(--color-cream-warm-400) 100%     /* #fefaf3 — warm cream at bottom */
  );
}

/* Hero text must be readable on dark green background */
.landing-hero .hero-content {
  /* h1, tagline, subtitle colors override needed — white/cream on dark hero */
}

[data-theme="dark"] .landing-hero {
  background: linear-gradient(
    160deg,
    #0f2a1a 0%,
    #1a3528 40%,
    var(--color-background) 100%
  );
}
```

**Critical:** Hero text (h1, tagline, subtitle, badge, price note) currently uses `var(--color-primary)` and `var(--color-text-muted)`. On a dark green background these will be unreadable. CSS overrides scoped to `.landing-hero .hero-content` must set text to white/cream.

### Pattern 3: Feature Card Doodle Icons
**What:** Replace `<span class="feature-icon" aria-hidden="true">📁</span>` with inline SVG doodle icons. The SVG inherits color from `.feature-icon { color: var(--color-primary) }`.

**When to use:** All 6 feature cards.

**Implementation choice — two options:**

Option A (preferred): Inline SVG directly in the `<span>`:
```html
<span class="feature-icon" aria-hidden="true">
  <svg width="36" height="36" viewBox="0 0 36 36"
       fill="none" stroke="currentColor" stroke-linecap="round" stroke-linejoin="round">
    <!-- folder doodle with leaf growing from top -->
  </svg>
</span>
```

Option B: The i18n system in `landing.js` re-renders feature icons when language switches. The `icon` field in `LANDING_I18N.features` is injected as `innerHTML` of `.feature-icon`. Since doodle icons are not language-dependent, the same SVG string should be used across all languages — DRY approach: define icon SVGs as JS constants at top of `landing.js`, reference them in each language's feature array.

**Recommended:** Option B (JS constants) because the i18n renderer controls the DOM. If icons are only in HTML, they get wiped on language switch.

```javascript
// In landing.js — define once, reuse across all 4 languages
var DOODLE_ICONS = {
  folder: '<svg width="36" height="36" ...><!-- folder doodle --></svg>',
  lock:   '<svg width="36" height="36" ...><!-- lock doodle --></svg>',
  // ...
};

// Then in each language's features array:
{ icon: DOODLE_ICONS.folder, title: 'Everything organized, nothing lost', ... }
```

### Pattern 4: Dark Mode Illustration Inversion
**What:** Current botanical SVGs use `color: var(--color-primary)` on container — in dark mode `--color-primary` becomes `#5aacac` (soft teal). For illustrations, the design decision is white/cream lines on dark, not teal. Override needed.

```css
[data-theme="dark"] .hero-botanical {
  color: var(--color-cream-warm-200); /* #fdf5e8 — warm cream lines */
}

[data-theme="dark"] .botanical-divider {
  color: var(--color-cream-warm-300);
}

[data-theme="dark"] .feature-icon svg {
  color: var(--color-primary); /* Keep teal for icons — they're functional, not decorative */
}
```

### Anti-Patterns to Avoid
- **Using `fill` in doodle SVGs:** Breaks the hand-drawn feel and dark mode — use stroke-only
- **Hardcoding colors in SVG attributes:** Must use `currentColor` so CSS tokens control the color
- **Using `direction: ltr` or physical CSS properties in new rules:** Must use logical properties (inline/block) for RTL
- **Modifying only HTML icon spans:** The i18n JS re-renders them on language switch — HTML-only changes will be lost
- **Making hero text the same color as app sections:** Hero gradient is dark green — text must contrast (white/cream)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Dark mode illustration color | Custom JS color switcher | CSS `[data-theme="dark"] .class { color: ... }` | Already established pattern in tokens.css and landing.css |
| Responsive illustration hiding | JS viewport detection | CSS `@media` (already in landing.css) | Media queries for `.hero-botanical` already exist at 1100px and 820px |
| RTL illustration mirroring | JS transform manipulation | CSS `transform: scaleX(-1)` already on `.hero-botanical-right` | Established pattern — extend it |
| SVG icon i18n sync | Separate icon rendering system | `DOODLE_ICONS` constants in existing `landing.js` i18n renderer | Simpler than a second rendering pass |

**Key insight:** The existing CSS architecture (tokens, `currentColor`, `[data-theme]`, logical properties) handles 90% of visual complexity automatically. New work is primarily authoring SVG path coordinates.

## Common Pitfalls

### Pitfall 1: Feature Icons Disappear on Language Switch
**What goes wrong:** Doodle icons added to `landing.html` HTML are correct on first load, but vanish when user switches language.
**Why it happens:** `landing.js` re-renders all feature cards via `innerHTML` from `LANDING_I18N.features[].icon`. HTML-only changes are overwritten.
**How to avoid:** Define doodle icon SVGs as JS constants in `landing.js`. Use them in ALL four language objects' `features` arrays.
**Warning signs:** Icons render correctly in English, disappear after switching to Hebrew.

### Pitfall 2: Hero Text Unreadable on Dark Green Gradient
**What goes wrong:** After adding the hero gradient (dark green top), the h1 and tagline remain `color: var(--color-primary)` — dark green text on dark green background.
**Why it happens:** Current hero styles assume a light background.
**How to avoid:** Add `.landing-hero .hero-content` overrides for h1, tagline, subtitle, badge, price note — set to white or `var(--color-cream-warm-100)`. Scope all overrides to `.landing-hero` to avoid affecting other sections.
**Warning signs:** Title is invisible or very low contrast in Chrome DevTools.

### Pitfall 3: SVG Illustrations Use Physical CSS Properties
**What goes wrong:** Adding `left: -15px` or `top: 0` on new illustration containers breaks Hebrew RTL layout.
**Why it happens:** Existing `.hero-botanical-left { left: -15px }` is a physical property — it's a known accepted exception in this codebase for absolute-positioned decorative elements. New additions must follow the same pattern or use `inset-inline-start`.
**How to avoid:** For decorative absolute-positioned illustrations, mirror the existing pattern exactly. For any new non-absolute layout, use logical properties.
**Warning signs:** In Hebrew (`dir="rtl"`) the illustrations overlap or appear on wrong side.

### Pitfall 4: SVG Paths Too Regular (Not Hand-Drawn)
**What goes wrong:** SVG paths with perfectly symmetric coordinates look computer-generated, not hand-drawn.
**Why it happens:** Natural tendency to make clean numbers.
**How to avoid:** Introduce small asymmetries — vary control points slightly, use odd numbers in coordinates (e.g., `C107,310` not `C110,310`), vary stroke-widths along a single illustration, add slight wobble to "straight" lines using curves with minimal arc.
**Warning signs:** Illustrations look like a CSS shape or icon library, not a notebook sketch.

### Pitfall 5: Dark Mode Hero Gradient Insufficient Contrast
**What goes wrong:** Dark mode hero gradient too similar to the rest of the page — loses the visual separation the gradient is supposed to create.
**Why it happens:** Dark mode backgrounds are already dark, reducing gradient range.
**How to avoid:** In dark mode, use a distinctly deeper/different hue at the top of the gradient (near-black forest green) transitioning to the standard dark background color at the bottom. Test both light and dark in browser before finalizing.

## Illustration Design Guide

### Reference Style Summary (from Sapir's 4 reference images)
The 4 images share these visual characteristics:
- **Line weight:** Thin primary strokes (0.8–1.2pt equivalent), slightly thicker stems (1.5–2.5pt)
- **Character:** Organic curves, no perfectly straight lines, slight wobble
- **Density:** Detailed but not cluttered — 3–8 distinct elements per illustration area
- **Silhouette closure:** Many leaf shapes are closed paths (stroke-only closed shapes read as filled from distance)
- **Cross-hatching:** Optional — references #4 used cross-hatching for texture; keep minimal for SVG complexity

### Suggested Illustration Set for this Phase
| Location | Illustration | Complexity |
|----------|-------------|------------|
| Hero left | Tall flowering plant (extend current plant with doodle style) | High |
| Hero right | Branching botanical with berries/buds (extend current) | High |
| Section divider (×1) | Branch with small leaves + center flower | Low-medium |
| Feature: organize | Open folder with small leaf/sprout inside | Low |
| Feature: privacy | Simple padlock with vine wrapped around | Low |
| Feature: offline | Laptop/device silhouette with small leaf | Low |
| Feature: tracking | Small chart bars with leaf growing from top | Low |
| Feature: no subscription | Coin/circle with leaf motif | Low |
| Feature: paper | Open notebook with pen and small flower | Low |

### SVG Authoring Strategy for Hand-Drawn Look
1. Start with a smooth cubic Bezier path for the main form
2. Slightly offset control points from symmetric positions (by 3–8 units in a 200×300 viewBox)
3. For leaves: draw one side as a curved path, close with `Z` — the closure itself creates the leaf shape
4. For flowers: 5–6 petal paths each rotated ~60–72 degrees, overlapping slightly at center
5. For wavy lines (roots, grass): use short S-curves strung together
6. Always set `stroke-linecap="round" stroke-linejoin="round"` at SVG root
