# Phase 46.1: Preview & Edit Experience Redesign - Research

**Researched:** 2026-07-18
**Domain:** Redesign of two already-shipped rich-text surfaces (7 session note fields + export Step-2) in a vanilla local-first PWA — codebase archaeology, not greenfield
**Confidence:** HIGH (all findings verified against live source in this session)

## Summary

This is a **re-presentation phase**: RTXT-01/04/05 already ship (Complete under Phase 46), and every
engineering mechanism the new surfaces need already exists in `assets/rich-toolbar.js`,
`assets/text-edit.js`, `assets/export-modal.js`, and `assets/app.css`. The work is a **surface rewire**
— replace the target-state preview toggle with a current-state Edit/Preview switcher, replace the
`--color-surface-alt` preview background with the ratified Frame treatment, and switch both surfaces
from a below-the-editor preview *pane* to an **in-place swap** (preview replaces the editor in the same
box) — while keeping the load-bearing mechanisms (46-15 sizing floor, sticky/persistent bar,
bar→field dispatch, Gap-16 mousedown guard, module undo history, autogrow) byte-intact underneath.

There are **no new dependencies** (no build step, no framework), no data-tier changes, and no external
libraries to vet. The renderer (`window.MdRender.render`, escape-first) and the edit chokepoint
(`window.TextEdit.editInsert` via `document.execCommand('insertText')`) are both irreplaceable and must
not be modernized. The single biggest planning risk is **naming drift**: the UI-SPEC and the two ratified
sketches (008/009) use *aspirational* class names (`.swap`, `.tb-scroll`, `.rt-toolbar.is-muted`,
`.never-mute`, `.preview-chip`, `.preview-frame`, `.step2-editor`) that **do not exist in shipped code** —
the shipped names are `.rich-toolbar`, `.rich-toolbar-btn`, `.rich-toolbar-preview`, `.export-editor`,
`.export-edit-area`. The planner must decide the mapping (rename vs. add) per element and NOT assume the
sketch selectors exist.

**Primary recommendation:** Plan this as an in-place edit of the three existing modules + `app.css`,
driven RED-first against jsdom behavior tests plus the existing WebKit Playwright probe. Rewire the
preview mechanism from a *sibling pane* (`ta._notePreview` inserted `afterend`) into an *in-place swap*
(preview element occupies the field's box; field hidden while previewing). Add the segmented switcher to
the bar OUTSIDE the scroll strip. Repoint the preview background token. Keep every other mechanism.

- **Copy (EN, corpus of record):** switcher segments **Edit** (pencil) / **Preview** (eye); chip
  **PREVIEW**; export CTAs **Continue** / **Back** / **Export**; ephemeral note *"Edits here apply to this

### Pattern 1: In-place swap (replaces the sibling-pane pattern)
**What:** On enter-preview, render the value into a Frame element that occupies the field's box and hide
the textarea; on leave-preview, show + refocus the textarea. **When:** both surfaces. **Example (from the
ratified sketch 009-A — canonical visual reference):**
```javascript
// Source: .planning/sketches/009-note-field-preview/index.html:626-648 (setMode, variant a)
function setMode(mode) {
  const ta = currentField.querySelector('textarea');
  const pv = currentField.querySelector('[data-pv]');
  previewing = mode === 'preview';
  tb.querySelectorAll('[data-mode]').forEach(b => b.classList.toggle('on', b.dataset.mode === mode));
  tb.classList.toggle('is-muted', previewing);              // dim formatting buttons
  if (previewing) {
    renderInto(pv, ta);                                     // MdRender.render in production
    pv.classList.add('is-on');
    pv.querySelector('.preview-frame').style.minHeight = ta.offsetHeight + 'px'; // same box height
    ta.classList.add('is-hidden');
  } else {
    pv.classList.remove('is-on'); ta.classList.remove('is-hidden'); ta.focus();
  }
}
```

### Pattern 2: Pinned current-state switcher, outside the scroll strip
**What:** Wrap the formatting controls in a scrolling strip; render the segmented switcher as a sibling
AFTER it so it never scrolls. **Example (sketch 008-B):**

```javascript
// Source: .planning/sketches/008-export-preview-edit-model/index.html:972 + 958-968

```
**Formatting-click-in-preview returns to Edit first, then applies** (008-B:979-983): `if (previewing)
setMode('edit'); ta.focus(); ...apply`. [VERIFIED: sketch source]

### Anti-Patterns to Avoid
- **Reusing `--color-surface-alt` for the preview background** — this IS Gap 15. The Frame uses
  `--color-surface` + `--color-green-border-ghost` border. [CITED: UI-SPEC §Color LOAD-BEARING rule]
- **A single target-state toggle button** whose active (green) styling contradicts its label — this IS
  Gap 14b ("Edit"-labelled-and-green while actually previewing). Use a two-segment current-state switcher.
- **Stacked editor+preview in one scroll container** — rejected concept (Gap 14c). In-place swap only.
- **Modernizing `editInsert`** into the range-replace textarea API — it wipes the native undo stack in
  Chrome/Safari. `document.execCommand('insertText')` is irreplaceable. [CITED: text-edit.js:25-35]
- **Assigning raw note value to innerHTML.** Only `MdRender.render()` output may be assigned. [CITED:
  rich-toolbar.js:581-587]
- **Feeding a `getBoundingClientRect` (physical) value into a logical inline-inset property** — mirrors
  wrongly in RTL. Use physical left/top for measured overlays; logical props for static layout. [CITED:
  rich-toolbar.js:506-513, MEMORY reference-rtl-logical-props-physical-coords]

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Markdown → HTML | A new renderer | `window.MdRender.render` | Escape-first, XSS-safe, matches PDF grammar exactly |

## Common Pitfalls

### Pitfall 1: Sketch class names don't exist in shipped code
**What goes wrong:** A plan/executor writes CSS for `.swap`, `.tb-scroll`, `.preview-frame`,
`.preview-chip`, `.step2-editor`, `.rt-toolbar` — none of which the shipped DOM emits. The shipped names
are `.rich-toolbar`, `.rich-toolbar-group`, `.rich-toolbar-btn`, `.rich-toolbar-preview`, `.export-editor`,
`.export-edit-area`. **Why:** the sketches are throwaway HTML with their own markup; the UI-SPEC quotes
sketch selectors as the *visual* reference, not the shipped DOM. **How to avoid:** the plan must state, per
new class, whether it is (a) a rename of a shipped class or (b) a genuinely new element, and the executor
edits `rich-toolbar.js`'s `buildToolbar`/`makeButton` to emit whichever names the plan chose.
**Warning sign:** CSS referencing a selector no `document.createElement(...).className = ...` in the JS
ever sets.

### Pitfall 2: Deleting `revealPreviewPane` / `_notePreview` too aggressively
**What goes wrong:** In-place swap makes the below-fold scroll math moot, tempting a full delete — but the
export surface still lives inside `.export-edit-area` (`overflow-y:auto`), and if the swap keeps the frame
as a sibling that can still overflow, some reveal may still be needed. **How to avoid:** convert, then
prove with the WebKit probe (assertion set E) that the frame is visible on open WITHOUT scroll before
removing the math. **Warning sign:** preview opens but sits below the fold on the export surface at
1440×820.

### Pitfall 3: Focus/keyboard on in-place swap
**What goes wrong:** Hiding the focused textarea (`display:none`) drops focus; `Ctrl+E` handler bound to
the textarea's keydown stops firing while previewing (the textarea is hidden/blurred). **Why:** the
existing keydown handlers (`onFieldKeyDown`) are bound to the textarea; a hidden textarea receives no
keys. **How to avoid:** bind `Ctrl/Cmd+E` where it still reaches while previewing (bar-level or document-
level, guarded to the active field), and on leave-preview call `ta.focus()` (sketch does this). **Warning
sign:** can enter preview but `Ctrl+E` won't exit it.

### Pitfall 4: RTL — switcher pin + chip position + icon mirroring
**What goes wrong:** Pinning the switcher with physical `right`, or the chip with physical `left`, mirrors
wrong in Hebrew. **How to avoid:** switcher pins at the bar's logical end (flex order); chip uses
`inset-inline-start` (as both sketches do). Directional glyphs (indent/outdent/undo/redo) carry
`.icon-flip` and mirror via `[dir="rtl"] .icon-flip svg { transform: scaleX(-1) }`; eye/pencil/bold/italic
do NOT mirror. [CITED: UI-SPEC §RTL, sketch 008:182]

### Pitfall 5: WebKit-only geometry bugs
**What goes wrong:** Chromium-only screenshot gates miss Safari bugs (viewBox-only SVG = 0×0 in WebKit;
date-input alignment; sticky-bar behavior). **How to avoid:** run the Playwright **WebKit** probe on the
real geometry, not just jsdom. [CITED: MEMORY reference-webkit-chromium-svg-visual-verification;
tests/webkit/46-export-step2-layout.mjs]
