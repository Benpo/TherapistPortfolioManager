# 46.1 — Bold/Italic Toggle Semantics: Research + Behavior Contract (for ratification)

**Status: RATIFIED by Ben, 2026-07-19 — D1 = swap (option 1), D2 = empty pair, caret between
(option 1), D3 = mixed → apply (option 1). All three per recommendation. The contract below
is the implementation spec.**

Scope: replace the naive span-wrapping in `TextEdit.toggleWrap` (assets/text-edit.js)
with a state-based emphasis toggle. Same public surface (`toggleWrap(value, s, e, marker)`
→ `{value, selStart, selEnd, replacement}`), same single call site
(rich-toolbar.js `applyTransform` → `undoRecord` + one `editInsert`). Pure transform,
no UI change, no renderer change.

---

## 1. Real constraints — what the grammar actually accepts (verified by execution)

The inline grammar lives in THREE character-identical places: md-render.js `applyInline`
/ `stripInline`, pdf-export.js `stripInlineMarkdown` (regex form) and `parseInline`
(char-scanner form). Facts, each verified against the live files:

- **Per-line only.** Both content classes forbid `\n`; `parseInline` aborts a span scan
  at a newline. `**line one\nline two**` renders the markers literally on screen AND in
  the PDF. Emphasis can never cross a newline.
- **Hug rule (D-08).** Span content must open AND close on a non-whitespace, non-star
  char. `** bold **` and a wrap of an untrimmed selection are literal.
- **No inner `*` at all.** The content class is `[^*\n]` — a bold span cannot contain an
  italic span or any star. Nested emphasis is inexpressible inside one span.
- **Italic prefix/suffix guards.** Italic requires a non-star (or line-start) before its
  opening `*` and no star after its closing `*`. Consequence: an italic span directly
  adjacent to a bold span (`**bold***it*`, `*it***b**`) is accepted by md-render's
  sequential regex passes but **refused by parseInline's scanner** — screen shows
  italic, PDF shows literal stars. Any emitted `***` cluster diverges the two pipelines.
- **`***word***` diverges AND breaks a shipped invariant.** Screen:
  `<em><strong>word</strong></em>`. PDF: literal `*` + bold `word` + literal `*`. Worse:
  `stripInlineMarkdown('***word***')` = `word` while `parseInline` join = `*word*` —
  the T-45-03 join-invariant (`parseInline(x).join === strip(x)`) is **violated in
  shipped code today** for `***` input (the Phase 45 fuzzer never generated adjacent
  markers). Under bidi reorder this can misplace bold runs in the PDF.
- **No bold-italic font face.** The PDF registers exactly three faces: Heebo normal,
  Heebo bold, Rubik-as-Heebo italic. Even with grammar work, a combined bold+italic
  segment has nothing to render with.
- **Block prefixes are position-sensitive.** Wrapping across a list marker
  (`**- item**`) or heading prefix (`**## Title**`) destroys list/heading-ness in both
  pipelines (the leading `*` defeats `isListItem` / the heading regex). Markers must be
  emitted only inside a line's content region (after indent + `-`/`*`/`N. ` list marker
  or `#{1,3} ` heading prefix). Emphasis *inside* a heading's text is fine (screen
  renders `<strong>` in the `<h*>`; PDF strips markers — heading is already bold).

### Verified defects in today's `toggleWrap` (repro'd against live code)

| # | Action | Result today | Rendered |
|---|--------|--------------|----------|
| 1 | Bold a multi-line selection | one pair around the block | literal `**` on screen + PDF |
| 2 | Bold `text` inside `**bolded text**` | `**bolded **text****` | literal `**bolded ` + bold `text` + `**` |
| 3 | Bold a selection that includes the markers (`**word**`) | `****word****` | literal `**` + bold + `**` |
| 4 | **Italic** toggle on the content of `**word**` | `*word*` | silently converts bold → italic (the 1-star "wrapped" check matches half of the `**`) |
| 5 | Bold toggle on the content of `*word*` | `***word***` | screen bold-italic, PDF literal stars + T-45-03 violation |

## 2. Reference behavior (cited)

| Case | Obsidian | Typora | GitHub (markdown-toolbar-element) | VS Code (Markdown All in One) |
|------|----------|--------|-----------------------------------|-------------------------------|
| (a) selection fully inside an emphasized span | Un-bolds the **containing span** (historic/expected "smart toggle"; regressions to nesting are filed as bugs) | Un-bolds the **entire containing span** (issue #5528 complains it un-bolds the *whole* span, confirming that is its behavior) | Only detects markers hugging the selection (`expandSelectedText` one marker-width out); deeper inside a span it re-wraps (nests) | Only `isWrapped(selection)` exact check; inside a span it nests |
| (b) partial overlap with a span | No documented handling (naive wrap → nesting artifacts filed as bugs) | Rich-text model: applies/clears like a word processor | Naive wrap | Naive wrap ("no logic for partial overlaps … potentially nested or malformed") |
| (c) multi-line selection | Single pair around the whole selection (valid in its CommonMark scope — emphasis spans soft breaks) | Per-paragraph via its rich-text model | Single pair (bold/italic use single-line `blockStyle`, not `multilineStyle`) | Single pair ("multi-line selections are treated as a single unit") |
| (d) selection containing several bold fragments | Naive wrap → nests | Word-processor rule: mixed → make all bold | Naive wrap | Naive wrap |
| (e) caret-only, no selection | Plain text: inserts empty pair, caret between. Inside a span: un-bold containing span. At end of span content: **skips past the closing markers** (0.14.4 release behavior) | Un-bolds containing span from caret | Expands to the word under the caret | Expands to word under caret; special-cases `**|**` → remove the empty pair |
| (f) across list marker / heading prefix | Not documented in cited sources | Rich-text model sidesteps it | Not handled for bold (lists have separate multiline styles) | Not handled |

Sources: Obsidian forum [115393](https://forum.obsidian.md/t/toggle-bold-toggle-italic-only-applies-formatting-does-not-remove-it-no-longer-a-toggle/115393) (expected = true toggle, remove on already-bold), [66671](https://forum.obsidian.md/t/toggle-bold-italic-including-extra-spaces-after-1-4-5/66671) (space-trim semantics + selection-update complaints), [32239](https://forum.obsidian.md/t/make-the-hotkey-for-bold-and-italic-behave-like-in-document-processors-word-google-docs/32239) (0.14.4 "smartly skip over the closing markers"); Typora [issue #5528](https://github.com/typora/typora-issues/issues/5528); [github/markdown-toolbar-element `src/index.ts`](https://github.com/github/markdown-toolbar-element) (blockStyle/expandSelectedText/trimFirst/wordSelection); [yzhang-gh/vscode-markdown `src/formatting.ts`](https://github.com/yzhang-gh/vscode-markdown) (isWrapped/wrapRange, `**|**` removal).

Takeaway: **nobody handles (b)/(d) well except word processors.** Obsidian's genuinely
good ideas are: containing-span un-bold, space-trimming, and caret skip-out. Where
Obsidian has no answer (multi-line in OUR grammar, mixed fragments, `***`), the contract
below borrows the Word/Google-Docs state rule, which Typora also follows.

## 3. The behavior contract

Core reframe: the toggle is **state-based, not wrap-based**. Every invocation first
computes the selection's effective emphasis state for the pressed marker, then applies
exactly one of REMOVE / APPLY / a caret micro-rule. Emitted markup is, by construction,
markup BOTH pipelines accept.

**Definitions.**
- *Span*: a maximal same-marker run the grammar accepts on that line (tokenize each
  affected line bold-first-then-italic, mirroring renderer order — this is what kills
  defect #4: the italic tokenizer never sees the stars a bold span owns).
- *Content region* of a line: everything after leading indent + list marker + its
  whitespace (via the existing `parseListLine`) or after a `#{1,3} ` heading prefix.
- *Countable chars* of a selection: its non-whitespace chars inside content regions,
  excluding marker chars.
- *State*: FULL if every countable char sits inside a same-marker span's content and
  there is at least one countable char; else MIXED/NONE.

**Clauses.** (Each: behavior — why.)

- **C1 — Normalize first.** Trim leading/trailing whitespace off the selection; a
  selection whose edges include a span's markers is treated as that span's content.
  *Why: the hug rule makes untrimmed wraps literal; matches Obsidian's space-trim and
  GitHub's `trimFirst` + marker expansion.*
- **C2 — Selection with state FULL → un-emphasize every intersected span entirely**
  (the containing span, however deep inside it the selection sits). No splitting, no
  nesting, ever. *Why: Ben's directive; Obsidian/Typora semantics; Word-style splitting
  is frequently inexpressible under the hug rule (a remainder ending in a space cannot
  be re-wrapped).*
- **C3 — Selection with state MIXED/NONE → make it emphasized:** unwrap every span the
  selection intersects (partially or fully), then wrap, per line, the union of the
  trimmed selection with the intersected spans' content, in ONE pair per non-empty
  line-segment. *Why: Word/Typora "mixed → apply"; the union avoids splits and marker
  collisions; per-line pairs are the only form the grammar accepts (constraint 1).*
  Partial overlaps (case b) fall out of C3: the touched span's un-selected remainder is
  absorbed into the union and stays emphasized — bold *grows*, never fragments.
- **C4 — Multi-line: one state, computed across the whole selection; then C2/C3 applied
  per line.** Line-segments that clamp to empty (blank lines, whitespace-only, pure
  marker/prefix regions) are skipped and never wrapped. *Why: per-line independent
  toggling would un-bold some lines while bolding others in one press; skipping blanks
  keeps markers hugging.*
- **C5 — Clamp to content regions.** Emitted markers never sit before a list marker or
  heading prefix and never cross one. A selection that contains only prefix/marker chars
  is a no-op. *Why: constraint "block prefixes are position-sensitive."*
- **C6 — Caret rules (no selection):**
  (i) caret inside a same-marker span's content → C2 on the containing span (caret
  mapped to the same character);
  (ii) caret exactly at the end of a span's content (just before the closing marker) →
  move the caret past the closing marker, **zero text edit** — the type-bold-then-
  continue-plain flow;
  (iii) caret between an empty pair (`**|**` / `*|*`) → remove the pair;
  (iv) otherwise → per DECISION D2.
  *Why: (i)+(ii) are Obsidian 0.14.4 smart-toggle; (iii) is the existing behavior and
  MAIO's; without (ii), toggling off after typing a bold word would destroy it.*
- **C7 — Cross-marker collisions are resolved, never emitted.** The result may not
  contain `***` (three-plus consecutive stars) or an italic `*` adjacent to a bold
  `**`. Toggling one marker over the other marker's span behaves per DECISION D1.
  *Why: every `***` cluster diverges screen vs PDF and can break T-45-03 (constraint
  section); no bold-italic font face exists. This is a deliberate, permanent deviation
  from Obsidian.*
- **C8 — One toggle = one undo step, no-ops are free.** The transform returns a single
  spanning `{start,end,text}` replacement covering all affected lines (precedent:
  `renumberOrderedBlock`), applied through one `undoRecord` + one `editInsert`. A no-op
  outcome (C5 empty, C6-ii caret move, D1-if-no-op) returns an empty-length replacement
  so the caller makes ZERO `editInsert` calls and the undo stack is untouched.
- **C9 — Selection maps to the same text.** After APPLY, the selection covers the
  wrapped content (per-line: first segment's content start → last segment's content
  end); after REMOVE, the formerly-emphasized content. *Why: makes the toggle
  operationally reversible in place — and it is exactly what Obsidian's 1.4.x selection
  bugs got wrong.*
- **C10 — Idempotence, stated precisely.** On a *pure* state (FULL or NONE), toggle ∘
  toggle = identity (text and selection). A MIXED selection converges to FULL after one
  APPLY; from there the involution holds. (Toggle-twice on mixed = "all plain", exactly
  as in Word — the original interleaving is recoverable only via undo. This is the
  correct reading of "toggle twice = original", and the undo step guarantees the literal
  reading is one keystroke away.)
- **C11 — Italic is bold with `m=1`.** All clauses apply symmetrically to `*`, with the
  italic tokenizer running on the bold-tokenized residue (C-definitions) and C7 guarding
  both directions. `***` bold+italic combinations are **excluded from the emitted
  language entirely** (see C7/D1); hand-typed `***` remains the renderers' problem and
  is flagged below.

**Explicit deviations from Obsidian** (each forced, none stylistic):
1. Multi-line → per-line pairs (our grammar has no soft-break emphasis).
2. Mixed/multi-fragment → Word's "apply" rule (Obsidian nests, which is its own filed-bug behavior, and the exact defect Ben reported).
3. `***` / bold+italic → excluded (PDF font set, parser divergence, T-45-03).
4. (Only if D1 resolves to swap) cross-marker toggle swaps instead of nesting.

**Flagged pre-existing issue (out of scope, needs a home):** the T-45-03 join-invariant
violation for hand-typed `***word***` exists in shipped code today, independent of the
toolbar. The contract stops the toolbar from ever *creating* it; a typed one still
diverges screen/PDF. Recommend a separate quick-fix ticket (either harden `parseInline`'s
literal fallback to match the regex pass, or harden both to refuse — needs its own
mini-design; do not fold into this phase).

## 4. DECISIONS for Ben

**D1 — Toggling marker X over a span of marker Y (bold↔italic collision).**
The grammar + PDF fonts cannot express bold+italic. Options:
  1. **Swap (recommended):** the pressed marker wins — unwrap the other marker, wrap the
     pressed one (`**word**` + Italic → `*word*`). Visible, honors the literal button
     press, every state stays reachable (plain →B bold →I italic →I plain).
  2. No-op: pressing Italic on bold text does nothing. Content-safe but reads as broken
     — there is no channel to tell the user why.
  3. Extend the grammar to accept `***`. Rejected: no bold-italic font face in the PDF,
     `parseInline` rework, and it resurrects the T-45-03 hazard. Not this phase, likely
     not ever.
Recommendation: **1** — with the D1 rule applied only when the collision is real (the
selection intersects the other marker's span); disjoint bold and italic on one line stay
untouched.

**D2 — Caret-only toggle in plain text (C6-iv).**
  1. **Insert an empty pair, caret between (recommended):** current behavior, matches
     Obsidian, optimized for the type-flow (press B, type, press B to exit via C6-ii).
  2. Expand to the word under the caret (GitHub/MAIO): nicer for "make this word bold"
     after the fact, but destroys the type-flow and surprises mid-word carets.
  3. Hybrid: word if the caret touches a word, empty pair in whitespace. Most magic,
     least predictable.
Recommendation: **1** — the type-flow is the therapist's main path, and C6-i already
covers "un-bold this word from inside".

**D3 — MIXED-state direction (C3).**
  1. **Mixed → apply (recommended):** Word/Google-Docs/Typora rule; the second press then
     removes everything (C10).
  2. Mixed → remove all: "toggle strips first". Rarely what users mean; makes bolding a
     region containing one bold word a two-press dance.
Recommendation: **1**.

## 5. Test sketch (all pure-transform, zero-npm node runner)

**Tier 1 — contract unit tests** (`tests/<phase>-emphasis-toggle.test.js`, vm-sandbox
idiom of `tests/text-edit.test.js`; no jsdom needed — the transform is pure). One
falsifiable case per clause, both markers where meaningful:

- C1: wrap trims (`sel = " word "` → `**word**` with the spaces outside); all-whitespace selection → no-op.
- C2: unwrap-exact; unwrap-inside (`**bolded text**`, sel=`text` → `bolded text`); unwrap when the selection includes the markers (kills defects #2, #3).
- C3: multi-fragment (`**a** mid **b**` select-all → `**a mid b**`); partial overlap grows the span (`pre **bold** post`, sel straddling the opening → union, single pair).
- C4: multi-line wrap emits one pair per non-empty line, blank line skipped, markers never adjacent to `\n` (kills defect #1); multi-line with all-bold lines → per-line unwrap.
- C5: selection over `- item` clamps after the marker (`- **item**`, never `**- item**`); heading line clamps after `## `; selection covering only the marker/prefix → no-op.
- C6: caret-inside-span unwraps containing span; caret at end-of-content returns a zero-edit caret move past the closer; `**|**` toggle removes the pair; plain-text caret per D2.
- C7: italic toggle on `**word**` per D1 (and never `*word*`-by-accident — kills defect #4); bold toggle on `*word*` per D1 (never `***word***` — kills defect #5); result never contains `***` anywhere (assert on every case's output).
- C8: every result carries exactly one spanning replacement; no-op cases return an empty replacement (caller-skippable, mirrors `noopRenumber`).
- C9/C10: for each pure-state case, apply the transform to its own output with the returned selection and assert byte-identical round-trip (text + selection).

**Tier 2 — emit-acceptance harness** (jsdom loader idiom of
`tests/45-pipeline-agreement.test.js`): for every Tier-1 output line, assert
(1) `MdRender.render` shows `<strong>`/`<em>` with no stray literal `*` from emitted
markers, (2) `PDFExport.__test.parseInline` emits the matching bold/italic segments, and
(3) the T-45-03 join-invariant holds on the emitted text. Plus a small deterministic
fuzz (seeded, no `Math.random` in workflows — a fixed seed array): random doc + random
selection + random marker → invariants: no `***` in output, join-invariant preserved,
double-toggle from pure state restores.

**WebKit probe: not warranted.** The transform emits full replacements through the
existing `editInsert` chokepoint; the only empty-replacement (pure-deletion) path it can
take is C6-iii pair-removal, which rides the already-probed `execCommand('delete')`
route (covered by `46.1-list-exit-pure-deletion.test.js` + the Phase 46.1 WebKit
screenshots). No new engine-dependent surface is introduced, so jsdom + the vm sandbox
fully cover the contract.
