# Phase 46.1 Plan 03: In-Place Preview Swap + Ctrl/Cmd+E + Export Lifecycle Summary

Upgraded the preview open/close from the shipped below-editor sibling pane to an in-place swap — the preview Frame replaces the editor in the same box (one thing on screen at a time) — added a `Ctrl/Cmd+E` mode toggle that keeps reaching while the textarea is hidden, and fixed the export-modal lifecycle (reset-preview-on-close + a Back->Continue dirty discard guard).

## What was built

- **In-place swap Frame (Task 1).** `buildPreviewPane` now builds a wrapper `div.rich-toolbar-preview` (`aria-live="polite"`) holding a floating `span.rich-toolbar-preview-chip` ("PREVIEW", via `toolbar.previewChip`) and a `div.note-rendered.rich-toolbar-preview-body` render target. The wrapper no longer carries `note-rendered` — the body does. `renderPreview` targets the BODY (so the chip survives re-renders): empty/whitespace value → `is-empty` on the wrapper + empty-state block into the body via `textContent`; non-empty → `body.innerHTML = window.MdRender.render(val)`, the SOLE innerHTML sink, with a `textContent` fallback.
- **The swap itself.** `openPreview` shows the Frame, renders, captures the textarea's `offsetHeight` and sets the Frame's `style.minHeight` to it BEFORE adding `is-hidden` to the textarea (so the autogrowing note fields' box does not collapse), then runs the reconciled reveal. `closePreview` restores the editor in place (removes `is-hidden` from the textarea, clears the Frame min-height, hides the Frame) without forcing focus. The editor-hide ownership moved out of `setMode` into the swap (plan 02 left it as a bridge).
- **Reveal reconcile (Pitfall 2).** `revealPreviewPane` is retargeted at the Frame wrapper and left as a harmless no-op safety (delta ≈ 0 when the Frame sits where the editor was) rather than deleted — it still guards the export edit-area's overflow-y fold, vertical-only, RTL-safe.
- **Preview-aware focusout guard + outside dismissal.** The deferred `onFieldFocusOut` hide now returns early when `_previewOpen && _previewField === field` (the swap-initiated blur is not a real focus-leave), so a note-field preview never self-closes the instant it opens. Genuine dismissal is a one-shot document `mousedown` listener bound (deferred) only for NOTE-field previews — a press outside the field's bar AND its Frame closes preview + undocks the bar; torn down in `closePreview`.
- **Public `resetPreview()`.** Restores an open preview to Edit (editor shown, Frame hidden, `.is-preview` cleared, Edit segment `is-active`/`aria-pressed`) WITHOUT focusing the field, for a closing modal.
- **Ctrl/Cmd+E toggle (Task 2).** `onFieldKeyDown` handles `Ctrl/Cmd+E` before the coarse-pointer gate — `setMode(_previewOpen ? "edit" : "preview", ta)` — and calls `stopPropagation` so the document fallback does not double-fire. A document-level `onDocKeyDown` (bound once in `mount`, removed in `unmount`) drives the return direction while the textarea is hidden: it fires only when the active element is not a registered textarea and a preview is open, calling `setMode("edit", _previewField)`.
- **Export lifecycle (Task 3).** `exportCloseDialog` calls `window.RichToolbar.resetPreview()` before hiding the modal, so a reopen of Step 2 starts in Edit with a fresh render. `onNext` is now `async`: the Step-1 regenerate awaits `App.confirmDialog({export.discard.*})` when `hasEditedPreview` is true, returning early (edits + step untouched) on cancel and proceeding (regenerate + advance) on confirm — reusing the same discard dialog the close path uses.

## Deviations from Plan

### Auto-fixed issues

**1. [Rule 1 - Bug] closePreview does not force focus (plan prose said it should refocus)**
- **Found during:** Task 1.
- **Issue:** The plan's Task 1 action says `closePreview()` should "refocus the textarea (`ta.focus()`) — matching the sketch." But `closePreview` is also called from `dockTo` on a field CHANGE (`if (_previewOpen && _previewField !== textarea) closePreview()`) and from `hideToolbar` when focus leaves the whole set. If `closePreview` refocused the old/leaving field there, it would yank focus back and fight the new focus (field change) or trap focus on blur.
- **Fix:** `closePreview` performs the DOM restore only (no focus). The refocus is owned by `setMode`'s edit branch (`closePreview(); ta.focus();`) and the Ctrl/Cmd+E return path (which routes through `setMode("edit", …)`). The "leaving preview refocuses" test still passes because the Edit segment click and Ctrl+E both go through `setMode('edit')`.
- **Files modified:** assets/rich-toolbar.js
- **Commit:** 9131cb4

**2. [Rule 2 - Missing critical functionality] Field-level Ctrl/Cmd+E stops propagation**
- **Found during:** Task 2.
- **Issue:** `Ctrl/Cmd+E` on a focused field bubbles to the document-level fallback. In a real browser, `setMode('preview')` hides (display:none) the focused textarea, dropping focus to `<body>`; the bubbling document handler would then see a non-registered active element + an open preview and immediately toggle back to edit — an instant self-cancel invisible to jsdom (no CSS).
- **Fix:** The field-level handler calls `ev.stopPropagation()` after `preventDefault`, so the document fallback never fires for the same keydown. The fallback still drives the return direction when the keydown originates at the document (hidden/blurred field).
- **Files modified:** assets/rich-toolbar.js
- **Commit:** 8432a36
