## Summary

Reviewed the full Phase 46.1 change set (diff base `d488085^..HEAD`): the state-based
emphasis toggle + module-owned undo stack in `text-edit.js`, the in-place preview swap /
current-state switcher / inert-in-preview bar in `rich-toolbar.js`, the export Step-2
content-driven sizing + section-toggle dirty guard in `export-modal.js`, the nav-guard
widening in `add-session.js`, the md-render leading-space fix, help/changelog/i18n content,
CSS, and all 22 test files. Every reviewed test file was executed and passes (verified by
running them, not by trusting exit banners); count guards are present throughout and the
fake-test detector allowlist addition for the CSS source-audit is justified. Security posture
is sound: `MdRender.render` remains the sole innerHTML sink for note content (escape-first,
audited by `46.1-frame-no-surface-alt`), the chip/empty-state/heading-menu/maximize icon are
all DOM-API-built, the `&nbsp;` injection happens post-escape on static replacement text, the
`<li value>` attribute stays `parseInt`-derived, and nav-guard destinations come from
app-authored hrefs only. No debug artifacts and no new planning-ID comment contamination in
shipped assets.

Four real defects were found, all traced and reproduced. The most serious: the export
preview Frame is left rendering the OLD document after a section-selection rebuild — the
exact surface the phase's own help copy tells users to trust before exporting.

## Critical Issues

### CR-01: Export preview shows a stale document after a section-selection rebuild

**File:** `assets/export-modal.js:992-1006` (rebuildEditorFromSelection), with contributing gap at `assets/export-modal.js:975-978` (onBack) / `assets/export-modal.js:621` (exportSetActiveStep — no preview reset on step change)
**Issue:** The preview swap state survives navigating Back from Step 2 (`onBack` →
`exportSetActiveStep(1)` never calls `resetPreview`; only the modal *close* path does, at
line 697). `rebuildEditorFromSelection` then replaces the buffer with a direct
`editor.value = md` — which fires no input event, so `schedulePreviewRender` never runs and
the open Frame keeps its old render. Reproduction: open export → Continue to Step 2 → switch
to Preview → Back → toggle any section checkbox (silent when the buffer is clean; or confirm
the discard dialog when dirty) → Continue. Step 2 now shows the Frame in Preview mode
rendering the **previous** selection's document while the actual export buffer holds the new
one — the editor is hidden, so the stale render is the *only* thing on screen. The user can
walk straight to Step 3 and export content they never saw. This directly contradicts the
shipped help copy ("Preview shows the finished document in place of the editor — check it
before choosing a format") and the changelog claim ("a framed view of the finished result").
Side effect of the same root cause: while parked on Step 1 with the preview still open,
document-level Ctrl/Cmd+E (`onDocKeyDown`) calls `setMode("edit", _previewField)` and focuses
the hidden Step-2 editor. Not covered by `46.1-export-lifecycle.test.js` (its Back/Continue
cases never open the preview first).
**Fix:**
```js
const rebuildEditorFromSelection = (selected) => {
  // A rebuilt buffer must never sit behind a stale Frame: return the surface
  // to Edit (same call the close path makes; forces no focus).
  if (window.RichToolbar && window.RichToolbar.resetPreview) {
    window.RichToolbar.resetPreview();
  }
  _exportState.selectedKeys = selected;
  // ... existing body unchanged
};
```
(Alternatively — or additionally — reset the preview in `onBack` when leaving Step 2, which
also removes the Ctrl+E-on-Step-1 quirk.) Add a lifecycle test case: preview open → Back →
toggle → Continue → assert the editor is visible / the Frame hidden / Edit segment active,
or that a still-open Frame contains the rebuilt content.

## Warnings

### WR-01: `currentLine`/`currentLineText` misidentify the line at caret 0 when the value starts with a newline

**File:** `assets/text-edit.js:84` and `assets/rich-toolbar.js:498` (same pattern in `linesTouching`, `assets/text-edit.js:207`)
**Issue:** `value.lastIndexOf("\n", sel - 1)` with `sel === 0` passes fromIndex `-1`, which
`lastIndexOf` clamps to `0` — so a newline at index 0 **matches**, and the helper returns an
invalid line record `{start: 1, end: 0, text: ""}`. Every transform keyed on the caret line
then acts on the wrong position. Verified by execution:
- `toggleWrap('\n- item', 0, 0, '**')` → `"\n****- item"` — the empty pair is inserted
  **before the next line's list marker**, violating the module's own documented clamp
  invariant ("a wrap can never destroy list- or heading-ness") and emitting a literal
  4-star run hugging a marker.
- `insertListMarker('\nfoo', 0, 0, 'ul')` → `"\n- foo"` — the bullet lands on line 2 while
  the caret stays on the empty line 1.
- `applyHeading('\nfoo', 0, 0, 1)` → `"\n# foo"` — same class.
Reachable whenever a note begins with a blank line (paste, or Enter + arrow-up/Home) and the
user presses a toolbar control at position 0. Also skews `isListLine`/`isHeadingLine`/
`refreshButtonState` at that caret. None of the 45 emphasis contract cases nor the fuzz
corpus start a value with `"\n"` at caret 0, so the suite stays green over this hole.
**Fix:**
```js
function currentLine(value, sel) {
  var start = sel > 0 ? value.lastIndexOf("\n", sel - 1) + 1 : 0;
  ...
}
```
Apply the same `sel > 0` / `s > 0` guard in `rich-toolbar.js currentLineText` and
`text-edit.js linesTouching`, and add a regression case (`'\nfoo'`, caret 0) to
`tests/46.1-emphasis-toggle.test.js` and `tests/text-edit.test.js`.

### WR-02: `toolbar.modeAria` i18n key missing from all four dictionaries

**File:** `assets/rich-toolbar.js:250`; `assets/i18n-en.js` / `i18n-he.js` / `i18n-de.js` / `i18n-cs.js` (key absent)
**Issue:** `buildSwitcher` labels the Edit/Preview group with
`t("toolbar.modeAria", "Edit or preview")`, but the key exists in **none** of the four
dictionaries — every locale, including Hebrew/German/Czech, gets the hard-coded English
fallback as the switcher's `aria-label`. All other toolbar keys are present in 4/4 locales
(swept every `t()`/`key:` reference in the file). The 25-11 i18n-parity gate passes
vacuously here because the key is missing from EN too, so parity holds trivially.
**Fix:** Add `"toolbar.modeAria"` to all four dictionaries (EN "Edit or preview", plus HE/DE/CS
translations), next to the existing `toolbar.*` block.

### WR-03: New section-toggle confirm dialog interleaves with the export modal's Escape handler on the shared confirm modal

**File:** `assets/export-modal.js:1032-1048` (onSectionToggle) interacting with `assets/export-modal.js:971` (onKey) and the single shared `#confirmModal` (`assets/app.js:1061-1166`)
**Issue:** `App.confirmDialog` is a single shared modal; both its own document-level Escape
handler AND the export modal's `onKey` (registered earlier, so it fires first) receive the
same Escape keydown. While the new section-toggle discard dialog is open with a dirty
buffer, pressing Escape (a) starts `exportCloseDialog(false)`, which opens the close-discard
confirm **on the same modal element** while the toggle dialog's promise is still pending,
then (b) the toggle dialog's Escape handler hides the modal — leaving the second dialog
pending and invisible, its listeners still bound, silently swallowing the next
Escape and stacking two sets of button listeners on the shared OK/Cancel buttons. The user
pressing Escape to mean "keep editing" is instead routed into a hidden discard flow. The
Escape-reopen loop on `exportCloseDialog`'s own confirm predates this phase, but
`onSectionToggle` adds a second concurrent consumer and widens the surface. Additionally,
`rebuildEditorFromSelection` runs after an `await` with no `_exportState` re-check — if any
future path nulls the state mid-dialog it throws on `_exportState.selectedKeys = selected`.
**Fix:** (1) In `onKey`, ignore Escape while the shared confirm is visible:
```js
const onKey = (e) => {
  if (e.key !== "Escape") return;
  const cm = document.getElementById("confirmModal");
  if (cm && !cm.classList.contains("is-hidden")) return; // a confirm owns this Escape
  exportCloseDialog(false);
};
```
(2) In `onSectionToggle`, after the `await`, guard: `if (!_exportState) return;` before the
revert/rebuild branches.

## Info

### IN-01: `togglePreview` is dead code — and it held the only cross-field preview close

**File:** `assets/rich-toolbar.js:880-885`
**Issue:** After the switcher rewire, nothing calls `togglePreview` (mode segments and both
Ctrl/Cmd+E directions route through `setMode`; verified by grep — zero call sites in assets
and tests). Note that its `if (_previewOpen) closePreview()` line was the only guard closing
another field's open preview before opening a new one; `setMode`'s preview branch calls
`openPreview(ta)` directly without it (currently unreachable cross-field in practice because
`dockTo` closes the old preview on focus change, but the invariant is no longer enforced at
the open site).
**Fix:** Delete `togglePreview`, or keep the defensive close by adding
`if (_previewOpen && _previewField !== ta) closePreview();` at the top of `openPreview`.
Related micro-cleanup: `onNext` (`export-modal.js:1006`) is declared `async` but contains no
`await` since the dirty check moved to the toggle moment — drop the modifier.

### IN-02: Typing immediately after a toolbar formatting action coalesces into the same undo step

**File:** `assets/text-edit.js:789-813` (undoRecord / undoNoteInput)
**Issue:** `undoRecord` seals the pre-edit state and resets `lastKind` to null; the
formatting edit's own input event then sets `lastKind='insert'` with a fresh `lastTime`.
Ordinary typing within the 700 ms window of the same kind therefore folds into the
formatting step — e.g. the D2 type-flow (click Bold → immediately type the word) makes one
undo remove the empty pair AND the typed word together, mildly at odds with the shipped help
copy ("undo and redo — one change at a time"). Design judgment call, flagged for awareness.
**Fix (if wanted):** after a structural edit, force the next input to open its own boundary
(e.g. have `undoRecord` set a "boundary-pending" sentinel that `shouldOpenBoundary` honors
once, or set `H.lastTime = 0` after the structural edit's own input is absorbed).

### IN-03: List-line detection disagrees on tabs between `LIST_RE` and `parseListLine`

**File:** `assets/text-edit.js:49` (`LIST_RE`, `^\s*`) vs `assets/text-edit.js:554` (`parseListLine`, `^( *)`) — and `assets/md-render.js:43` (`isListItem`, `^\s*`)
**Issue:** A pasted line like `"\t- text"` is a list line to `LIST_RE` (Tab key intercepted,
renumber anchors on it) and to the renderer, but `parseListLine` returns null (space-only
lead), so Enter-continuation never fires on it and `insertListMarker` treats it as a
non-list line — the bullet button prepends a second marker, producing `"- \t- text"`
(renders as a bullet whose body is a literal `"- text"`). Only reachable via pasted tabs.
**Fix:** Align the grammars — either make `parseListLine`'s lead `[ \t]*` (and define tab
depth) or tighten `LIST_RE` to `^ *` so the module consistently treats tab-led lines as
plain text.
