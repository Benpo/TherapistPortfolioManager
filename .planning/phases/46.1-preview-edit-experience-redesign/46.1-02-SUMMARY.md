# Phase 46.1 Plan 02: Current-State Edit/Preview Switcher Summary

Replaced the rejected single target-state preview toggle with a two-segment current-state Edit/Preview switcher pinned outside a horizontally-scrollable formatting strip, on both the shared note-field bar and the persistent export bar, driven by a new `setMode(mode, field)` controller.

## What was built

- **Scroll strip + pinned switcher (Task 1).** `buildToolbar` now wraps every formatting group (bold/italic, lists, heading dropdown, indent/outdent, undo/redo) in a new `div.rich-toolbar-scroll`, and appends a `span.rich-toolbar-swap` (`role="group"`, aria-labelled) as a sibling AFTER the strip. The switcher holds two `button.rich-toolbar-swap-btn` segments — `data-mode="edit"` (pencil + "Edit") and `data-mode="preview"` (eye + "Preview") — each carrying `aria-pressed` that mirrors `is-active`. Both surfaces get it for free because they share the one `buildToolbar`.
- **Old toggle removed.** Dropped the `preview` entry from `GROUPS`, the `spec.action === "preview"` labeled-button branch in `makeButton`, and deleted `updatePreviewButton` entirely along with all four call sites (dockTo ×2, openPreview, closePreview). No `[data-action="preview"]` control remains.
- **RTL glyph marking.** Net-new `icon-flip` class added in `makeButton` to the directional glyphs only (indent/outdent/undo/redo) via a `DIRECTIONAL` lookup; semantic glyphs (bold/italic/eye/pencil) are left unmarked. The CSS rule lands in plan 04.
- **setMode controller (Task 2).** `setMode(mode, field)` resolves the target field, flips `is-active` + `aria-pressed` together on the two segments, toggles `.is-preview` on the bar, hides/shows the editor (`is-hidden` on the textarea), and reuses the existing `openPreview`/`closePreview` plumbing. Preview resolves the field WITHOUT focusing it; edit refocuses.
- **Mode dispatch + return-and-apply.** `_dispatch` gained a mode-segment branch placed BEFORE the focus-anchor block (so a Preview click never focuses the export editor). A formatting gesture while previewing calls `setMode('edit', ta)` first, then applies — one click exits preview and performs the action.
- **Retargeted shipped tests.** `tests/46-persistent-bar-dispatch.test.js` Case B and `tests/30-export-stepper.test.js` case F now drive preview via the `[data-mode="preview"]` segment (real mousedown) and assert the segment's `aria-pressed`/`is-active` state; Case B's "preview did NOT focus the field" assertion is preserved.

## Deviations from Plan

### Auto-added critical functionality

**1. [Rule 2 - Missing critical functionality] setMode hides the editor (`is-hidden` on the textarea)**
- **Found during:** Task 2 (making the plan-gated tests green).
- **Issue:** The plan prose for `setMode` describes toggling `.is-preview` on the bar and calling `openPreview`/`closePreview`, but does not spell out hiding the editor. The two wave-1 RED tests this plan MUST turn green (`46.1-switcher-current-state.test.js` line 107, `46.1-format-in-preview.test.js` lines 78/85) assert `taP.classList.contains('is-hidden')` while previewing and its removal on return to edit. The test is the contract.
- **Fix:** `setMode` toggles `is-hidden` on the resolved textarea (`preview` → add, `edit` → remove). This is deliberately placed in `setMode`, NOT inside `openPreview`/`closePreview`, so those functions stay byte-intact for plan 03 to rewire into the full in-place swap (frame `-body` element, sizing, Ctrl/Cmd+E). `46.1-inplace-swap.test.js` therefore stays RED as intended (its `.rich-toolbar-preview-body strong` and Ctrl/Cmd+E cases are unmet here).
