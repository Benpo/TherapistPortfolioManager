  <files>scripts/docs-gate.js</files>
  <action>
    Create `scripts/docs-gate.js` (a plain-node CLI, `#!/usr/bin/env node` optional). Parse `--range
    <A..B>` from argv. Resolve the TARGET repo root via `git rev-parse --show-toplevel` (run in cwd) and
    derive `<root>/assets` — pass that assets dir to help-loader / invariants so the gate reads the repo
    it is invoked in (hook, CI, or the throwaway test repo), NEVER its own install directory.

    STEP 1 — invariants first (D-17): run all four `scripts/lib/invariants.js` checks. Fail closed (exit
    non-zero, loud banner) on ANY breach, before touching the range.

    STEP 2 — push-range rule: `git diff --name-only <range>` → changed paths. Classify each via
    role-table.js. Read the trailers with `%(trailers:…,valueonly,only)`.

      TRAILER PARSING (D-14 rules, Claude's Discretion; two orchestrator decisions with Ben, 2026-07-10):

      - **A single `*-Unaffected:` trailer MAY name MULTIPLE files**, comma-separated, sharing one reason:
          `Help-Unaffected: assets/i18n-en.js, assets/i18n-he.js, assets/app.css — tour strings only, no help surface`
        Parse: split the value on the em-dash/`--`/`—` separator into <paths> and <reason>; split <paths>
        on commas; trim. Reason MUST be non-empty. Brace-expansion (`i18n-{en,he}.js`) is NOT supported —
        write the paths out; the gate does no globbing. A trailer naming a path that did not change in the
        range is a WARNING (printed, not blocking) — it means the declaration is stale.
        *Why:* replaying Phases 41 and 42 against this gate showed 8 and 12 required trailers respectively.
        The recurring core (`app.js`, `i18n-{en,he,de,cs}.js`, `app.css`, `tokens.css`, `shared-chrome.js`)
        is touched nearly every phase and can never earn a help topic. Twelve one-file lines per push is
        boilerplate, and boilerplate is not a forcing function — it is the gate Ben would disable
        (CONTEXT §Specifics, the noise principle). The decision is still forced and still recorded: every
        uncovered file must be named, with a reason. Only the ceremony shrinks.

      - `Help-Unaffected:` and `Changelog-Unaffected:` are honored from **ANY commit in the range**. They are
        file-scoped, so a trailer inherited via a merge can only ever excuse the file it names — harmless.

      - **EMERGENCY — `Docs-Emergency-Skip:` is honored ONLY when it appears on the TIP COMMIT of the range**
        (`git log -1 --format='%(trailers:key=Docs-Emergency-Skip,valueonly,only)' <range-tip>`), never from
        an arbitrary commit in the range. If present on the tip: print a LOUD banner naming the commit, the
        reason, and every skipped watched file, then PASS.
        *Why the tip restriction:* this repo merges GSD executor worktree branches into `main`, and git reads
        trailers from every commit reachable in a range — including those pulled in by a merge. Without this
        restriction, a genuine 2am hotfix carrying an emergency skip would, when its branch was later merged
        alongside ordinary work, silently excuse that ordinary work too. Requiring the tip commit means an
        emergency skip can only ever excuse the push it was written for. It costs a real emergency nothing:
        you are authoring that commit anyway.
        A `Docs-Emergency-Skip:` found on a NON-tip commit in the range must be **ignored, and reported** in
        the output ("ignored an inherited Docs-Emergency-Skip from <sha>; emergency skips are honored only
        on the tip commit") so the behavior is never mysterious.
      - CHANGELOG (D-08): if any watched trigger changed and neither `assets/changelog-content-en.js` was
        edited in the range NOR a `Changelog-Unaffected:` trailer covers it → BLOCK naming the file(s).
      - RELEASE MOMENT (D-08/GATE-04): if APP_VERSION changed in the range, require a changelog entry for
        exactly that version with non-empty highlights + a date; origin:true tolerated. Missing → BLOCK.
      - HELP (D-11/D-12): for each changed watched trigger, look up its claiming topic(s) via the EN
        covers[] reverse index. If claimed but no claiming topic was edited in the range AND no
        `Help-Unaffected:` trailer covers it → BLOCK, naming the file and each claiming topic id+title,
        then the escape line (D-13 message shape). If UNCOVERED (no topic claims it) and no
        `Help-Unaffected:` trailer → BLOCK with the "add it to a covers[] or declare Help-Unaffected"
        guidance (D-12 anti-rot).
    Denylisted and ignored paths raise no demand; satisfier files never trigger (D-07). Fail closed on any
    throw/parse error (D-04). All failure paths print a human-readable BLOCKED banner then `process.exit(1)`;
    full pass prints a short OK line and exits 0. Behavior-only comments; no phase/decision-ID citations.
  </action>
