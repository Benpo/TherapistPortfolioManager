# Phase 25 — UI Design Contract

> Visual and interaction contract for the backup architectural rework. Generated by gsd-ui-researcher (autonomous run, user asleep) on 2026-05-15. All decisions traced back to 25-CONTEXT.md (D-01..D-30), 25-RESEARCH.md, existing tokens.css/app.css, Phase 24 conventions, and Phase 21 confirm-card pattern. Verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla HTML/CSS/JS PWA — no React/Vite/Next, so shadcn gate is N/A) |
| Preset | not applicable |
| Component library | none — composition of existing app.css primitives (`.button`, `.button.secondary`, `.button.ghost`, `.button.danger`, `.modal`, `.modal-card`, `.confirm-card`, `.settings-tabs`, `.input`, `.textarea`, `.label`, `.helper-text`, `.section-helper`, `.form-field`, `.toast`, `.success-text`) |

**Pre-populated source:** RESEARCH.md "Standard Stack" table; `tokens.css` font-face declarations; `app.css` button/modal/tab patterns; `settings.html:55-58` existing tab pattern.

**Why no shadcn:** This is a vanilla-JS PWA with hand-rolled tokens (`assets/tokens.css`) and a stable component set in `assets/app.css`. Phase 25 is an integration/composition exercise on those primitives (per RESEARCH.md Summary), not a green-field build. Adding shadcn would be net-negative (introduces React-only dependency, breaks the offline-first PWA constraint, contradicts CONTEXT D-30 single-source-of-truth on the existing modal helpers).

## Spacing Scale

The project's existing spacing tokens (`--space-xs`, `--space-sm`, `--space-md`, `--space-lg`, `--space-xl`) are already on a 4/8 scale across `app.css`. Phase 25 inherits them.

Declared values (must be multiples of 4):

| Token | Value | Usage in Phase 25 |
|-------|-------|-------------------|
| xs (`--space-xs`) | 4px | Icon ↔ label gap inside the backup-contents checklist; chip inner padding |
| sm (`--space-sm`) | 8px | Inline button gap inside Export-section button row; form-field spacing inside the Test Password sub-section |
| md (`--space-md`) | 16px | Default modal section padding; gap between Backup Contents block and Export block; settings tab content padding |
| lg (`--space-lg`) | 24px | Vertical gap between major modal sections (Header → Contents → Export → Import → Test Password); Settings card outer padding |
| xl (`--space-xl`) | 32px | Modal outer padding (`modal-card` reuse); top spacing above destructive Delete-all-photos action in Photos tab |
| 2xl | 48px | Reserved for major page-level breaks (not used in Phase 25 — modal stays compact) |
| 3xl | 64px | Reserved for landing-page-style hero spacing (not used in Phase 25) |

Exceptions for this phase:
- **Icon-button touch target = 44px** for the optional inline backup chip's "i" info icon and the modal close button (`.modal-close` is already 36px in app.css:1548 — keep current; mobile touch is already validated in Phase 21 MOB-04 audit).
- **Chip vertical height = 28px** with `padding: 4px 12px` (a known UI-pattern outlier — fits beside the primary "Backup & Restore" button without dominating the action row, matches the existing `.snippets-tag-chip` height in app.css:3270).

## Typography

The project uses Rubik exclusively at three weights (400 regular, 600 semibold, 700 bold; 800 black is also loaded but is brand-mark-only). Phase 25 uses **two weights** for the new surfaces: 400 body / 700 emphasis. Heading sizes deliberately collapse to a small set so the modal doesn't drift from existing modal type ramps (`.modal-title` 1.4rem, `.confirm-title` 1.35rem, `.section-title` 1.4rem).

| Role | Size | Weight | Line Height | Phase 25 Usage |
|------|------|--------|-------------|----------------|
| Body | 16px (1rem) | 400 | 1.5 | Modal section descriptions ("Choose where to save your backup…"); Settings field helper text; Photos storage usage paragraph |
| Label | 14px (0.875rem) | 700 | 1.35 | `.label` for form inputs; chip text "Last backup · 3 days ago"; backup contents checklist items ("Clients", "Sessions", …) |
| Heading | 22px (1.4rem) | 700 | 1.2 | Modal section headings ("Export backup", "Import backup", "Test backup password"); Settings tab headings ("Scheduled backups", "Photos") |
| Display | 28px (1.75rem) | 700 | 1.2 | Modal title bar ("Backup & Restore") — slightly larger than `.modal-title` 1.4rem to give the unified surface page-replacement gravitas (per D-26: "every new surface MUST have clear headers") |

**Pre-populated source:** existing `.modal-title` (app.css:1595), `.confirm-title` (app.css:1519), `.section-title` (app.css:206), `.label` (app.css:986), body declaration (app.css:23-29).

**Hebrew/RTL note:** Rubik covers Hebrew. The inherited `html[dir=rtl]` pattern (app.css:31-33) auto-flips the layout. All sizes/weights are dir-agnostic. Per D-27, Hebrew strings use noun/infinitive forms only (e.g., "שיתוף גיבוי" not "שתף גיבוי").

### Schedule password — not persisted (resolves RESEARCH.md Open Question 1 / Assumption A1)

Per the autonomous-mode default cited in the prompt: schedule password is **re-prompted on each fire**, **not persisted**. The Settings → Backups tab tracks only "user has acknowledged the password requirement" via a single boolean `localStorage.portfolioBackupSchedulePasswordAcked`. This default matches the no-recovery encryption philosophy and avoids the Pitfall 2 disaster (forgotten password locks all scheduled backups). The Test-password feature (D-12) is the user's safety net.
