<objective>
Close verifier blocker **CR-01** from `25-VERIFICATION.md`: `BackupManager.checkBackupSchedule` currently writes `localStorage.setItem('portfolioBackupSchedulePromptedAt', now)` at `backup.js:1324` BEFORE checking whether `window.openBackupModal` exists at line 1332. On every page that loads `app.js` but not `overview.js` (settings.html, add-client.html, add-session.html), the 1-hour debounce is consumed silently, suppressing the next legitimate prompt on the overview page.

Purpose: Restore D-17 reliability — the schedule interval-end prompt must reach the user reliably regardless of which page they land on when the interval lapses.

Output: A guarded debounce write that only advances when the modal was either opened in-place or the page redirected to `index.html?openBackup=1`. Two new behavior tests prove the gate works both ways (modal available → stamp advances; modal absent + non-overview page → either redirects + stamp advances, OR returns without writing stamp).

**Note on scope:** UAT-D2 ("Backup icon from inside Settings navigates to overview first") is the user-facing manifestation of the same root cause. The cleanest fix is the redirect path (`window.location.href = './index.html?openBackup=1'`) — same approach the cloud-icon click handler already uses in `assets/app.js:475-481`. UAT-D2's deeper "modal opens in-place on every page" rework is OUT OF SCOPE for this gap-closure plan because mounting the modal markup + handler cross-page is itself a Phase 25 architectural change that would touch every HTML page; it belongs in a follow-up phase if Ben still wants in-place opening after seeing the redirect behavior. The redirect closes both CR-01 and UAT-D2's bug (the prompt is never silently lost), even if it doesn't deliver the "no navigation" ideal.
</objective>
