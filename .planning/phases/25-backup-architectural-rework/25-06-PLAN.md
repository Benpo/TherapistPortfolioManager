<objective>
Resize photos on upload to max 800px on long edge at JPEG quality 0.75, store ONLY the cropped final output (never the original), and use createImageBitmap with imageOrientation+resizeWidth/resizeHeight to honor EXIF and avoid iPhone OOM on multi-MB camera photos.

Purpose: D-21/D-22/D-23 reduce backup payload size dramatically (from ~MB-per-client to ~80-120KB-per-client) and make the crop output the canonical photo storage. Side-benefit: canvas re-encode strips EXIF metadata including GPS, mitigating threat T6 from the planning context.

Output: A new CropModule.resizeToMaxDimension helper, the add-client upload pipeline rewired to resize-then-crop, the existing crop output quality bumped down from 0.85 to 0.75, a soft 25MB safety threshold for the iPhone OOM Pitfall, 2 new i18n keys × 4 locales, and TWO behavior tests — one for the D-21 800px ceiling (capturing createImageBitmap opts) and one for the D-22 crop-only persistence (driving the full upload pipeline and asserting the IDB write only sees the cropped sentinel).

Note on testing: per RESEARCH "Wave 0 Gaps", node `<canvas>` is non-trivial. We CANNOT decode a real JPEG in Node, but we CAN behaviorally prove the 800px ceiling by stubbing `createImageBitmap` to record the `opts` argument and asserting `resizeWidth`/`resizeHeight` are set proportionally to a 800px longest edge. We CAN also behaviorally prove crop-only storage by stubbing the entire CropModule (resize → returns sentinel; openCropModal → invokes onSave callback with sentinel) and spying on PortfolioDB.addClient/updateClient to inspect the photoData argument value AND the absence of the raw File reference at any path.     async function resizeToMaxDimension(blob, maxEdge, quality) {
      maxEdge = (typeof maxEdge === 'number' && maxEdge > 0) ? maxEdge : 800;
      quality = (typeof quality === 'number' && quality > 0 && quality <= 1) ? quality : 0.75;

      // First-pass decode to read dimensions (cheap when imageOrientation is 'from-image')
      // Note: passing resizeWidth/Height in the SAME call requires knowing target dims;
      // the canonical pattern is two passes: probe → decide → re-decode with resize options.
      // For simplicity (and correctness on Safari), do it in two steps via createImageBitmap.
      var probeBitmap = await createImageBitmap(blob, { imageOrientation: 'from-image' });
      var srcW = probeBitmap.width;
      var srcH = probeBitmap.height;
      probeBitmap.close();

      var longEdge = Math.max(srcW, srcH);
      var scale = longEdge > maxEdge ? maxEdge / longEdge : 1;
      var w = Math.round(srcW * scale);
      var h = Math.round(srcH * scale);

      // Second-pass decode WITH resize hints (Pitfall 3 mitigation: lower peak memory on iOS)
      var decodeOpts = { imageOrientation: 'from-image' };
      if (scale < 1) {
        decodeOpts.resizeWidth = w;
        decodeOpts.resizeHeight = h;
        decodeOpts.resizeQuality = 'high';
      }
      var bitmap = await createImageBitmap(blob, decodeOpts);

      var canvas = document.createElement('canvas');
      canvas.width = w;
      canvas.height = h;
      var ctx = canvas.getContext('2d');
