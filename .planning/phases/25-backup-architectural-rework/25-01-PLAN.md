<threat_model>
## Trust Boundaries

| Boundary                                 | Description                                                                          |
|------------------------------------------|--------------------------------------------------------------------------------------|
| App in-memory blob → OS share sheet      | navigator.share hands the file to whatever destination the user picks (Mail, Drive, AirDrop) |
| App in-memory blob → mailto: handler     | OS opens the user's mail client with subject + body; the file lives separately in Downloads |
| User passphrase choice (encrypt vs skip) → blob produced by exportEncryptedBackup vs exportBackup → shareBackup | The blob handed to shareBackup must reflect the user's encryption decision; no implicit re-encoding |

## STRIDE Threat Register

| Threat ID  | Category               | Component                                  | Disposition | Mitigation Plan                                                                                                                                                                                                                                                  |
|------------|------------------------|--------------------------------------------|-------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| T-25-01-01 | T1 Encryption bypass / Information Disclosure | shareBackup invocation path                | mitigate    | shareBackup signature accepts (blob, filename) — it CANNOT call exportBackup() or exportEncryptedBackup() internally. tests/25-01-share-encryption-inherit.test.js asserts neither is called. The existing sendToMyself path that bypassed encryption is DELETED (T1 closure). |
| T-25-01-02 | T2 Web Share leakage   | navigator.share files payload              | mitigate    | The File constructed inside shareBackup wraps the EXACT blob passed in by the caller (no re-read, no re-encode). The MIME type is taken from blob.type (already 'application/octet-stream' for .sgbackup, set in _encryptBlob at line 67). Test 25-01-share-encryption-inherit asserts files[0].name matches the input filename. |
| T-25-01-03 | R Repudiation / user-trust | mailto fallback body                       | mitigate    | The fallback body uses the literal `_t('backup.share.fallback.body')` which contains "Please attach {filename} to this email manually." — no claim of attachment. EN/HE/DE/CS strings all use the same honest phrasing (verified by Task 2 acceptance criteria). |
| T-25-01-04 | I Information Disclosure (Pitfall 1) | Safari macOS rejecting `.sgbackup` after canShare lies | accept      | Wrapped in try/catch — if share() throws non-AbortError, code falls through to mailto fallback. User sees download-and-attach flow; no failure mode silently leaks the blob. Probe via isShareSupported(file) further reduces but does not eliminate the false-positive window. |

This plan mitigates threats T1, T2 (entirely) and T3 (the repudiation surface from CONTEXT). T4-T6 are addressed by Plans 03, 05, 06.
</threat_model>
