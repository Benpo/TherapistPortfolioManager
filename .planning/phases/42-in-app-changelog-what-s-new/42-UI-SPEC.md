# Phase 42 — UI Design Contract

> Visual and interaction contract for the In-App Changelog & What's-New system.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** This phase adds TWO new surfaces to the existing garden PWA — (1) a
> standalone **changelog page** (`changelog.html` + `assets/changelog.js` + per-locale
> `assets/changelog-content-en.js`) in the help family, and (2) a modest **What's-New
> teaser modal** governed by the Phase 40 attention coordinator's reserved `'whats-new'`
> slot. It also adds one `data-i18n` chrome slot to the "?" popover (`app.js` `initHelpEntry`)
> and — at Claude's discretion (resolved YES below) — makes the footer version line a quiet
> link to the changelog page.
>
> **This is a REUSE-HEAVY phase.** Every value below is *inherited* from the existing garden
> design system (`assets/tokens.css`), the approved **sketch 005 Variant B**
> (`.planning/sketches/005-changelog-register/index.html` — structure locked by D-11, copy is
> placeholder), the `26-UI-SPEC.md` contract, and the live surfaces this phase plugs into. It
> introduces **no new palette, scale, typography, component library, or npm dependency.** Where
> a checker greenfield heuristic (2-weight max / strict-4px grid / generic-CTA) appears to fire,
> see **Accepted Exceptions** — these are documented reuse decisions, not violations.

## Spacing Scale

Authored in `rem` (1rem = 16px), the existing garden rhythm. The changelog page and popup reuse
the `26-UI-SPEC.md` step set; the values below are the **actual** paddings/margins the approved
sketch renders, mapped to the nearest scale step.

| Token | rem | px | Usage in this phase |
|-------|-----|-----|---------------------|
| xs | 0.25rem | 4px | Category-heading-to-list gap; entry-date inline gap |
| sm | 0.5rem | 8px | Bullet list-item spacing; popup action-button gap |
| md | 1rem | 16px | Entry lede-to-category spacing; popup inner rhythm |
| lg | 1.5rem | 24px | Entry-card padding (`1.5rem 1.75rem`); inter-entry gap; popup body padding |
| xl | 2rem | 32px | Page-head-to-first-entry gap (`2.25rem` ≈ xl); page top padding |
| 2xl | 3rem | 48px | Page bottom safe padding on large viewports |

**Exceptions (accepted — reuse of live values, not greenfield):**
- Entry card horizontal padding is `1.75rem` (28px) and `border-radius: 14px`; popup
  `border-radius: 16px`, `max-inline-size: 420px`. These are the **exact approved sketch
  values** (D-11) ported verbatim, not new grid steps. Off-4px-grid radii are inherited garden
  house-style (help/welcome cards use the same organic radii) — do NOT "correct" them.
- The 44×44px minimum tap target on the "?" popover row and popup buttons is an **accessibility**
  constraint (inherited global rule, `app.css`), not a rhythm token.

## Typography

The changelog page and popup reuse the app's live Rubik type system and the approved sketch's
role sizes. Roles below are drawn directly from `sketch 005 Variant B`.

| Role | Size | Weight | Line Height | Used for |
|------|------|--------|-------------|----------|
| Page title (Display) | 1.7rem (≈27px) | 400 | 1.2 | Changelog page `<h1>` "What's new in Sessions Garden" — calm, regular weight (garden house-style for page heads) |
| Entry heading | 1.25rem (20px) | 500 | 1.3 | Per-version `<h2>` "Version 1.3"; popup `<h2>` headline (1.2rem) |
| Body | 0.95rem (≈15px) | 400 | 1.55 | Entry ledes, bullet text, page intro, popup highlights (1.5 in popup) |
| Category label | 0.8rem (≈13px) | 500 | 1.3 | New / Improved / Fixed block headings — `text-transform: uppercase`, `letter-spacing: 0.08em`, colored |

**Rules:**
- Weights used: **400 (regular), 500 (medium), 700 (bold, for the popup primary CTA / links)**.
  This mirrors the live app (`--weight-regular` / `--weight-bold` tokens + Rubik 400;500;700) and
  the approved sketch exactly.
- Body line-height 1.55 for the entry/bullet reading rhythm (sketch value); popup body 1.5.
- Entry / category headings authored as **nouns**, not imperatives (Pitfall 5; Phase 24 D-05) —
  "New" / "Improved" / "Fixed" and "Version 1.3", never imperative phrasing — so the Phase 42.1
  HE/DE/CS translation produces natural noun forms.
- **No emojis** anywhere in-app copy (D-10; upholds the `26-UI-SPEC.md` no-emoji contract).

**Accepted Exception (checker 2-weight rule):** this contract declares **three** weights
(400/500/700). The `26-UI-SPEC.md` "two weights only" rule was **Phase-26-surface-scoped**; the
production app's real type system uses 400/500/600/700/800 and the approved Phase 42 sketch uses
400/500/700. Constraining the changelog page to two weights would make it inconsistent with the
`help.html` shell it mirrors. Grep-verified against live `app.css` (2026-07-09). This is a
reuse-consistency decision — do not spin the checker loop on it (see memory
`feedback-ui-checker-greenfield-false-positives.md`).

## Color

Inherited entirely from `assets/tokens.css` semantic tokens (auto-resolving in
`[data-theme="dark"]`). The table maps 60/30/10 onto the garden palette; the implementation MUST
use the listed vars, never literal hex. All tokens verified present in production `tokens.css`.

| Role | Token (use this var) | Usage in this phase |
|------|----------------------|---------------------|
| Dominant (60%) | `--color-background` | Changelog page background; overlay backdrop uses `--color-modal-overlay` |
| Secondary (30%) | `--color-surface` | Entry cards, popup surface, sketch-bar-equivalent chrome |
| Accent (10%) | `--color-primary` | **Reserved list below — not "all interactive elements"** |
| Destructive | `--color-danger` | **Not used** — this phase has no destructive actions |

**Accent (`--color-primary`) reserved EXCLUSIVELY for:**
1. Popup **primary CTA** "See everything new" — filled primary button (`.btn-primary`).
2. The **active "?" popover / nav state** where the changelog entry-point inherits the existing
   `.is-active` treatment (unchanged from Phase 39/40/41).
3. **Focus-visible outline** `2px solid var(--color-primary)`, `outline-offset: 2px` — on every
   interactive element in both new surfaces (existing app pattern).

**Category-heading semantic colors (data-level register, NOT the 10% accent):** the New /
Improved / Fixed headings use their own low-saturation semantic tints, exactly as the approved
sketch renders — these are a *categorization signal*, deliberately quieter than the primary
accent, and each has a dark-mode override:

| Category | Light token | Dark token |
|----------|-------------|------------|
| New | `--color-green-600` | `--color-green-400` |
| Improved | `--color-green-500` | `--color-green-200` |
| Fixed | `--color-orange-500` | `--color-orange-500` (sketch keeps orange in dark) |

Everything else — page headings (`--color-primary-deeper` light / `--color-green-400` dark),
entry ledes, bullets, dates, the quiet **Close** button (`.btn-quiet`, neutral), the footer
version link — uses neutral / surface / text tokens (`--color-text`, `--color-text-muted`,
`--color-green-border-subtle`, `--color-green-border-faint`). The Close path is visually
first-class but **not** accent-colored (calm, non-forced tone).

**Independence guard (critical):** the changelog color surfaces MUST NOT reuse or visually
collide with the footer integrity **⚠ marker** (`--color-danger` territory, owned by
`version.js` / `shared-chrome.js`). The two are separate signals — no double-signalling (ROADMAP
§Phase 42 dependency note; CONTEXT Claude's-Discretion footer guard).

## Copywriting Contract

Tone target: the calm, warm, non-technical garden register (`26-UI-SPEC.md` copywriting
contract) — NOT generic SaaS release-notes energy. **All copy below is STRUCTURE-level guidance
and UI-chrome only.** The actual per-version entry BODY copy (backfill v1.0→v1.3) is
placeholder in the sketch and MUST pass the Phase 39 D-19 wording pipeline (factual → native-
speaker → DNA/voice gate) **and Ben's explicit approval** (D-04) before ship — plan an explicit
copy-approval checkpoint like the Phase 40 welcome-copy and Phase 41 tour-copy gates, covering
the backfill AND the v1.3 entry.

UI-chrome strings ship as `data-i18n` keys in all 4 locales THIS phase (D-17). Suggested key
namespace: `changelog.*` (page) and `whatsNew.*` (popup).

| Element | Copy (EN canonical — Ben approves final) | i18n key |
|---------|------------------------------------------|----------|
| Page title | "What's new in Sessions Garden" | `changelog.page.title` |
| Page intro (privacy-reassurance register — D-specifics) | "Every release, in plain language — newest first. Everything always stays on your device." | `changelog.page.intro` |
| Category heading — New | "New" | `changelog.cat.new` |
| Category heading — Improved | "Improved" | `changelog.cat.improved` |
| Category heading — Fixed | "Fixed" | `changelog.cat.fixed` |
| v1.0 origin one-liner (warm marker, no bullets — D-01) | "Where it all began — the first seed of Sessions Garden." | (entry data, `changelog-content-en.js`) |
| Popup headline | "What's new in version {X.Y}" | `whatsNew.title` |
| Popup subline | "A quick look at what's waiting for you." | `whatsNew.sub` |
| **Popup primary CTA** | "See everything new" | `whatsNew.seeAll` |
| Popup quiet dismiss | "Close" | `whatsNew.close` |
| "?" popover row (opens the PAGE, D-14) | "What's new" | `whatsNew.menuRow` (or `help.entry.whatsNew`) |
| Empty state | **N/A by design (D-07):** the popup NEVER shows for a version without an entry (silent skip; last-seen still updates). The page always has ≥1 entry (v1.0 origin line) so it has no empty state. |
| Error state | **N/A:** fully offline, static local data (in `PRECACHE_URLS`); no fetch, no async failure surface. If `APP_VERSION` has no matching entry the popup silently no-ops (D-07) — this is a designed skip, not an error. |
| Destructive actions | **NONE.** No data is created, deleted, or mutated. `--color-danger` intentionally unused. |

**Microcopy rules:**
- No emojis in-app (D-10). Ben's emoji-flavored register stays for WhatsApp/marketing exports.
- Category headings are the fixed nouns New / Improved / Fixed; **empty categories are omitted**
  entirely (D-11), never rendered empty.
- Per-version lede: one benefit-led sentence before the category blocks (part of the winning
  Variant B shape — D-11, specifics).
- The v1.0 entry is a single warm origin line with **no** category blocks (D-01).

**Accepted Exception (checker generic-CTA rule):** "See everything new" and "Close" are the
approved sketch's deliberate calm labels (D-05). "Close" is intentionally quiet/neutral, not a
generic-CTA violation — it mirrors the welcome-overlay dismiss vocabulary.

## Interaction Contract (phase-specific — consumed by checker, planner, executor)

Hard constraints the executor must honor. Anchored to real source, cited inline.

### What's-New popup (modest centered modal — D-05/D-06)

| Aspect | Contract |
|--------|----------|
| Weight | **Modest centered modal**, NOT full-screen. Full-screen stays unique to the welcome overlay (its "highest emotional impact" status is preserved — D-06). DOM: `.overlay > .popup[role="dialog"][aria-modal="true"][aria-labelledby]`, ported from sketch 005. |
| Content | Headline "What's new in version {X.Y}" + subline + the release's hand-picked top **2–4 highlights** (the entry's explicit `highlights` field, D-08) + primary "See everything new" + quiet "Close" (D-05). The popup is an **announcement, not a reader** — the full story lives on the page. |
| Focal point | The **hand-picked highlights list + the primary "See everything new" CTA** are the modal's single focal point. Headline/subline set context; the highlights carry the payload; the filled `.btn-primary` is the strongest call and holds the only 10% accent (Color §). The quiet "Close" is deliberately subordinate. |
| Gating | Registered via **one** `AttentionCoordinator.register({ id:'whats-new', eligible(), show() })` call into the pre-reserved `PRECEDENCE` slot #2 (verified `attention-coordinator.js:47`). `eligible()` ≈ `APP_VERSION !== lsGet('sg.whatsNewLastSeenVersion') && entryExistsFor(APP_VERSION)`. First-run suppression falls out of the welcome's subsume-write (P40 D-03). |
| Silent skip (D-07) | If no entry exists for the running `APP_VERSION`, the popup does NOT show and the last-seen version still updates quietly. Users never see a contentless announcement. |
| Version source | Reads **only** `window.AppVersion.APP_VERSION` (`version.js`, currently `1.3.0`). MUST NOT read or touch the SW/integrity layer (`checkIntegrity`, `buildNudge`) — independent surface (ROADMAP §Phase 42). |
| Dismiss (D-09 — deliberate only) | **Backdrop / outside clicks do NOTHING.** Only **Close**, **"See everything new"**, or **Esc** dismiss — and EACH records `APP_VERSION` into `sg.whatsNewLastSeenVersion` (via the coordinator's `lsSet`). Rationale: one-shot announcement must not be eaten by an accidental click (Ben hit exactly this in the sketch). |
| A11y (Claude's discretion → resolved: follow welcome-overlay idiom, `attention-coordinator.js:120-263`) | `aria-modal="true"` + `aria-labelledby` the headline; **scroll lock** on open; **focus trap** cycling between the two actions (Close ↔ See-everything-new) — Tab never escapes to the inert background; move focus to the primary CTA on mount; **restore focus to the opener** on dismiss; **Esc** dismisses. Optional: a polite SR announcement, matching the welcome idiom. Buttons meet the 44×44 tap-target minimum. |
| Multi-version jump | Popup shows the **latest** entry only (CHLG-03); the page carries everything missed. `entryExistsFor(APP_VERSION)` keys on the running version's entry. |
| Re-trigger | The popup is **never** re-triggerable from the "?" menu (D-14) — announcements are one-time. The menu row opens the PAGE instead. |
| Demo mode | Impossible in demo (coordinator fully disabled there — P40 D-09); no extra guard needed. |
| RTL / dark | Logical properties only; `[data-theme="dark"]` popup tints per sketch. |

### Changelog page (standalone, help family — D-13)

| Aspect | Contract |
|--------|----------|
| Shell | New `changelog.html` mirroring the `help.html` per-page pattern: shared `<head>` script/stylesheet chain (`tokens.css` → `app.css` → `changelog.css`), `SharedChrome.renderFooter()` + nav, `data-i18n` chrome, `assets/changelog.js` loader. NOT a section inside `help.html` (D-13 — help.html stays lean). |
| Focal point | The **most recent version's entry card** (top of the reverse-chronological list) is the page focal point — it renders first, carries the freshest lede + category blocks, and is the anchor the popup deep-links to (`#{latest-entry-anchor}`). Older entries recede down the page in the same calm rhythm; no single entry is accent-highlighted (recency = position, not color). |
| Data | Per-locale files, EN canonical: `assets/changelog-content-en.js` (this phase), mirroring the `help-content-en.js` global-registration + documented-schema precedent (D-17). Phase 42.1 adds `-he/-de/-cs` siblings + a parity integrity test (mirrors `tests/39-help-integrity.test.js`). |
| Entry schema (Variant B — D-11) | Reverse-chronological array of `{ version, date, lede, highlights[], categories: { new?[], improved?[], fixed?[] } }`. Render: per-version `<h2>` (version + release date) → one-sentence lede → New / Improved / Fixed blocks with small colored uppercase category headings; **empty categories omitted**. `highlights[]` is the hand-picked popup source (D-08) + a Phase-43-checkable hook. v1.0 renders as a single one-line origin marker (no categories — D-01). Exact field names beyond these are Claude's discretion. |
| EN fallback (D-16) | An entry (or part) not yet translated renders in English inside an otherwise-localized page — history is always complete in every locale. Matches the help center's EN-fallback loader plan (42.1). |
| Entry anchors | Each version entry gets a stable kebab `id` (e.g. `id="v1-3"`, major-minor) — the deep-link target. |
| Offline | `changelog.html` + `changelog.js` + `changelog.css` + `changelog-content-en.js` MUST join `sw.js` `PRECACHE_URLS` (CHLG-01 offline requirement). CACHE_NAME auto-rolls via the deploy-stamped INTEGRITY_TOKEN — no manual bump (verified 2026-07-08). |
| Demo mode (D-15) | The page is **not reachable** from the demo iframe: the "?" What's-new row hides (below) and the discretionary footer link is suppressed in demo. |
| RTL / dark | Logical properties only, `[data-theme="dark"]` reads dark tokens, no physical `left/right`, no literal hex (validated per project convention). |
