<objective>
Round-3 gap fix for Phase 46. Both gaps were reproduced live in Playwright WebKit on build e0c48c5 and diagnosed with confirmed root causes in 46-UAT.md (gaps 12 and 13). This plan fixes BOTH with scoped JS changes to `assets/rich-toolbar.js` only, proven RED→GREEN by a new jsdom unit test (Gap 12) and a new WebKit probe assertion set (Gap 13 + Gap 12 end-to-end).

**Gap 12 — persistent-bar controls are DEAD until the editor is focused once (high).** `_dispatch`/`togglePreview` resolve their target field from `_focused` (the last-focused registered field) and return early when it is null. That "visible bar ⇒ focused field" invariant held for the focus-attached shared bar but was broken by the round-1 `persistent: true` mount: the export bar is ALWAYS visible, so users click its controls BEFORE ever focusing the editor (default page focus sits on `#sessionDate`). Every control on the persistent bar (bold, lists, undo, preview…) silently no-ops on first click; the mousedown+preventDefault focus-preservation also means the click itself never focuses the editor. Fix: dispatch on a persistent bar resolves the target from the clicked control's OWN bar (the `_persistentBars` map already knows field↔bar). For formatting/undo/redo/heading it focuses that field as part of the action (placing a deterministic cold caret at end-of-document when the field was never focused) AND sets the module focus state directly (dockTo/`_focused`, not relying solely on the focusin side-effect). For **preview** it resolves the field WITHOUT focusing (preview is view-only; focusing would pop the iOS soft keyboard over the just-revealed pane — `updatePreviewButton`/`barFor` already fall back to `_previewField`, so button state stays correct without focus). So the persistent bar never depends on prior focus. Note-field shared-bar behaviour unchanged.

**Gap 13 — export preview pane opens 100% below the scroll fold (high).** With the editor focused, Preview DOES open, but the pane is inserted `afterend` the editor inside the fixed-height `.export-edit-area` (overflow-y:auto), and the editor (flex-fill, min-block-size 240px) already fills/overflows the visible area — measured pane top 659.6px vs area visible bottom 652.0px at 1440x820: the pane starts entirely below the fold. The user sees no change. On note fields the page grows so the pane is naturally visible — the export's scroll container is the only surface with a fold. Fix: when the preview opens for a field whose scroll container is the export edit area, bring the pane into view by scrolling the container so the pane top clears the sticky bar — offset scrollTop by the bar's height (`pane.offsetTop − bar.offsetHeight − gap`), NOT a bare `scrollIntoView` (which aligns to the container top and would land the pane under the sticky bar). Must not disturb note-field preview, the sticky pinned bar, or undo/caret state; RTL-safe.

Purpose: make every persistent-export-toolbar control work on first click and make Preview actually reveal its pane, without changing any of the seven note-field behaviours.
Output: two scoped JS edits to `assets/rich-toolbar.js`, a new RED-first jsdom test, and a new RED-first WebKit probe assertion set.

**Explicitly NOT in this round:**
- No CSS change is expected (the sticky/pinned bar + sizing floor already shipped in 46-15). If a CSS change proves strictly necessary it must be scoped exactly like 46-15's `.export-card.is-editor-step` gating — flag it, do not broaden.
- No i18n keys, no sw.js change, no new shipped asset, APP_VERSION stays 1.4.0.
- The residual ~8px cosmetic gap under the pinned bar (46-15 NOT-list, gate item 12h) is untouched here.
- **Keyboard shortcuts stay focus-dependent.** Ctrl/Cmd+B/I and Ctrl/Cmd+Z/Y still require the field to be focused — they bind on the field's own keydown, not the bar. Only the persistent-bar BUTTONS become focus-independent; the shortcuts are out of scope (a user pressing a shortcut has necessarily focused the field).
- **No reveal-on-Step-2-entry.** The reveal fires only on an explicit preview-open action. A stale-open preview pane inherited from a previous dialog session must NOT trigger a scroll when Step 2 is entered.
- **No explicit scroll-restore on preview close** this round. Closing the preview relies on the container's natural scrollTop clamp to re-reveal the editor; if the device gate finds the editor left disoriented after toggling back to Edit, that routes to a follow-up scroll-restore (46-16 item 13c watches it).
