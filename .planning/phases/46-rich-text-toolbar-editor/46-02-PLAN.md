<objective>
Implement D-13 (RTXT-F2 promoted): TRUE italic glyphs in the exported PDF for Latin AND Hebrew, by vendoring a subset Rubik-Italic face and extending the PDF segment model from bold-only `{text,bold}` to `{text,bold,italic}`. Heebo has no italic face and jsPDF cannot synthesize slant, so a real embedded face is required (RESEARCH §Don't Hand-Roll). Register the italic bytes UNDER family `'Heebo'` style `'italic'` so the renderer only ever calls `setFont('Heebo', style)` — normal/bold stay Heebo, italic transparently gets Rubik glyphs, and the Phase 23 bidi pipeline is untouched.

Feasibility gate (D-13, Ben confirms threshold at plan review): subset Latin+Hebrew FIRST and measure the base64 size. GREEN <= ~90 KB (proceed, mix Heebo+Rubik-italic); AMBER ~90-140 KB (proceed only if Ben accepts the weight at the Task 4 checkpoint); RED > ~140 KB OR Hebrew italic renders visibly broken OR bidi integration risks Phase 23 -> fall back to D-14 (italic flattens as shipped in Phase 45, plus disclosure). The font lazy-loads only during export (never at app boot), so a proportionate payload never touches page load.

This plan is INDEPENDENT of the toolbar (different files) and runs in Wave 1. Purpose: prove "does italic survive export?" is answerable as the toolbar is built, not after.

D-14 contingency (RED only): if the checkpoint rejects true italic, the small disclosure work — appending an "appears upright in PDF export" clause to the italic tooltip (all 4 locales) and a one-line export-modal note — is applied as a follow-up in the export plan (46-06) reading THIS plan's SUMMARY outcome. GREEN is the expected outcome (RESEARCH A1); the disclosure is a genuine contingency, not pre-built.
</objective>
