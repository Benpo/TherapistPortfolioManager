# Phase 46 — UI Design Contract

> Visual and interaction contract for the rich-text toolbar editor. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Reuse phase.** This phase adds an EDITING layer on top of a mature, shipped design system (the "garden" tokens in `assets/tokens.css` + `assets/app.css`) and Phase 45's proven rendering (`.note-rendered`, `window.MdRender`). It introduces net-new *chrome* (a toolbar, a preview pane, an info note, a maximize affordance) but ZERO new colors, ZERO new fonts, and no new base scale — everything reuses existing tokens. See **Accepted Checker Exceptions** at the bottom: the greenfield 2-weight / raw-value rules do not apply here (repo memory `feedback-ui-checker-greenfield-false-positives.md`).

## Design System

| Property | Value |
|----------|-------|
| Tool | none (vanilla-JS, zero-npm, zero-build PWA — custom "garden" design system) |
| Preset | not applicable (no shadcn / React / Vite; IIFE `window.*` modules) |
| Component library | none — new toolbar is a vanilla module following the established `window.*` shape (likely shared between `add-session.js` and `export-modal.js`) |

**Token source of truth:** `assets/tokens.css` (semantic `--color-*`, `--shadow-*`, `--z-*`) and the Phase-45 `.note-rendered` block (`app.css` ~3597). New chrome MUST consume these; it MUST NOT hardcode hex or introduce a new scale.

## Spacing Scale

The app's live scale is 4-based (4 / 8 / 12 / 16 / 24 / 32); 12px is an established member (buttons, radii). All are multiples of 4.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | Gap between adjacent toolbar icon buttons; caret-inset padding |
| sm | 8px | Toolbar inner padding; gap toolbar↔field; gap field↔preview pane; info-note internal padding block |
| md | 12–16px | Info-note padding inline; group separators inside toolbar; preview-pane padding |
| lg | 24px | List indent step in preview (`.note-rendered ul/ol` = `padding-inline-start:24px`, reused) |
| xl | 32px | Step-2 modal internal gutter (maximized layout) |

**Exceptions (accepted):**
- **Touch targets 44×44px** on mobile/touch toolbar buttons — WCAG 2.2 target-size, matches shipped `.header-icon-btn` (44×44). Desktop toolbar buttons may render at 32–36px visual within a ≥40px-tall bar (comfortable pointer hit area). 44 and 36 are multiples of 4; 32 is a multiple of 4.
- Toolbar row is **horizontally scrollable on overflow** on mobile (D-02) — no wrap, no reflow of the field below.

## Typography

Reuse the app's rem-based scale and the Phase-45 subordinate note register. New chrome uses **exactly two weights: 400 (body) + 600 (emphasis / active / note-headings)**. Rendered **bold** inside the preview uses 700 (the existing `<strong>` weight) — that is rendered content, not chrome.

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Editing text (textarea) | 1rem (~16px) | 400 | inherit field | Unchanged — existing note fields |
| Toolbar (buttons) | icon-only, **no text labels** | — | — | Language-independent (D-14); meaning carried by icon + tooltip |
| Toolbar tooltip | 0.85rem | 400 | 1.4 | Native `title` or existing tooltip idiom |
| Preview body | 1rem | 400 | 1.55 | **Reuse `.note-rendered`** verbatim — zero new rendering CSS (D-05, code_context) |
| Preview note-headings (H1/H2/H3) | 17 / 16 / 15px | 600 | 1.35 | **Reuse `.note-rendered h1/h2/h3`** — subordinate register, weight-led, no size jump, no rule/diamond |
| Info note (Step 2) | 0.9rem | 400 (label emphasis 600) | ~1.5 | Ephemeral-edit disclosure (D-03) |
| Heading-selector control | 0.9rem | 400 (selected 600) | 1.4 | Presents H1 / H2 / H3 / Regular text (dropdown vs. segmented — see Component Inventory) |

Body/heading line-height defaults: **1.5–1.55 body, 1.35 headings** (already the shipped `.note-rendered` values — do not deviate).

## Color

60 / 30 / 10 split, all from existing semantic tokens (light values shown; every token has a dark-mode value in `tokens.css`).

| Role | Token / Value | Usage |
|------|---------------|-------|
| Dominant (60%) | `--color-background` (#fdf8f0 warm cream) | Page / form / modal-body background behind the editor |
| Secondary (30%) | `--color-surface` (#ffffff) / `--color-surface-alt` | Toolbar background, preview-pane surface, modal card, heading-selector popover |
| Accent (10%) | `--color-primary` (#2d6a4f garden green) | See reserved-for list below |
| Destructive | `--color-danger` (#ea4b4b) | **Only** the discard-export-edits confirm dialog (`--z-modal-confirm`, already exists) |
| Info | `--color-info-bg` (#cce5ff) / `--color-info-text` (#004085) | The Step-2 ephemeral-edit note (D-03) — the "blue/info background, worth knowing" band Ben described |

**Accent (`--color-primary`) reserved for — explicit list, never "all interactive elements":**
1. **Active/pressed toolbar toggle** — bold/italic/list/heading button when the format is applied at the current caret/selection (the only state that turns green; idle buttons use `--color-text`).
2. **Preview-toggle active state** — the eye/preview icon while the preview pane is open (D-07).
3. **`:focus-visible` outline** — `2px solid var(--color-primary)`, `outline-offset:2px` (reuse `.header-icon-btn:focus-visible`).
4. **The Step-2 modal's existing primary action** (Continue / Export) — unchanged.

Idle toolbar icons, separators, borders, and the info note do NOT use accent green. Borders use `--color-border*`; shadows use `--shadow-*` (e.g. toolbar/preview `--shadow-nav` or a hairline `--color-border`).

**RTL & dark mode:** every token above resolves per `[data-theme="dark"]` and `html[dir=rtl]`. Layout uses **logical properties** (`padding-inline`, `margin-block`, `inset-inline`) EXCEPT positioned overlays: the docked toolbar and any popover anchored by `getBoundingClientRect` MUST position with **physical** `left`/`top` (repo memory `reference-rtl-logical-props-physical-coords` — rect-based `inset-inline` mirrors wrongly in RTL). The toolbar and preview coexist with the snippets popover at `--z-popover: 360` (must sit above the modal, below toast).

## Interaction Contracts

These are the behaviors the checker/executor treat as the visual+interaction source of truth (all require **real-browser / real-device** verification — jsdom is blind to caret/selection/paste/keyboard, repo memory).

1. **Toggle semantics (D-04):** bold/italic button AND Ctrl/Cmd+B/I. On a formatted selection → unwrap the markers. With no selection → insert a marker pair, caret inside. Never emit `****text****`.
2. **Auto-format while typing (RTXT-03):** `- ` / `1. ` at line start begins a list; Enter continues; Enter on an empty **top-level** item exits the list.
3. **Empty-item outdent (D-10):** Enter on an empty **nested** item outdents one level per press (Word/Docs/Notion standard).
4. **Full auto-renumber (D-11):** insert/delete/paste renumbers following ordered-list items so raw text always reads 1..N. Acceptance: delete item 3 → old 4 immediately reads 3; paste at end → takes the next number. Constraint: **preserve caret stability + native undo** — prefer undo-preserving insertion APIs over blind `.value` assignment; the `autoGrow`/snippets input listeners must still observe the resulting `input` event (RTXT-09 danger zone).
5. **Indent/outdent (D-09):** toolbar buttons (touch/a11y baseline) AND Tab/Shift+Tab when the caret is on a list line. On ordinary text, Tab keeps native focus-move — **no keyboard trap**.
6. **Preview (D-05/06/07):** eye/preview toggle in the toolbar; live pane below the field; per-field; state resets on blur (no stickiness).
7. **Shortcuts scope (D-02):** Ctrl/Cmd+B/I on desktop only. On touch the toolbar is the ONLY formatting affordance — no keyboard-accessory bar.
8. **Mobile Step-2 (D-17):** full-screen 100%×100% keyboard-aware takeover.
9. **Undo/redo buttons (D-20, added at mockup sign-off):** toolbar undo/redo mirror native Ctrl/Cmd+Z / Ctrl/Cmd+Shift+Z over the SAME native undo stack — every programmatic edit (toggles, auto-format, renumber) must remain natively undoable (this extends, not replaces, the D-11 constraint). If native-stack triggering proves unreliable in a target browser, the fallback is a module-level undo stack — decided at plan-phase research, not here.

## Accepted Checker Exceptions (reuse phase)

Repo memory `feedback-ui-checker-greenfield-false-positives.md`: the greenfield rules mis-fire on reuse-only phases. For this phase, the following are **intentional and approved** — do not spin the loop:

- **Font weights beyond 2 in the codebase** (the shipped app uses 400/500/600/700/800). This phase's **net-new chrome** is constrained to 400 + 600; rendered `<strong>` uses the existing 700. No new weight is introduced.
- **rem-based type scale + established 12px step** — pre-existing design-system values, all multiples of 4 (12/16/24), reused not invented.
- **No new page-level CTA** — the phase is an in-place editing layer; "generic CTA" checks are N/A (toolbar is icon+tooltip).
- **Two-color-per-band** honored via existing semantic tokens; the info band (`--color-info-*`) is a documented Phase-27-era pattern, not a new color.
