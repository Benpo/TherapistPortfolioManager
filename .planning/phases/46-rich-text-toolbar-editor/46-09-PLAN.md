<objective>
Gap 3 (the NAMED D-20 follow-up). Ben's real-device finding: he "wrote several lines but undo removed all of them." Native `execCommand('undo')` lets the browser coalesce typing into large chunks, so undo step boundaries feel arbitrary and one undo can wipe multiple lines. The 46-08 gate designated this exact failure as the trigger for a pre-described follow-up: replace the native-first undo path (shipped in 46-03) with a SINGLE module-level undo stack in `window.TextEdit` that ALSO intercepts Ctrl/Cmd+Z (per RESEARCH §A4 / Pitfall 3 — never a dual stack).

This is a self-contained swap behind the already-tested `TextEdit` transforms and the existing toolbar chrome: the pure transforms (`toggleWrap`/`insertListMarker`/`applyHeading`/`autoFormatEnter`/`indentLine`/`outdentLine`/`renumberOrderedBlock`) and the rest of the toolbar (button set, focus docking, preview, list mechanics) stay untouched. Only the undo/redo mechanism changes.

Design goal (Ben): one undo step per toolbar action; typing grouped into small human-scale units — a boundary at each line break and at a typing pause — so undo reverts a line or a short burst, never the whole field.

Purpose: make undo "feel real" so Ben trusts the editor for long-form notes.
Output: a module-level undo stack owning undo/redo for all mounted fields, replacing the native-first path, with headless unit coverage of the boundary/coalesce logic.
</objective>

  <name>Task 1: Module-level undo stack in window.TextEdit (single stack, per-field history)</name>
  <files>assets/text-edit.js, tests/undo-stack.test.js</files>
  <behavior>
    - record-then-undo: after recording a boundary snapshot, mutating the value, then undo → the value AND caret/selection return to the recorded pre-edit state.
    - redo re-applies: undo then redo → returns to the post-edit state.
    - branch truncation: a new record after an undo drops the redo tail (no stale redo).
    - typing coalesces: a run of single-character inserts with no newline and no pause collapses into ONE undo step; undo removes the whole burst.
    - line-break boundary: typing across a newline produces SEPARATE undo steps — undoing removes one line's worth, not the whole multi-line block (Ben's exact scenario).
    - pause boundary: a gap longer than the coalesce window since the last keystroke starts a new undo step.
    - input-type switch boundary: switching from inserting to deleting starts a new step.
    - empty/no-op safety: undo with an empty history is a no-op; redo at the tip is a no-op.
  </behavior>
  <action>
Add a single module-level undo mechanism to `window.TextEdit` (extend the existing IIFE). Per-textarea history is held in a WeakMap keyed by the element; the mechanism is ONE stack model (not a native/module dual stack). Expose these methods on `window.TextEdit` (keep the existing surface intact):
  - `undoTrack(textarea)` — begin tracking; seed a baseline snapshot { value, selStart, selEnd } and reset the pointer.
  - `undoUntrack(textarea)` — stop tracking and drop the field's history.
  - `undoRecord(textarea)` — capture the CURRENT state as a boundary snapshot and truncate any redo tail. Called by the toolbar immediately BEFORE each structural edit so the pre-edit state becomes the undo target.
  - `undoNoteInput(textarea, inputType)` — coalescing entry point called on every input event; decides whether to open a new boundary (see the coalesce rule) versus fold into the current step.
  - `undo(textarea)` / `redo(textarea)` — move the pointer and restore that snapshot. Apply the restore through the existing `editInsert` chokepoint over the whole range (start 0, end current length, replacement = snapshot.value) so the restore fires a real input event autoGrow/snippets/preview observe, then `setSelectionRange` to the snapshot caret. Guard the whole restore with a re-entrancy flag so the restore's own input event does NOT record a new snapshot. Return a boolean (did anything change) so callers can refresh button state.

Coalesce rule for `undoNoteInput` (human-scale units): open a NEW boundary snapshot when ANY of these hold, else fold the input into the current step — (a) the input added or removed a newline; (b) more than a short coalesce window (choose ~600–900 ms) elapsed since the previous tracked input; (c) the input type category switched between inserting and deleting. Implement the decision as a small pure helper so it is unit-testable in node without a DOM.

Cap the per-field history to a sane bound (e.g. keep the most recent ~200 snapshots) so a very long editing session cannot grow memory without limit; dropping the oldest entries is acceptable.

Write the head-comment for the new region in PLAIN PROSE — describe WHY a module-owned stack replaces the browser's native stack (native coalescing groups typing too coarsely, so undo felt like it wiped everything; and once the buttons own undo, the keyboard shortcut must route to the same stack or the two diverge). Do NOT write any planning identifiers (no decision codes, requirement codes, phase/plan numbers, or process words) in the shipped file — word the constraint by concept. Keep the deliberate use of the deprecated-but-irreplaceable insert primitive documented by concept as the existing banner already does.

Create `tests/undo-stack.test.js` as a node test (the repo's zero-npm runner style, mirroring tests/text-edit.test.js). Use a minimal fake textarea object ({ value, selectionStart, selectionEnd, focus, setSelectionRange, addEventListener/dispatchEvent as needed }) and stub the insert primitive so the history/pointer/restore/coalesce logic is exercised headlessly. Cover every case in the behavior block above. Register nothing new in the runner — top-level tests/*.test.js is auto-discovered.
