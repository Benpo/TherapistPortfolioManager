<objective>
Build the runtime integrity self-check and its two honest user-facing surfaces, and fix update
delivery. The integrity check (VER-03, D-08 fully-local offline-safe self-check) compares the loaded pieces'
deploy token against the source-of-truth token from version.js — agreement = clean, disagreement =
the v209 stale/partial "Frankenstein" cache — with zero network (VER-06). On a real mismatch: the footer upgrades from a
clean `v1.2.0` to the honest marker `v1.2.0 ⚠` (D-09), and a dismissible nudge appears whose words
always match reality (D-10/D-11/D-12) — online runs a GENUINE recovery, offline makes no completion
promise, wedged degrades to honest guidance handing off to Phase 29. Separately, fix update delivery
(VER-01): remove the forced mid-page `controllerchange` reload (apply-on-next-navigation, D-05) and
add `registration.update()` on launch/visibilitychange (D-06) — a general SW-lifecycle reliability fix.

Purpose: This is the heart of the phase — it makes the displayed version unable to silently lie and
makes installed PWAs reliably pick up updates. It consumes the version.js constant (Plan 01) wired
into the app pages (Plan 03).
Output: integrity check + nudge + 4-lang strings in version.js; `.integrity-nudge` CSS; footer marker
in shared-chrome.js; update-delivery fix in app.js; two behavior test files.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service worker → page | the integrity check compares the SW/cached token against the page's source token; a wrong comparison could hide a real mismatch or cry wolf |
| nudge CTA → cache/SW | the genuine recovery deletes caches and re-activates the SW; it must not touch IndexedDB user data |
| nudge copy → user trust | dishonest copy (promising a fix that won't happen) is the core failure this phase exists to prevent |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-10 | Repudiation / Integrity | integrity self-check | mitigate | Pure tested resolver (Task 1) with explicit clean/online/offline/wedged states; footer upgrade is one-directional (never claims clean after detecting dirty, D-09); no network (VER-06) |
| T-28-11 | Spoofing (dishonest UI) | nudge copy | mitigate | Copy is bound to the resolved state by the behavior tests (Task 2); offline makes no completion promise (D-11); wedged never loops a false "refresh" (D-12) |
| T-28-12 | Tampering (data loss) | genuine recovery action | mitigate | Recovery deletes only HTTP/asset caches + re-activates the SW; IndexedDB is never touched (UI-SPEC: non-destructive to user data, no confirm dialog needed); apply-on-next-navigation (D-05) avoids yanking a typing session |
| T-28-13 | Denial of Service | visibilitychange update loop | mitigate | Idempotent listener-install guard (mirrors _backupScheduleListenerInstalled); registration.update() is the SW's own throttled mechanism; tagged catch, no tight loop |
| T-28-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs — zero-build project; nothing to verify |

No high-severity threat is left unmitigated.
