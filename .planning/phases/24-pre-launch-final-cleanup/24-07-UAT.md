# Phase 24 Plan 07 — Browser UAT script

Verifies the photo-crop modal stacking fix. The bug rendered the crop popup
behind the edit-client modal, forcing the user to close edit-client first and
silently losing the photo on crop-confirm.

## Scenario 1 — re-upload photo on existing client with existing photo

1. Open `add-session.html` (any flow — new session or edit).
2. Pick a client who already has a photo in the spotlight.
3. Click the pencil icon to open the Edit Client modal.
4. Click the photo input ("Choose file" or the file picker) and select a
   new image.
5. **Expected:** the crop modal opens **visibly on top of** the Edit Client
   modal. You can see Edit Client behind it (semi-faded by overlay).

**Pass:** crop modal is visible and interactive. You did NOT have to close
Edit Client to see it.

## Scenario 2 — confirm crop → photo persists

Continuing from Scenario 1:
1. Adjust the crop (zoom, drag) as desired.
2. Click "Confirm" on the crop modal.
3. **Expected:** crop modal closes. Edit Client modal remains open. The photo
   preview inside Edit Client now shows the cropped image.
4. Click "Save" on Edit Client.
5. Reload the page or switch to another client and back.
6. **Expected:** the new photo persists on this client.

**Pass:** photo round-trips correctly through the crop → edit-client save flow.

## Scenario 3 — cancel crop → no change

1. Repeat steps 1–3 of Scenario 1.
2. Click "Cancel" on the crop modal.
3. **Expected:** crop modal closes. Edit Client modal remains open. The photo
   preview inside Edit Client is **unchanged** from before the crop attempt.
4. Click "Save" on Edit Client (with no other changes).
5. **Expected:** original photo intact.

**Pass:** cancelling crop leaves both the modal state and the persisted data
unchanged.

## Scenario 4 — new client (no existing photo) → first photo upload

1. Open `add-session.html` for a new session.
2. Pick "+ New client" from the dropdown.
3. In the inline form, click the photo input and pick an image.
4. **Expected:** crop modal opens on top of the inline form (which is in the
   main form, not a modal — crop should still be on top of everything).
5. Confirm crop → preview img shows the cropped photo.

**Pass:** inline-new-client photo flow still works (no regression).

## Scenario 5 — desktop + mobile widths

Repeat Scenario 1 at:
- Desktop width (≥1024px).
- Mobile width (≤480px) via DevTools device emulation.

**Pass:** both viewport sizes show crop above edit-client.

## Scenario 6 — locale + RTL

Repeat Scenario 1 in:
- English (LTR).
- Hebrew (RTL).

**Pass:** layout intact in both, no overflow / clipping issues.

## Scenario 7 — console hygiene

While running scenarios 1–6, DevTools console must show:
- No `TypeError` / `ReferenceError`.
- No "CropModule: DOM not ready" error.

**Pass:** clean console.
