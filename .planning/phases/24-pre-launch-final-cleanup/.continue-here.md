## Critical Anti-Patterns (carry-forward from 24-01)

| Pattern | Severity | Why it matters |
|---------|----------|---------------|
| Marking a plan task `completed` based on grep gates alone for runtime-behavior code | **blocking** | 24-01 incident: structural grep passed (function exists, comments present, call sites migrated) while behavior was broken because an unrelated TypeError aborted the handler. Behavior verification BEFORE marking done is the only mitigation. See `[[feedback-behavior-verification]]` memory. |
| Skipping identifier-resolution trace for code that crosses inside/outside DOMContentLoaded or IIFE boundaries | **blocking** | 24-01 root cause was a top-level `function resetInlineClientForm()` referencing a `const` declared inside DOMContentLoaded. Bare reference resolved to `window.X` via legacy named-element access on `<div id="X">`. For any function-scope-crossing edit, walk the identifier lookup chain explicitly: local → closure → module → window-named-element → undefined. |

## Required Protocol for Remaining Plans (04, 05)

Before implementing any task in 24-04 or 24-05:

1. **Read the PLAN.md task in full.** Identify which surfaces are runtime-behavior (IDB ops, event handlers, DOM mutation, scope-sensitive code, regex with input variability) vs pure-structural (i18n key additions, file creation, comment additions).
2. **For each runtime-behavior task, write a falsifiable behavior test FIRST.** Test file lives at `tests/24-XX-{topic}.test.js`:
   - jsdom + `fake-indexeddb` for IDB
   - vm-sandbox + element shims for pure-helper logic (see `tests/24-06-spotlight-session-info.test.js` as the working template — 7 scenarios, no jsdom, no framework)
   - documented browser-console snippet with numbered steps for visual UI (see `24-06-UAT.md`, `24-07-UAT.md`, `24-08-UAT.md` as templates)
3. **Plan 04 specifically — high-leverage test surfaces:**
   - IDB v5 migration: testable with `fake-indexeddb` — assert objectStoreNames, version, idempotent re-run, `deletedSeedIds` tracking
   - 60-snippet seed pack: testable by `require()` — assert array length, shape of each record (`id`, `trigger`, `expansions: {he,en,cs,de}`, `tags`, `origin: 'seed'`), 4-locale coverage spot-check on 3 random records

   - Snippet shape validator: pure-function, test happy paths + every rejection condition
   - Caret-mirror popover positioning: hardest to test; document a browser UAT script with screenshot expectations instead
4. **Run tests after implementation. Show actual output.** Not "I believe it works." If tests pass → commit. If browser UAT is required → mark task `in_progress`, NOT `completed`, until Ben confirms.
5. **Commit cadence:** one commit per task. Pre-commit hook auto-bumps `sw.js CACHE_NAME` — do NOT pre-bump, do NOT bypass with `--no-verify`.

## Snippet engine (24-04) — what makes it complex

The reason 24-04 warrants extra care vs. the small fixes that closed Plans 06/07/08:

- **IDB v5 migration**: adds a new `snippets` store. Existing users on v4 must migrate cleanly. Idempotent on second open. Backup-restore for pre-v1.1 must still work (D-35).
- **60-snippet seed pack**: 4-locale × 60 entries = 240 strings. Hebrew D-05 noun-form lockdown applies.
- **Trigger regex**: input is user-typed text in any textarea on `add-session.html`. ReDoS risk — adversarial test vectors required.
- **Caret-mirror popover**: the most fragile UI element in the app — must position correctly across all 7 wired textareas, in both LTR and RTL, on mobile + desktop, with auto-resize.
- **BroadcastChannel sync**: `App.getSnippets()` cache must invalidate cross-tab when Plan 05's Settings UI mutates a snippet.
- **sw.js precache**: new assets (`snippets-seed.js`, `snippets.js`) must be added to the precache list; the pre-commit hook auto-bumps CACHE_NAME.

All of the above are runtime-behavior surfaces. Every one needs a falsifiable test BEFORE implementation.
