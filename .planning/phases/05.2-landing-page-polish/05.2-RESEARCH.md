## Summary

This phase is a polish pass on the existing `landing.html` — a single self-contained HTML file with inline SVGs, driven by a JavaScript i18n system in `assets/landing.js`. All visual content (icons, text, layout) is managed through the `LANDING_I18N` object and a `DOODLE_ICONS` constant. New sections (screenshots, contact) must be added both to the HTML structure and to all language objects in `landing.js`.

The two visual improvements (botanical refinement and icon replacement) are purely SVG/CSS changes — no new dependencies, no build tools. The botanical illustrations are already well-architectured using `stroke="currentColor"` on SVG roots, meaning CSS `opacity` and `color` adjustments are the primary levers.

The screenshots section needs to be built with tasteful placeholder tiles (styled with the project's existing design tokens) that accept real images by swapping an `<img src>`. The contact section is a simple new HTML block with translated strings in the i18n system.

**Primary recommendation:** Work within the existing patterns — inline SVG, `var(--color-*)` tokens, `DOODLE_ICONS` constant, `LANDING_I18N` object, `applyLang()` renderer. Do not introduce external icon libraries, new CSS files, or new JS files.

## Architecture Patterns

### How the Landing Page Works

```
landing.html          — HTML structure + initial static content
assets/landing.css    — All landing-specific styles
assets/landing.js     — i18n strings (LANDING_I18N), icon SVGs (DOODLE_ICONS),
                        applyLang() renderer, theme + lang detection
assets/tokens.css     — Design tokens (--color-*, font declarations)
assets/app.css        — Shared component styles (.select, etc.)
```

### Pattern 1: Adding a New Section

**What:** New sections are added as HTML in `landing.html` + translated strings in all language objects in `LANDING_I18N` inside `landing.js` + a rendering block inside `applyLang()`.

**When to use:** Screenshots section, Contact section.

**Structure:**
```html
<!-- In landing.html, at the correct position in the DOM -->
<section id="screenshots" class="landing-section" aria-labelledby="screenshots-title">
  <h2 id="screenshots-title" class="landing-section-title">See it in action</h2>
  <p id="screenshots-subtitle" class="landing-section-subtitle">...</p>
  <div class="screenshots-grid">
    <!-- 3 placeholder tiles -->
  </div>
</section>
```

```javascript
// In LANDING_I18N.en (and he, de, and the 4th language):
screenshotsTitle: 'See it in action',
screenshotsSubtitle: 'A closer look at what\'s inside',
screenshotLabels: ['Dashboard', 'Session view', 'Client view'],

// In applyLang():
setText('screenshots-title', t.screenshotsTitle);
setText('screenshots-subtitle', t.screenshotsSubtitle);
var screenshotLabels = document.querySelectorAll('.screenshot-label');
t.screenshotLabels.forEach(function(label, i) {
  if (screenshotLabels[i]) screenshotLabels[i].textContent = label;
});
```

### Pattern 2: Replacing Feature Card Icons

**What:** `DOODLE_ICONS` in `landing.js` is a plain JS object of SVG strings. Replacing an icon means replacing the SVG string value for a key. The i18n system sets `icon.innerHTML = f.icon` during `applyLang()`.

**Critical:** The current 6 icon keys are `folder`, `lock`, `offline`, `sparkle`, `coin`, `notebook`. The `features` arrays in all 4 language objects reference these keys directly (e.g., `DOODLE_ICONS.folder`). Replacing the SVG content of those keys replaces the icons everywhere across all languages — no per-language changes needed.

**SVG requirements for icons:**
- `width="36" height="36" viewBox="0 0 36 36"`
- `fill="none" stroke="currentColor"`
- `stroke-linecap="round" stroke-linejoin="round"`
- Organic/hand-drawn path style matching the existing botanical aesthetic

### Pattern 3: Making Botanicals More Delicate

**What:** The hero botanical illustrations (`.hero-botanical`) currently render at `opacity: 0.4` in light mode. The "ugly execution" problem is a combination of opacity being too dark/heavy and stroke-widths that are too thick. CSS is the first lever to try; SVG path adjustments are secondary.

**Primary approach — CSS only:**
```css
/* Current */
.hero-botanical {
  opacity: 0.4;
  color: var(--color-primary);
}

/* More delicate — reduce opacity, shift to a softer color */
.hero-botanical {
  opacity: 0.22;          /* lighter */
  color: var(--color-green-400);   /* softer green */
}
```

**Secondary approach — if CSS isn't enough:**
Reduce stroke-widths on the main stems and leaves in the SVG paths within `landing.html`. The current range is `stroke-width="1.1"` to `stroke-width="2.3"`. A delicate result typically uses a range of `0.8` to `1.6` with the main stem at `1.4–1.6` (down from `2.3`).

**Dark mode botanical:** Currently `color: var(--color-cream-warm-200)` at `opacity: 0.4` via `[data-theme="dark"] .hero-botanical`. The same opacity reduction applies.

**Botanical dividers** (`.botanical-divider`, `.botanical-divider-alt`) are currently at `opacity: 0.55`. These may also benefit from a slight reduction.

### Pattern 4: Screenshot Placeholder Tiles

**What:** A visually appealing placeholder that communicates "screenshot goes here" without looking broken. Uses existing design tokens.

**Recommended HTML structure:**
```html
<div class="screenshot-tile">
  <div class="screenshot-placeholder-img" aria-hidden="true">
    <!-- Small botanical SVG or simple plant icon -->
  </div>
  <p class="screenshot-label">Dashboard</p>
</div>
```

**CSS approach:**
```css
.screenshot-tile {
  border-radius: 1rem;
  border: 1.5px solid color-mix(in srgb, var(--color-primary) 25%, transparent);
  background: color-mix(in srgb, var(--color-primary) 5%, var(--color-surface));
  aspect-ratio: 16 / 10;   /* desktop app proportions */
  display: flex;
  flex-direction: column;
  align-items: center;
  justify-content: center;
  overflow: hidden;
  position: relative;
}
```

When real images arrive, the `.screenshot-placeholder-img` is replaced with `<img src="..." alt="...">` and CSS switches from placeholder to image display mode. Structure the HTML so the swap is a 1-line change per tile.

### Pattern 5: Contact Section

**What:** A simple section with contact email or a brief note. No form (no backend). Keep it static: "Have questions? Email us at [email]."

**Position in HTML:** Between the FAQ section and the Impressum/legal sections.

**i18n requirement:** Add `contactTitle`, `contactText`, `contactEmail` (or equivalent) to all language objects. The contact email itself is not translated but the surrounding copy is.

### Anti-Patterns to Avoid

- **Adding a new CSS file:** All landing styles go in `assets/landing.css`. The existing file is 860 lines and well-organized — add new section styles at the bottom.
- **Using emoji as icons:** Phase 05.1 already eliminated emoji icons. All icons must be inline SVG.
- **Modifying per-language icon arrays:** Feature card icons are referenced as `DOODLE_ICONS.key` in all 4 language arrays. Replace the SVG in `DOODLE_ICONS`, not in the language arrays.
- **Hardcoding text in HTML:** All visible text goes through `applyLang()`. HTML should only contain the initial English/default text as fallback.
- **Adding French text manually:** See Open Questions — the language selector currently shows `cs` (Czech), not French. If the brief means French, this requires careful investigation.

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Icon set | Custom icon font, external icon library | Hand-drawn inline SVG matching existing DOODLE_ICONS style |
| Placeholder images | Complex canvas drawing | CSS aspect-ratio + background-color + simple SVG decoration |
| Contact form | Backend form, email service | Static mailto link or plain text email address |
| Animation on botanicals | JS animation library | CSS `opacity` and `transform` (already has hover transitions) |

## Common Pitfalls

### Pitfall 1: Language Mismatch — 4th Language is Czech, Not French

**What goes wrong:** The CONTEXT.md says "4 languages: Hebrew, English, German, French" but `landing.js` implements `en`, `he`, `de`, `cs` (Czech). The language selector dropdown shows Czech (`<option value="cs">Čeština</option>`), not French.

**Why it happens:** The CONTEXT.md was written referencing the desired final state, but the actual code has Czech from the main app's i18n phase.

**How to avoid:** Before adding a new language to LANDING_I18N, confirm with Sapir whether the 4th language should be Czech (as implemented) or French (as stated in CONTEXT.md). See Open Questions below.

**Warning signs:** If you add French strings (`fr`) to LANDING_I18N but the dropdown has `cs`, the language will never be reachable by users.

### Pitfall 2: Botanical Opacity Stack

**What goes wrong:** The hero botanical wrappers have inline CSS `opacity` on `.hero-botanical`, plus the `.botanical-divider` and `.botanical-divider-alt` have their own separate opacity values. Reducing only the hero plants while forgetting the dividers leaves an inconsistent visual weight across the page.

**How to avoid:** When adjusting botanical opacity, audit all 5 botanical contexts: `.hero-botanical` (lines 88–103 in landing.css), `.botanical-divider` (line 135), `.botanical-divider-alt` (line 795), `.botanical-accent` (line 743), `.botanical-positioned` (line 776).

### Pitfall 3: Screenshot Section Positioning (Page Flow)

**What goes wrong:** The screenshots section could logically go in several places. Putting it too early (before features) makes it harder to understand what you're looking at. Putting it too late (after pricing) means prospects who are excited after reading features have to scroll past pricing before seeing the UI.

**How to avoid:** Place the screenshots section after the "Sound Familiar?" section and before the "Why I Built This" section. The page flow should be: features → who it's for → pain points → see the product → founder story → pricing → FAQ → legal.

### Pitfall 4: Icon Style Inconsistency

**What goes wrong:** New icons feel geometric or too clean relative to the existing hand-drawn botanical doodle style.

**How to avoid:** All DOODLE_ICONS SVGs use: `stroke-linecap="round"`, `stroke-linejoin="round"`, varied stroke-widths, organic bezier curves rather than straight lines, and small decorative "botanical accents" (a leaf, a small vine, a berry) extending from the main icon shape. New icons must follow the same conventions.

### Pitfall 5: Contact Section Scope Creep

**What goes wrong:** Adding a full contact form that requires a backend, email service, or third-party integration.

**How to avoid:** This is a static site. Contact is a simple `mailto:` link or plain display of an email address. No form, no AJAX, no external service. The email address itself is a placeholder until Sapir provides it.
