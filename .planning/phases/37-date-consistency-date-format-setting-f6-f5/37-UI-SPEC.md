### Hebrew RTL rendering rules (date engine â€” DATE-04 / D-07)

| Rule | Contract |
|------|----------|
| Numeric date strings in Hebrew | Wrap in Unicode directional isolates `U+2066` (LRI) â€¦ `U+2069` (PDI) â€” a **bare-string** wrap, never `<bdo>` markup (most call sites use `.textContent`). Applies to `mm/dd/yyyy`, `dd/mm/yyyy`, `yyyy-mm-dd` in `he` only. |
| Named-month Hebrew dates | Not wrapped â€” the strong-RTL Hebrew month token anchors direction (matches current `auto` behavior). |
| Consumption contexts | DOM display, `document.title`, PDF (`doc.text` via `bidi.min.js`), markdown export â€” all four must render LTR digit order without flipping. PDF isolate-tolerance is **test-gated** (assert `2026` present / `6202` absent). |
| Chrome strings (tab, editor labels) | Plain text â€” no RTL special-casing beyond what existing settings tabs already handle. |

### Accessibility & theming (inherit â€” do not regress)

- All user-entered labels rendered via `.textContent` / `input.value` â€” **never `innerHTML`** (XSS + the data-i18n gate).
- Icons built via `createElementNS` DOM APIs with `aria-label` (lock tooltip, delete button).
- Focus-visible outlines: 2px `--color-primary`, offset 2px (existing `:focus-visible`).
- Both light and dark mode inherited automatically via semantic tokens â€” no per-theme code needed. Verify the new tab, select, list rows, and lock/delete icons in **both** themes.
- Toasts announce via existing `App.showToast(message, key)` `aria-live` path.
