# Phase 34 — UI Design Contract

> Visual and interaction contract for the **client-facing session-export PDF**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is NOT an HTML/CSS UI.** The deliverable is a **jsPDF-generated A4 PDF** (`assets/pdf-export.js`). Every value below must be realizable with jsPDF primitives only: flat fills (`rect` / `roundedRect`), lines (`line` / `setLineWidth` / `setDrawColor`), embedded PNG (`addImage`), colored text (`setTextColor` + Heebo normal/bold). **No gradients, no box-shadows, no CSS.** Sizes are in **points (pt)** — the document's native unit (A4 = 595 × 842 pt).
>
> **Source of truth:** `34-DESIGN-DECISIONS.md` (locked with Ben 2026-06-29) + `design-mockups/FINAL-mockup.html`. This spec **formalizes** those locked decisions — it does not re-litigate them. Genuinely-unresolved items appear only under **Open Micro-Decisions** as FLAGs, never as blockers.

## Hard Constraints (non-negotiable — carried from `34-DESIGN-DECISIONS.md`)

1. **Hebrew RTL / bidi must NOT regress.** Phase 23 was a full RTL rewrite. Every `doc.text()` keeps the `isInputVisual:false` invariant; `docDir` stays `(opts.uiLang === 'he') ? 'rtl' : 'ltr'` (`pdf-export.js:682`). New structural elements (header, card, pill, severity bars, footer) must mirror correctly under RTL — content anchors to the start edge, bar fills grow from the start edge, numerals render in correct visual order.
2. **Phase 30 PDF test suite stays green** (`npm test`). The 7+ PDF characterization tests must pass; layout-assertion tests are updated **deliberately** where the new geometry changes a measured baseline (the redesign moves type sizes and y-positions — those assertions are expected edits, not silent breaks).
3. **Offline / zero-network.** The logo is an **embedded PNG** (`assets/branding/icon-512.png` via `doc.addImage`), bundled/precached — never fetched.
4. **A4 portrait locked.** Page = 595 × 842 pt. Existing **71pt symmetric margins** are the content-margin baseline (`MARGIN_X = MARGIN_TOP = MARGIN_BOTTOM = 71`, `pdf-export.js:701–703`).

## Spacing Scale

PDF vertical rhythm in **points**. The mockup expresses spacing in mm; values below are the mm rhythm snapped to a 4pt grid (1mm ≈ 2.835pt). The executor should use these pt values, not re-derive from mm.

| Token | Value (pt) | Usage (mockup origin) |
|-------|-----------|-----------------------|
| xs | 4pt | hairline gaps: bullet→heading, rule→text, bar caption→track |
| sm | 8pt | section-heading bottom margin (3mm); severity block top (3mm); intra-card row gaps |
| md | 12pt | pill horizontal padding; client-name→meta-row gap |
| lg | 16pt | body line gaps; card vertical padding (6mm); severity-row vertical padding |
| xl | 20pt | card bottom margin (7mm); card horizontal padding (7mm) |
| 2xl | 24pt | header band vertical padding (8mm); card top margin (9mm); section-heading top margin (8mm) |
| 3xl | 36pt | body top inset below the band (13mm) |

**Horizontal content inset:** **71pt** (= existing `MARGIN_X`). The mockup band/body/footer use a 20mm (~57pt) inset; per Hard Constraint #4 the locked **71pt** baseline wins for body text alignment. The **header band and footer band fills are full-bleed** (rect from x=0 to x=595), but their *content* (logo, title, footer text) insets to the 71pt content edge so it aligns with the body. → See Open Micro-Decision FLAG-1.

Exceptions:
- Logo tile = **48pt** square (fixed, not on the 4pt grid — it's an icon dimension).
- Severity track width = **118pt**, track height = **8pt**, fill radius = **4pt**, caption column = **34pt**, bars block = **190pt** (severity geometry, fixed — see Component Inventory).
- Logo keyline corner radius ≈ **10pt**; card corner radius ≈ **10pt**; pill = fully rounded (radius ≥ half its height).

## Component Inventory (PDF — top → bottom)

The visual contract, in render order. Each item names its fills, strokes, type, and RTL behavior.

### 1. Header band (full-bleed)
- **Fill:** `rect(0, 0, 595, bandHeight)` filled `#e2f3e3` icon-mint. `bandHeight` ≈ logo(48) + 2×24pt vertical padding ≈ 96pt.
- **Logo tile:** 48pt square at the **start** edge (left in LTR / right in RTL), inset 71pt. `addImage(icon-512.png, x, y, 48, 48)` edge-to-edge (NO white padding frame). On top, a **green keyline**: `roundedRect(x, y, 48, 48, 10, 10, 'S')` stroke `#3a7d5f` at ~1.5pt. → jsPDF cannot clip the PNG to rounded corners (FLAG-7).
- **Title block:** beside the logo (12–14pt gap). Title 20pt bold `#345e34`; subtitle 11pt normal `#456b42` below it.
- **Copy:** title = "Session Summary" / "סיכום מפגש". Subtitle = "A personal record for the client" / "מסמך אישי עבור המטופל/ת" (→ FLAG-3, wording open). **No "Sessions Garden" wordmark as a letterhead** — on a client's copy it would read as if the clinic is named Sessions Garden.
- **RTL:** logo on the right, title block to its left, text right-anchored.

### 2. Client card
- **Surface:** `roundedRect` filled cream `#fdf8f0`, stroke `#bfe0b0` (FLAG-2) at ~1pt, corner radius ~10pt. Inset 71pt; vertical padding 16pt, horizontal padding 20pt. Top margin 24pt below the band, bottom margin 20pt.
- **Client name:** 23pt bold `#345e34`, start-anchored.
- **Meta row** (12pt below the name, single wrap-capable row): **Date** (key `#456b42` bold + value muted) · **Session #N** (key + value) · **green pill** carrying the localized in-person/remote value. (NO session type / modality field.)
- **Pill:** fully-rounded `roundedRect` filled `#e8f5ee`, text 11pt bold `#1e5c3a`, padding ~12pt horizontal / 2pt vertical.
- **`N` = derived chronological ordinal** — see Functional Contract FN-1.
- **Pill value** = the session's existing localized in-person/remote field, verbatim — see FN-2.
- **RTL:** name and meta right-anchored; meta items flow right→left; pill mirrors.

### 3. Section heading (repeats per section)
- **Bullet:** a `#7da877` rotated-square "leaf-diamond" ~9pt, drawn as a 45°-rotated filled square (jsPDF: `triangle`/polygon path or a rotated rect via `lines`/matrix) at the start edge.
- **Label:** 16pt bold `#456b42`, 4pt after the bullet.
- **Rule:** a `#bfe0b0` line at ~1.5pt spanning the content width, ~4pt beneath the label baseline.
- **Top margin 24pt; bottom margin 8pt.**
- **RTL:** bullet on the right, label to its left; rule spans full content width either way.

### 4. Trapped emotions / summary body
- **Free-text paragraph(s)** — NOT chips or a word-cloud (it is free text in real practice). 11.5pt normal `#2f2d38`, line-height 1.65, `splitTextToSize` wrap to content width, start-anchored. Inline-bold (`**x**`) continues to render via the existing `parseInlineBold` → Heebo bold path.

### 5. Severity — before & after (per complaint)
The signature visual. Each complaint = one **row** separated by a `#eef7ea` 1pt bottom rule (9pt vertical padding):
- **Label** (complaint name) takes the leading flex space — 11.5pt normal `#2f2d38`.
- **Bars block** (~190pt, on the trailing side) = **two stacked barlines on a shared 0–10 track**:
  - **Before barline:** caption "Before"/"לפני" (9.5pt bold muted, 34pt column) · **track** (118pt × 8pt, radius 4pt, bg `#eef7ea`) with a **red `#ea4b4b` fill at 0.85 opacity**, width = `before/10 × 118pt` · value numeral (10.5pt bold `#ea4b4b`).
  - **After barline:** caption "After"/"אחרי" · same track · **green `#2fb37d` fill**, width = `after/10 × 118pt` (shorter — the reduction reads as length) · value numeral (10.5pt bold `#2fb37d`).
- **Why two bars, not one gauge:** a single fill read ambiguously as "filling up"; severity is lower-is-better and not meant to max out. The shorter green bar shows the reduction by length. (Locked.)
- **Data source:** existing issue before/after payload (`getIssuesPayload` / the Phase 30 `30-issue-delta` characterization already pins the delta shape).
- **RTL:** the whole row mirrors — label on the right, bars block on the left; each track fills **from the start (right) edge**; caption→track→value order mirrors; numerals keep `isInputVisual:false` visual order.

### 6. Footer (full-bleed band, every page)
- **Top rule:** `#bfe0b0` line at ~1pt spanning content width (mockup uses mint-soft; decisions-doc says `#bfe0b0` — pin vein).
- **Three zones** (start / center / end), 10pt:
  - **Start — "made with":** small ~15pt logo + "Made with Sessions Garden · sessionsgarden.app" / "נוצר עם Sessions Garden · sessionsgarden.app" in `#456b42` bold. Brand appears here as the **tool**, subtly — never as a letterhead.
  - **Center — pagination:** "Page X of Y" / "עמוד X מתוך Y".
  - **End — created date:** e.g. `24.03.2026` (→ FLAG-4: may duplicate the card's session date).
- **RTL:** zones mirror; "made with" on the right, created date on the left.

## Functional / Behavioral Contract (visual-adjacent — implemented + tested in plan-phase)

These are visual values backed by behavior. They are **design contract here**; the falsifiable behavior tests are authored at plan time (per `feedback-behavior-verification` — test BEFORE implementation).

### FN-1 · Session number = derived chronological ordinal (MUST be tested)
- The "Session #N" in the card is **NOT** the DB key. `sessions` is `keyPath:"id", autoIncrement:true` (`db.js:225`) — that id never renumbers, so printing it leaves gaps after a deletion (e.g. `1, 3`).
- **N = 1-based position of this session among `getSessionsByClient(clientId)` sorted ascending by `date`** (deterministic tie-break by `id` for same-date sessions), computed **at export time**, never stored.
- **Expected behavior:** client has sessions d1<d2<d3 (ordinals 1,2,3); delete the d2 session → the d3 session's export now shows **#2**.
- **Required behavior test:** seed 3 dated sessions → assert exported ordinals 1/2/3 → delete the middle → re-derive → assert remaining two are 1/2 (no gap, renumbered).

### FN-2 · In-person/remote value
- Read the session's **existing** master-data in-person/remote field (already in the app, **localized**) and render its value verbatim in the green pill — **no new field, no hardcoded label**. Confirm the exact field name + i18n keys in code at plan time (FLAG-5).

### FN-3 · Logo embedding
- Embed `assets/branding/icon-512.png` (verified present: 512×512 RGB PNG) via `doc.addImage`; precache/bundle so export works fully offline. At 512px into a 48pt tile = ~10.6 px/pt (~768 DPI) — crisp. Confirm the addImage format string (`'PNG'`) and that the asset is in the SW precache list.
