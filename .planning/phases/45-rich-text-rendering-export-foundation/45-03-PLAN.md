<objective>
Solve the phase's single hardest architectural problem (D-03) and its visual constraint (D-02) — on the ACTUAL export path. The PDF, share, and markdown-download are all built from `editor.value` (the Step-2 editor, populated by `buildFilteredSessionMarkdown(selected)` at `export-modal.js:839`, consumed at `:679` PDF / `:742` share / `:710` `.md` download); only the clipboard **copy** calls `buildSessionMarkdown()` directly (`:906`). So a user-typed `## Progress` inside a note arrives in `editor.value` as a bare `## Progress` line indistinguishable from a document `## Section` — today it would wrongly get the branded leaf-diamond+green-rule chrome (`pdf-export.js:1326`+) AND corrupt the severity-block page-break count at `pdf-export.js:1313`. This plan classifies each parsed `## X` heading as **document** (its text matches a known localized section label) vs **note** (it does not) — the same technique the existing `severityAfterSections` computation already uses to re-derive document-heading identity from `editor.value` (`export-modal.js:650-653`) — and gates BOTH the section-count guard AND the branded-chrome branch on that classification, routing note headings to a subordinate chrome-free register (UI-SPEC §C: 12/11/10.5pt bold, no chrome). It also locks the RTXT-08 verbatim copy/share/download pass-through with a falsifiable test.

Purpose: Notes must render as notes, not forge document sections. The classification is passed into `buildSessionPDF` as data (a document-section label set) — NO sentinel is injected into the markdown, so `editor.value`, the copy clipboard, and the `.md` download all stay byte-clean (D-10) and Step-2 edits are honored. Sequenced after Plan 02 (both touch pdf-export.js) so the export surface is edited one hand at a time.

Output: A note-aware PDF pipeline (D-02/D-03) built on the real `editor.value` path + a byte-for-byte copy/share/download lock (RTXT-08).
</objective>
