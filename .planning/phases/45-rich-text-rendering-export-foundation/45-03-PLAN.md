<objective>
Solve the phase's single hardest architectural problem (D-03) and its visual constraint (D-02) â€” on the ACTUAL export path. The PDF, share, and markdown-download are all built from `editor.value` (the Step-2 editor, populated by `buildFilteredSessionMarkdown(selected)` at `export-modal.js:839`, consumed at `:679` PDF / `:742` share / `:710` `.md` download); only the clipboard **copy** calls `buildSessionMarkdown()` directly (`:906`). So a user-typed `## Progress` inside a note arrives in `editor.value` as a bare `## Progress` line indistinguishable from a document `## Section` â€” today it would wrongly get the branded leaf-diamond+green-rule chrome (`pdf-export.js:1326`+) AND corrupt the severity-block page-break count at `pdf-export.js:1313`. This plan classifies each parsed `#`/`##`/`###` heading (levels 1-3, WARNING 2) as **document** (its text matches a known localized section label OR the localized document title `session.copy.title`) vs **note** (it does not) â€” the same technique the existing `severityAfterSections` computation already uses to re-derive document-heading identity from `editor.value` (`export-modal.js:650-653`) â€” and gates BOTH the section-count guard (kept at `level >= 2`, document only) AND the branded-chrome branch (levels 1-3) on that classification, routing note headings to a subordinate chrome-free register (UI-SPEC Â§C: 12/11/10.5pt bold, no chrome). Including the level-1 document title in the label set is essential: the chrome branch (pdf-export.js:1326-1363) brands ALL heading levels, so without the title in the set a levels-1-3 classification would demote the document title on every PDF, and a level-â‰Ą2-only classification would leave a note-typed `# Big` with full branded chrome. It also locks the RTXT-08 verbatim copy/share/download pass-through with a falsifiable test.

Purpose: Notes must render as notes, not forge document sections. The classification is passed into `buildSessionPDF` as data (a document-section label set) â€” NO sentinel is injected into the markdown, so `editor.value`, the copy clipboard, and the `.md` download all stay byte-clean (D-10) and Step-2 edits are honored. Sequenced after Plan 02 (both touch pdf-export.js) so the export surface is edited one hand at a time.

Output: A note-aware PDF pipeline (D-02/D-03) built on the real `editor.value` path + a byte-for-byte copy/share/download lock (RTXT-08).
</objective>

## Documented limitations (executor records BOTH in 45-03-SUMMARY.md)

The label-set classification is a text-equality match (the same inherent design the existing `severityAfterSections` label-match carries). Two accepted, coherent consequences â€” record both, and pre-flag them at the Plan 06 gate so they are not filed as bugs:

1. **False-positive direction:** a NOTE heading whose text EXACTLY equals a document-section label OR the document title (`session.copy.title`) is classified as a DOCUMENT heading (keeps chrome, counts). Rare; acceptable.
2. **NOTE 6 â€” false-negative direction:** a DOCUMENT section heading RENAMED in the Step-2 editor so its text no longer matches a label is now classified as a NOTE heading â€” it demotes to the subordinate register AND stops incrementing the section count (before this plan, a renamed heading kept chrome and counted). This is the direct converse of (1) under the same model; accepted.
