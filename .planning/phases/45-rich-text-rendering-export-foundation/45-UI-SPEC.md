# Phase 45 — UI Design Contract

> Visual and interaction contract for the rich-text **rendering & export** foundation.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Nature of this phase: REUSE-ONLY, no new chrome.** Phase 45 renders markdown that already
> exists at-rest into three read surfaces (read mode, export-modal preview, PDF) and strips it in
> three compact surfaces. It introduces **zero new components, zero new colors, zero new CTAs** —
> it extends `window.MdRender`, the `pdf-export.js` parse/render pipeline, and the existing
> `.export-preview` rendered-markdown CSS. The editor/toolbar is Phase 46. See **Accepted
> Exceptions** at the bottom before running greenfield checker rules.

## Typography

The app is `rem`-based on a 16px root (Rubik). Two contexts render notes: **screen** (read mode +
export preview) and **PDF** (jsPDF, `pt`). Both are specified below.

### A. Screen — rendered note body content (read mode + export-modal preview)

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Note body (`p`) | 1rem / 16px | 400 | 1.5 | Matches existing `.export-preview p`; identical to the editable textarea it replaces |
| Bold (`strong`) | inherit | 700 | inherit | Weight only — no color change |
| Italic (`em`) | inherit | 400 *italic* | inherit | Rubik ships no italic face → browser-synthesized oblique on screen (accepted); regular-weight in PDF per milestone lock |
| Bullet / numbered list (`ul`/`ol` `li`) | 1rem / 16px | 400 | 1.5 | `list-style` shown (disc / decimal); nested levels indent 24px each |

### B. Screen — NOTE-content headings (D-01, D-02) — the new subordinate register

Note headings (`#`/`##`/`###` typed inside a note field) MUST read as **subordinate to the field's
own label** (e.g. "Trapped Emotions", ~17px/700) and MUST NOT be mistaken for the export document's
section headings (`.export-preview h1/h2` = document title/section chrome). Compressed, weight-led,
never rule-decorated:

| Role | Size | Weight | Line Height | Spacing |
|------|------|--------|-------------|---------|
| Note H1 (`#`) | 1.0625rem / 17px | 600 | 1.3 | `margin-block: 16px 8px` |
| Note H2 (`##`) | 1rem / 16px | 600 | 1.35 | `margin-block: 16px 8px` |
| Note H3 (`###`) | 0.9375rem / 15px | 600 | 1.4 | `margin-block: 16px 4px` |

Note headings carry **no border-bottom / no green rule / no leaf-diamond** (those belong to document
section chrome only). Differentiation from body is by weight (600 vs 400) + spacing, not size jump.

> **Selector-scoping requirement:** these note-heading styles must target the read-mode / preview
> **note** container ONLY (Claude's-discretion render mechanism, e.g. a `.note-rendered` wrapper).
> They must NOT bleed onto `.export-preview h1/h2/h3`, which continue to style the document skeleton
> built by `buildSessionMarkdown()` at their existing values. In the export-modal preview the content
> IS the full document (so document headings keep their register); in read mode the content is a bare
> note field (so note headings use the subordinate register above).

### C. PDF — NOTE-content headings (D-02, D-03)

Document section headings in the PDF are 14pt + leaf-diamond + green rule (Phase 34 D-06 chrome).
Note headings render **strictly subordinate** and **chrome-free**, as a distinct note-body block
category so the `parseMarkdown` section-counting page-break logic (~L1286–1313, counts `heading`
blocks `level >= 2`) is never confused by them (mechanism = planner's choice per D-03):

| Role | Size (recommended default) | Weight | Chrome |
|------|----------------------------|--------|--------|
| PDF note H1 | 12pt | bold | none — no diamond, no rule |
| PDF note H2 | 11pt | bold | none |
| PDF note H3 | 10.5pt | bold | none |
| PDF note body / bold | existing body size | reg / bold | reuses `parseInlineBold` bidi segment pipeline |

Exact pt values are planner + executor discretion **within the D-02 subordination constraint** (must
be < 14pt and must not carry section chrome). Verified on a **real opened PDF** with Hebrew content,
never jsdom-only.

## Color

No new colors. Rendered notes inherit the semantic text tokens; the 60/30/10 split is the app's
established garden identity (light) / teal-grey (dark), both from `tokens.css`.

| Role | Value (light → dark) | Usage |
|------|----------------------|-------|
| Dominant (60%) | `--color-background` (#fdf8f0 → #1a2020) | Page / form backdrop behind rendered notes |
| Secondary (30%) | `--color-surface` (#ffffff → #202828) | The rendered-note panel surface (matches `.export-preview` bg) |
| Accent (10%) | `--color-primary` (green #2d6a4f → teal #5aacac) | **Not introduced by rendered note bodies.** Reserved for existing chrome only (buttons, active states) — rendered markdown adds no accent-colored elements |
| Text | `--color-text` (#2f2d38 → #e8eeed) | Note body, bold (weight-only), and note headings |
| Destructive | `--color-danger` (#ea4b4b, light-only) | **Not used** — this phase has no destructive actions |

Accent reserved for: existing buttons / active toggles / links only. **Rendered note content
introduces no accent-colored, no danger-colored, and no new-token elements** — bold is weight, not
color; headings are `--color-text`. Dark-mode + RTL rendered notes must both remain WCAG-AA legible
using these existing, already-contrast-checked tokens.

## Copywriting Contract

This is a rendering foundation — it adds **no new buttons, labels, empty-states, or dialogs**. The
row values below record the *behavioral* copy/interaction contract for the surfaces it touches.

| Element | Copy / Contract |
|---------|-----------------|
| Primary CTA | **None added.** Read mode reuses the existing "Copy all" / export actions; rendered content sits inside the existing read-only form. Do not add a CTA. |
| Empty state | An empty note field renders as **nothing** (no placeholder text, no "no notes" string) — matches current read-mode behavior; an empty field simply shows no rendered block. |
| Error / fallback state | MdRender is escape-first and total (never throws on arbitrary user text). If `window.MdRender` is unavailable, the surface falls back to **plain `textContent`** (raw markdown shown literally) — the existing export-modal pattern (`export-modal.js` L541). **No error message is ever surfaced to the therapist**; malformed markdown degrades to safe literal text, never to raw `innerHTML`. |
| Destructive confirmation | Not applicable — no destructive actions in this phase. |

**Compact-surface strip (D-06) — display transformation, not copy:** the three compact spots
(`sessions.js:262` trapped-emotions cell, `overview.js:848` comments line, `add-session.js` ~1606
customerSummary spotlight quote) render markdown **stripped to plain text** via a shared strip helper
(`**bold**` → `bold`, `- item` → `item`, `# h` → `h`). No styled HTML, no layout risk in table rows
/ one-liners. Strip output goes through `textContent` — the render-hardening test locks stay intact.

**Verbatim copy/share (D-10):** stored note markdown flows **byte-for-byte** into
`buildSessionMarkdown()` note bodies — copy/share is a pass-through property to protect with tests,
not a transform. No copy strings change.

## Interaction & RTL Contract (rendering-specific)

| Concern | Contract |
|---------|----------|
| Escape-first invariant | ALL note text renders through MdRender's `escapeHtml`-then-markdown path. Read-mode rendered HTML is a **deliberate, narrow exception** to the app's `textContent`-only hardening — route it exclusively through MdRender; extend (never weaken) the render-hardening test locks. |
| Read-mode edit-toggle | The rendered view swaps in/out inside `setReadMode` (add-session.js ~L306) and MUST preserve the existing edit-toggle flow, copy-all, and export buttons. Editing toggles back to raw textareas. |
| RTL indentation | List/heading indent uses **logical properties** (`padding-inline-start`, `margin-inline`) for screen; the PDF needs **physical-direction** care for Hebrew list/heading indent (see memory: RTL logical-props-vs-physical-coords pitfall). Verify bullet AND numbered nesting in Hebrew on a real device. |
| `<br>` / paragraph contract | The single-newline-`<br>` / blank-line-paragraph contract in `md-render.js` is **LOCKED** (header comment) — do not change it while adding ordered-list, nesting, or hardened-inline support. |
| Inline-rule hardening (D-08) | `*`/`**` must hug non-whitespace (CommonMark-style) so legacy `2 * 3 * 4` stays literal. Applies in **both** MdRender AND the PDF `parseInlineBold`/strip helpers — the two pipelines must agree on what counts as formatting. Preserve the PDF invariant `parseInlineBold(x).map(s=>s.text).join('') === stripInlineMarkdown(x)` relied on by `drawSegmentedLine`. |

## Accepted Exceptions (read before running greenfield checker rules)

Per memory `feedback-ui-checker-greenfield-false-positives` — the checker's greenfield heuristics
mis-fire on reuse-only phases in an established design system. For Phase 45 the following are
**intentional and correct**, verified against live source (`tokens.css`, `app.css`):

1. **Three font weights (400 / 600 / 700), not two.** This is the *established, shipped* Rubik
   system (`tokens.css` `@font-face` L1–22), not a new decision. Bold markdown → 700; headings →
   600; body → 400. Reuse, not sprawl.
2. **`px`/`rem` typography values that aren't multiples of 4** (17px, 15px, 0.9375rem…) are
   type-scale values on the existing rem system, not spacing. The 4px-grid rule governs spacing —
   spacing here is 4/8/16/24 only.
3. **No primary CTA.** This phase adds no button; the "generic CTA label" dimension is N/A. Do not
   flag a missing/weak CTA.
4. **No new color / no accent introduction.** Rendered notes reuse `--color-text` only; the 60/30/10
   table documents the existing identity for context, not a new palette.

Reviewer action if a greenfield rule fires on the above: confirm against live source, record the
accepted exception, and PASS — do not spin the remediation loop.
