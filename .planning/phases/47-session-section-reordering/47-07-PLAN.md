<objective>
Wire the session form's severity semantics on top of tap-again-to-clear (47-01) and the D-02 structure (47-04). Under the unrated-by-default model (D-19), a topic's start rating is optional: clearing it (null) must stay save-safe and must auto-hide that topic's end-of-session rating row, keyed on EMPTINESS not on a marker (D-22). And make the 'Issue severity' section's enable toggle the app-level severity switch: on new/empty sessions, off hides the end-of-session block AND the per-topic start-rating column while the topics stay (D-08); on past sessions that carry severity data, both stay visible with the disabled badge, mirroring the shipped disabled-section-with-data rule (G-11). Swap the topic-row label to 'Severity at start' so severity is named once (in Settings), never repeated on the row. validateIssues already accepts any rating state (payload.length > 0 only) — this plan must NOT invent new rating-mandatoriness (G-17); the real deliverable is that a cleared (null) rating flows through getIssuesPayload → save → reload without being coerced or blocking save.

Purpose: ORDR-06 (severity switch) + ORDR-07 (unrated-by-default) on the form side. This is the second add-session pass (after the 47-04 restructure) so it stays in a later wave to avoid file conflict.

Output: unrated-aware form readers + the end-of-session-rating auto-hide + severity-off column coupling in add-session.js; the topic-row label swap; a form-severity behavior test.
</objective>
