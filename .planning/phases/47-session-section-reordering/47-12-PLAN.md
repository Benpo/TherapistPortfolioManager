  ZERO planning IDs in comments/identifiers/strings — no ORDR-NN, D-NN, G-N, R-N, 47-NN,
  or "Phase 47". Keep the diff clean of planning tokens.
- **Diagnosis before fix (G1) — mandatory.** Do NOT patch blind. Reproduce and root-cause
  against the actual event flow in `wireDrag` (assets/settings.js ~line 555) before
  changing anything. The drag currently binds `pointermove`/`pointerup` on the handle
  after `setPointerCapture`, adds `.dragging` on `pointerdown`, and removes it only in
  `onUp`. Candidate WebKit failure classes to INVESTIGATE (not assume): (a)
  `setPointerCapture` on the handle behaving differently on Safari desktop than Chromium
  so `pointermove` never reaches the handle listener; (b) no movement threshold/slop —
  Safari may suppress or reorder events for a capture that never moves; (c)
  text-selection / `user-select` interference stealing the gesture (the rows contain
  editable inputs and labels); (d) the `.dragging` highlight being cleared ONLY in a
  handler Safari does not fire on that release path (pointercancel / lostpointercapture),
  leaving it stuck. State the confirmed root cause in the SUMMARY.
- **Both input paths must keep working.** The fix must preserve the up/down arrows as the
  accessible fallback and keep BOTH drag and arrows funnelling every move through the one
  shared `App.sanitizeOrder` clamp (`enforceOrderClamp` / `markReorderDirty` path). Do not
  regress the existing arrow behavior or the sentinel-persist on Save.
- **Highlight cleanup is unconditional.** After the fix, `.dragging` must be removed and
  the capture released no matter how the gesture ends — add `pointercancel` and
  `lostpointercapture` handling so no highlight can persist after release on any path.
- **G2 root cause first.** The saved-notice check is CSS-drawn: `.settings-saved-notice-mark::after`
  (app.css ~3479) builds a checkmark from `border-block-end` + `border-inline-end` rotated
  45deg, anchored with `inset-inline-start`. Those are LOGICAL properties, so under RTL the
  corner mirrors and the checkmark reads as a wrong-facing chevron. A checkmark is a
  SEMANTIC glyph (like the app's bold/italic/eye/pencil), not a directional one — it must
  stay upright in RTL. Follow the app's own convention (app.css ~5711: directional glyphs
  flip via `.icon-flip`, semantic glyphs stay upright) and repo memory
  `reference-rtl-logical-props-physical-coords`: draw the mark with PHYSICAL geometry (or
  add an `html[dir=rtl]` override) so RTL never mirrors it. Confirm it is a CSS flip, not a
  wrong asset. Preserve light/dark tokens and the reduced-motion block.
- **jsdom is blind to WebKit drag operability and to computed CSS geometry.** Do not fake a
  green. Guard the machine-checkable contracts (highlight cleanup, drag-through-clamp) in
  jsdom; specify an explicit Safari manual checklist for the drag operability and an RTL
  visual check for the glyph.
</guidance>
